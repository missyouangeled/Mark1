## 18:00 — Frontstage Guardian 深度分析

### 触发器
- 点点问"frontstage-guardian failed 是什么意思"，要求详细报告
- 随后进一步要求：查今天 error 原因、全面分析、联网搜索、给方案

### 分析过程
- 读取了 `scripts/openclaw-frontstage-guardian.py` 源码（主脚本+两个子脚本 recovery/responsiveness）
- 提取 systemd journal 日志（600+ 条），统计故障率：546 次检查 / 132 次 ERROR = 24.2%
- 按时间轴聚类分析，发现 132 次 ERROR 聚合成 4 个独立故障簇

### 核心发现：4 个不同的根因

1. 🔴 **12:00 会话压缩风暴（43 次连续 ERROR，一次都没恢复）**
   - 根因链：会话 683 条消息/12 万 token → 触发自动压缩 → gpt-4o-mini 窗口 12k 装不下 50k token 的压缩 prompt → 连续失败 4 次 → gateway 事件循环 100% 饱和（最长卡 11.6 秒）→ guardian 每次 25 秒超时
   - 恶性循环：压缩失败 → 消息继续追加 → 再膨胀 → 再触发压缩 → 再失败 → guardian 每分钟打一次加重负担
   - 12:49 后会话恢复正常大小，ALL OK 恢复

2. 🟠 **07:00 网络断连（16 次）**：`getaddrinfo EAI_AGAIN api.deepseek.com`，DNS 解析失败

3. 🟠 **15:00 gateway 重启事故（15 次）**：restart 遇到 OpenClaw 已知 bug #86014（会话文件锁不释放），优雅关闭变强制重启，gateway 下线约 40 分钟

4. 🟡 **零星波动（约 58 次）**：正常负载下 `openclaw gateway call` 偶发超时，每次 1-3 次自愈

### 结构性问题

**循环依赖**：Guardian 设计目的是检测 gateway 健康，但它依赖 `openclaw gateway call` → 与 gateway 共享事件循环 → gateway 忙时 guardian 一起超时

### 解决方案（按优先级）

| 优先级 | 方案 | 效果 |
|--------|------|------|
| 🔴 高 | 换 compaction 模型：gpt-4o-mini → gpt-4o（128k 上下文） | 消除 12 点类事故根因 |
| 🟠 中 | guardian 加退避逻辑：连续 3 次 ERROR → 降频到 5 分钟 | 防止集群式连续错误 |
| 🟡 中 | guardian 直接读本地文件，不走 gateway API（速度 7s→0.5s） | 根本解耦 |
| 🟢 低 | gateway MemoryHigh 限制 | 防御 swap thrashing |
