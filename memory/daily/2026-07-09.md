## flush-sync 17:25：SIGTERM 强杀问题修复 ✅

### 根因
OpenClaw 源码 `process.on("SIGTERM")` 检测到 restart intent → in-process restart(不退出进程) → systemd `TimeoutStopSec=60s` 等不到 exit → 60s 后 SIGKILL 整个 cgroup → 连带杀 sidecar/mark42/caddy/php。

### 数据
7 天 74 次 SIGTERM,137 次强杀(SIGKILL),7/6 单日 62 次。

### 修复
修改 `timeout-fix.conf`: `TimeoutStopSec=60 → 600`(10 分钟)
- in-process restart 实测 7-10s 完成,600s 足够
- 主动 restart 验证:17:25:28 SIGTERM → 17:25:33 ready,**无 Killing 日志** ✅
- 主会话自动恢复:`recovered=1 failed=0`

### 副作用
真要 stop 时最多等 10 分钟(stop 只在 update/gateway stop 时发生),影响可控。

### 留痕
详细记录: `memory/daily/2026-07-09-sigterm-fix.md`

## flush-sync 16:25：SIGTERM 来源排查结论

### 已确认的 SIGTERM 触发链

#### 1. OpenClaw 内部热重载机制（最可能的源头）
OpenClaw 自身有 **config watcher + SIGUSR1 自重启** 机制：

- 文件：`/home/missyouangeled/.npm-global/lib/node_modules/openclaw/dist/server-reload-handlers-BMs4zVdr.js`
- 触发链路：
  - chokidar 监听配置文件变化（`openclaw.json` / skill 安装 / plugin 安装 / state 文件）
  - 检测到变化 → `buildGatewayReloadPlan()` → 判断需不需要 restart
  - 需要 restart → `emitGatewayRestart()` → `process.kill(process.pid, "SIGUSR1")`
  - Gateway 收到 SIGUSR1 → 内部重启（先 drain，drain timeout 300000ms）
- 关键代码：
  - `restart-DzdS7Ejv.js:329` `process.kill(process.pid, "SIGUSR1")`
  - `restart-DzdS7Ejv.js:611-661` "config change requires gateway restart"
- **但我们看到的是 SIGTERM 不是 SIGUSR1**，所以这条不是直接凶手，是"近亲"

#### 2. openclaw-resume-watch.sh（已确认但不是今天的）
- 文件：`/home/missyouangeled/.openclaw/workspace/scripts/openclaw-resume-watch.sh`
- 逻辑：每 5min 检测一次"Gateway 距上次活动 > 600s + 当前无活跃连接"→ `systemctl --user restart openclaw-gateway.service`
- **审计日志只到 7/9 之前，今天 8 次未触发这条**（审计日志断在 7/7）
- 历史记录显示 6 月份频繁触发（多次 sleep 后唤醒）

#### 3. systemd 直接 stop（已确认只在 15:49:08 一次）
- journalctl 记录：**今天 8 次 SIGTERM 中只有 15:49:08 这次**有 `Stopping openclaw-gateway.service` 记录
- 这次是 systemd 直接 stop（后面紧跟 session-backup.service 完成）

#### 4. **7 次 SIGTERM 源头未知**
- journalctl **完全没记录**这 7 次 "Stopping openclaw-gateway.service"
- gateway 自己的 log 显示 `signal SIGTERM received`（不是 SIGUSR1）
- 这 7 次 SIGTERM **不是** systemd stop（systemd 会记录 "Stopping ..."），**不是** SIGUSR1（log 显示是 SIGTERM），**不是** resume-watch（没审计记录）
- **最可能：OpenClaw 内部某个 watchdog / 健康监控 / lane task error 处理 主动 kill 自己的子进程**
