## H2：定位上午+7/2 同型 bug 根因 ✅

### 抓到的根因（从 `/tmp/openclaw/openclaw-2026-07-06.log` 663KB 解析）

#### 上午 10:03-10:06 的 4 次失败链
```
10:03:26  MiniMax-M3 → stopReason=length  → "Agent couldn't generate a response"
10:03:28  retry 1/2 with visible-answer continuation
10:04:19  long-running session 149s, active tool=process
10:04:21  fall back to litellm/agnes-2.0-flash → succeeded ✅
10:04:29  第 2 次 incomplete turn → fallback → succeeded
10:04:57  第 3 次 incomplete turn → fallback → succeeded (10:05:55)
10:05:58  第 4 次 incomplete turn
```

#### 关键证据
- **"reasoning-only assistant turn detected"**：模型只输出了 reasoning（思考过程），没生成正文 → 触发 retry
- **"stopReason=length"**：模型输出超长被截断
- **fallback 工作正常**：4 次全部成功切换到 agnes-2.0-flash
- **"long-running session 149s, active tool=process"**：retry 期间 tool call 被 queue 起来

#### 根因结论
不是"网络超时误报"（知乎 bug 那种），而是：
1. **MiniMax-M3 在大上下文（21k tokens）下容易 stopReason=length**
2. **reasoning-only 输出**让 Gateway 触发 retry
3. **retry 期间的 tool call 都被 queue 起来** → 用户感觉"工具被卡住"
4. **fallback 到 agnes 后系统恢复**（救命了）
5. **如果主模型 + agnes 都挂** → ollama 那个死节点（GitHub #100460）会让整个 fallback 链 dead-end

#### 上午"污染 + IO 没了"现象的真相
- 不是我以为的"上下文被污染"（那种神秘 bug 没那么玄）
- 而是**4 次主模型失败 + 4 次 fallback 切换 + retry 期间 tool queue**的视觉/体感效果
- 系统层面"没崩"，但体感"工具卡了"

### 与 7/2 那次的差异

- 7/6：症状完全一样，10 分钟内**靠 fallback 救回来了**

## H3：自动诊断补丁 ✅

### 新组件
- `scripts/jarvis-io-watchdog.py` (10KB, Python 3 stdlib only)
- 新 cron job `Jarvis IO 看门狗`，每 5 分钟跑一次（ID: `bf6e76ff-0d91-42b5-bf38-28c7c0eca251`）

### 检测逻辑
扫描 gateway 日志最近 5 分钟内：
- `incomplete turn` 命中 ≥ 3 → L1（建议清缓存）
- `incomplete turn` 命中 ≥ 5 → L2（建议 reset session）
- `DrainingError` 出现 → 立即 L1
- `stream ended` 出现 → 立即 L2（fallback 链有死路）
- `context overflow` / `prompt too large` 命中 ≥ 2 → L1
- 工具失败模式 ≥ 2 → L1

### 行动
- L1/L2 告警 → 写 `/tmp/jarvis-alert.txt`（含具体恢复命令）
- 30 分钟冷却：同一级别告警不重复发
- **不直接 restart gateway**（按崩案例 CASE-20260706-003）
- **不直接写 openclaw.json**（按崩案例 CASE-20260616-002）

### 验证
- 60min 窗口实测：4 次 incomplete → 严重度 L1 ✅
- 告警文件正确生成（含时间戳、级别、行动建议）✅
- 5min 窗口（系统稳时）：0 命中 → INFO ✅
- cron 注册成功，trigger 后 status=ok ✅

### 为什么用 cron 不嵌 Mark42 engine
- Mark42 templates 是 Python 写死的，重写代价大
- OpenClaw cron 已经是稳的（手动 trigger 跑通）
- watchdog 脚本独立、好测试、好回滚
- cron 失败会自己告警（不用我加）
