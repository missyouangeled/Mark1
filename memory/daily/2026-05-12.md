- 已把“主会话稳定性默认执行顺序”补进 `AGENTS.md`：先给短反馈 → 检查监工分身是否在位 → 判断是否阻塞前台 → 必要时立刻卸到后台任务分身 → 前台继续正常对话 → 媒体按真实可用验收 → 收尾时区分任务分身与监工分身。

- 在对这些规则做一致性审计后，又补修了几处容易继续造成误解的口径：第一，`AGENTS.md` 的总入口不再只写成“用户明确要求前后台分流时才适用”，而是扩展为“用户明确要求，或当前轮已经属于工作型任务并需要防卡住时”都适用；第二，`main-supervisor-lite` 模板里明确了“监工默认在，普通任务分身按是否阻塞前台决定是否需要”，避免误读成“只要是工作型任务就必须再开一个普通任务分身”；第三，监工数量口径统一为“整体数量是 0 或 1”，工作型任务期应为 1，无后台任务满 2 分钟后可回到 0；第四，把“保留最新且健康的一个”里的“健康”补成了更明确的判断：未失败、未超时、未被杀，且近期仍有活动或可见进度。
- 用户随后又追问了一个关键边界：如果网络不好、与模型通信阻塞，或者更底层链路出问题，监工是否还能起作用。当前结论已明确写进规则：监工是兜底层，不是独立于 Gateway / 渠道 / 模型链路之外的万能保险。若监工已提前在位，且 Gateway、当前渠道投递、前端连接、以及至少一条模型调用链路仍正常，则它通常还能继续起作用；但如果问题已经升级为 Gateway 整体卡死、前端连接断开、渠道投递失效、或同一路由模型调用全局阻塞，监工也可能一起受影响。
- 最后又继续把这套规则补成了一张真正可查的场景判定表：纯日常聊天 = 可不开监工；工作型但不阻塞前台 = 监工在位、任务分身可不开；工作型且阻塞前台 = 监工在位 + 按需开任务分身；已有任务分身 = 必须有监工；无后台任务 2 分钟 = 监工可回到 0；拿不准时默认按工作型任务处理。这样后续执行时更接近查表，而不是靠临场猜。
- 在此基础上，又继续补了“口袋版速查规则”：把最关键的 10 条压缩进 `AGENTS.md` 的前面和 `TOOLS.md` 的 automation 区，确保以后每次进入工作流时，我更容易先撞见这套规则，而不是做完一半才想起来。
