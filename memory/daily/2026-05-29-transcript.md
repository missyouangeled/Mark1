## 🔬 根因分析

QMD vsearch 的调用链是：

```
qmd vsearch → vectorSearchQuery() → expandQuery() → ensureGenerateModel() → 加载 1.2GB LLM
```

**vsearch 并不是"只用 embedding 模型"**——它在搜索前也会调 `expandQuery()` 做查询扩展，而这个函数需要 LLM 模型。这就是为什么：
- 120 秒超时（加载 1.2GB 模型 + CPU 推理）
- 触发 OOM（embedding 319MB + LLM 1.2GB + reranker 610MB = 2.1GB+）

## 🔬 根因分析

QMD vsearch 的调用链是：

```
qmd vsearch → vectorSearchQuery() → expandQuery() → ensureGenerateModel() → 加载 1.2GB LLM
```

**vsearch 并不是"只用 embedding 模型"**——它在搜索前也会调 `expandQuery()` 做查询扩展，而这个函数需要 LLM 模型。这就是为什么：
- 120 秒超时（加载 1.2GB 模型 + CPU 推理）
- 触发 OOM（embedding 319MB + LLM 1.2GB + reranker 610MB = 2.1GB+）

## 🔍 根因

你看到的那个错误 `Write: to memory/daily/2026-05-29.md failed`，发生在 **会话压缩（pre-compaction）触发的 memory flush 事件**里。

具体原因是一个**路径不匹配**：

| 谁 | 期望的路径 | 实际写入路径 | 结果 |
|---|---|---|---|
| **Memory flush 守卫** | `memory/2026-05-29.md` | — | 只允许这个 |
| **AGENTS.md 规范** | — | `memory/daily/2026-05-29.md` | 这是我们的归档结构 |

当压缩事件触发 flush 时，守卫只放行 `memory/YYYY-MM-DD.md`（扁平路径），而我们日常用的是 `memory/daily/YYYY-MM-DD.md`（目录分层）。写 `daily/` 子目录时就被拒绝了。

这就是为什么**偶尔出现**——它只在会话压缩触发 flush 时发生，不压缩就没这个问题。刚才我们聊了很长时…

## 🔬 根因分析

QMD vsearch 的调用链是：

```
qmd vsearch → vectorSearchQuery() → expandQuery() → ensureGenerateModel() → 加载 1.2GB LLM

**vsearch 并不是"只用 embedding 模型"**——它在搜索前也会调 `expandQuery()` 做查询扩展，而这个函数需要 LLM 模型。这就是为什么：
- 120 秒超时（加载 1.2GB 模型 + CPU 推理）
- 触发 OOM（embedding 319MB + LLM 1.2GB + reranker 610MB = 2.1GB+）

## 🔍 根因

你看到的那个错误 `Write: to memory/daily/2026-05-29.md failed`，发生在 **会话压缩（pre-compaction）触发的 memory flush 事件**里。

具体原因是一个**路径不匹配**：

| 谁 | 期望的路径 | 实际写入路径 | 结果 |
|---|---|---|---|
| **Memory flush 守卫** | `memory/2026-05-29.md` | — | 只允许这个 |
| **AGENTS.md 规范** | — | `memory/daily/2026-05-29.md` | 这是我们的归档结构 |

当压缩事件触发 flush 时，守卫只放行 `memory/YYYY-MM-DD.md`（扁平路径），而我们日常用的是 `memory/daily/YYYY-MM-DD.md`（目录分层）。写 `daily/` 子目录时就被拒绝了。

这就是为什么**偶尔出现**——它只在会话压缩触发 flush 时发生，不压缩就没这个问题。刚才我们聊了很长时…
