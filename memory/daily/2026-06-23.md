## 下午：DeepSeek 缓存优化方案（4 大杠杆调研）

### 背景
- 昨天 V4 Pro 账单 ¥81，输入未命中 22.6M token（30%），未命中占输入成本 98%
- 目标：减少未命中 token

### 4 大杠杆调研结果
1. **杠杆 1（移动态信息）**：原计划从 system prompt 移除时间戳/轮次/会话ID → **实际不需要做**，OpenClaw 顶层文件本来就是纯静态的，没有动态字段被注入
2. **杠杆 2（加 cache_control ephemeral 标记）**：原计划通过 LiteLLM 注入 → **实际做不了**，OpenClaw 走 openai-completions API 直连 DeepSeek，不经 LiteLLM proxy，cache_control 字段不传
3. **杠杆 3a（关 thinking）**：点点之前明确不想关，跳过
4. **杠杆 3b（瘦 system prompt）**：✅ **已落地**——TOOLS.md 从 56 行精简到 32 行（-43%），新建 TOOLS_INDEX.md 接住详细文档
5. **杠杆 4a（路由分流）**：点点否决，不做

### 已落地改动
- `TOOLS_INDEX.md`（新建，93 行）——修失链 + 工具文档二级索引
- `TOOLS.md`（精简 56→32 行，-43%）——杠杆 3b 落地

### 真实未命中根因（OpenClaw 框架行为）
剩下的 30% 未命中是 OpenClaw 框架的固有行为：
- 多轮对话小修小改
- tool 调用结果插入（让后续 user message 起始位置改变）
- compaction 触发（3MB 阈值压缩历史）

这些改不动。
