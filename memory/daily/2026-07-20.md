**memory_search 深度修复（核心问题）：**

上午 08:42 记录"memory_search 已修复"是重启 embed-sidecar 后的临时恢复，但根本问题没解决。

**根因分析：**
1. `memory.backend = "qmd"` -> OpenClaw 用 qmd 做记忆搜索
2. OpenClaw 有自己的 qmd home：`~/.openclaw/agents/main/qmd/xdg-cache/qmd/`
3. 之前误以为 qmd 用的是 `~/.cache/qmd/`（用户 home），花了很多时间在错误的地方建索引
4. OpenClaw 的 qmd 索引：120 文件已索引，但向量嵌入基本没做（只有 384/1500+）
5. `searchMode = "vsearch"` 要求向量搜索，但向量不够 -> 超时
6. 两个 qmd embed 进程同时跑，各吃 ~500MB 内存 -> 系统内存不够 -> Gateway 被 cgroup OOM kill
7. Gateway 反复重启，每次都把正在进行的会话中断

**修复措施：**
1. 将 `searchMode` 从 `"vsearch"` 改为 `"search"`（BM25 模式），不需要向量也能搜索
2. 杀掉所有 qmd embed 进程释放内存
3. 用 `systemd-run` 独立单元跑 embed，限制 `MemoryMax=450M`、`Nice=10`，不影响 Gateway
4. memory_search 恢复正常：2.6s 返回结果 ✅

**经验教训：**
- OpenClaw 的 qmd home 在 `~/.openclaw/agents/main/qmd/`，不是 `~/.cache/qmd/`
- 以后查 qmd 状态必须用 `XDG_CACHE_HOME` 指向 OpenClaw 的路径
- qmd embed 是内存密集型操作，不能在 Gateway cgroup 内直接跑
- BM25 模式（search）不需要向量，可以立即工作；vsearch 需要完整嵌入才能正常工作
- embed 进程必须用 systemd-run 独立跑 + 内存限制

### 12:30-13:25 vsearch 超时根因终局修复

**问题：** 上午把 searchMode 改回 vsearch 后，memory_search 再次 15s 超时。

**完整根因链：**
1. `node-llama-cpp` 在本平台没有预编译二进制，首次 vsearch 要从源码编译 llama.cpp（耗时 6 分钟）
2. 编译完成后，vsearch 单次运行需要 ~10s 加载 300M 嵌入模型，刚好卡在 15s 硬超时边界
3. OpenClaw memory_search 的 15s 超时是**硬编码**（`MEMORY_SEARCH_TOOL_TIMEOUT_MS = 15e3`），不可配置
4. Gateway cgroup MemoryMax=1.5G 不够（vsearch 需 ~1.3G + Gateway 自身 ~200M）
5. **最关键**：mcporter daemon 未启用，每次 memory_search 都 spawn 新 qmd 进程冷启动

**最终修复：**
1. 用独立 systemd-run 单元完成 `node-llama-cpp` 编译并缓存二进制（~6 分钟）
2. 停用并 disable `openclaw-embed-sidecar.service`，释放 ~1GB 内存
3. Gateway cgroup 内存限制从 1.5G 调到 3G（`MemoryHigh=2G, MemoryMax=3G`）
4. **启用 mcporter daemon**：配置 `memory.qmd.mcporter = { enabled: true, serverName: "qmd", startDaemon: true }`
5. 创建 mcporter 配置文件 `config/mcporter.json`，设置 qmd server 为 `lifecycle: keep-alive`
6. searchMode 恢复为 `vsearch`

**效果：**
- memory_search 从 15s 超时 -> **2.5s 稳定返回**（6 条结果）
- qmd 通过 mcporter daemon 常驻，不再每次冷启动
- Gateway 重启后 OpenClaw 自动拉起 mcporter daemon
- 内存使用正常：Gateway 596M/3G，系统总用 2.5G/7.7G

**关键文件变更：**
- `/home/missyouangeled/.openclaw/openclaw.json`：新增 `memory.qmd.mcporter` 配置
- `/home/missyouangeled/.openclaw/workspace/config/mcporter.json`：新建，mcporter qmd server 配置
- `/home/missyouangeled/.openclaw/workspace/config/systemd/openclaw-gateway-resource-limits.conf`：MemoryHigh 2G, MemoryMax 3G
- `/home/missyouangeled/.config/systemd/user/openclaw-gateway.service.d/resource-limits.conf`：同步覆盖

**经验教训（长期记住）：**
- OpenClaw memory_search 15s 超时是硬编码，不可配置 -> 解法是让 qmd 常驻而非加超时
- mcporter daemon 是 OpenClaw 推荐的 qmd 常驻方案，文档在 `memory-config.md` 的 mcporter integration 章节
- `node-llama-cpp` 首次运行要编译 llama.cpp，必须用独立进程完成并缓存
- qmd vsearch 需要完整向量嵌入才能正常工作，BM25 模式（search）不需要
