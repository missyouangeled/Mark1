# 2026-06-09 每日记录

## 主要工作：Gateway 稳定性根治

### 问题
本地健康检查间歇性报告"Gateway 不可达"（CRITICAL），用户询问 Gateway 是否容易断开。

### 调查过程
1. 检查 health-diagnostic.log 历史记录：6 月以来共 5 次 CRITICAL 事件，持续 3-7 分钟，均已自动恢复
2. 关键发现：Gateway 服务进程从未崩溃（systemd 始终 active/running），是 HTTP/WebSocket 层探测超时
3. 深入 Gateway 内部代码：健康探测 WebSocket 超时仅 **3 秒**（`timeoutMs: 3e3`）
4. 分析 06-08 晚间日志：告警前 Gateway 内部操作严重变慢——`usage.cost` 17.6 秒、`sessions.list` 6.5 秒、`chat.history` 7.0 秒
5. 定位根因链：**内存膨胀（>1GB）→ 触发 swap I/O → Node.js 事件循环被阻塞 → WebSocket 3s 探测超时 → 报告「不可达」**

### 解决方案
- **MemoryHigh=1.2G + MemoryMax=1.5G**：在 Gateway 内存接近 1.2G 时 systemd 开始限流，阻止进入 swap
- **不设 CPU 限制**：Gateway 是 I/O 密集型，CPU 限制会损害正常响应速度
- **安全重启脚本** `scripts/openclaw-gateway-safe-restart.py`：保存工作区 → 通知 → 重启 → 验证恢复
- **启动链自动同步**：`ensure-resource-limits.conf` ExecStartPre 每次启动前从仓库同步配置

### 新增核心方法论
点点明确要求把今天的理念写入长期规则：
> 遇到问题必须追根因，禁止躲避式修复。不能通过调参数、放宽阈值、增加重试等"绕开问题"的方式蒙混过关。

流程：收集数据日志 → 定位根因链 → 给出具体方案 → 讨论确认 → 实施彻底修复 → 验证不复现。
