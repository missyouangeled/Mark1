## 12:55-13:05 trajectory 压缩（OpenClaw session-size-watcher 的真实问题被根治）

### 问题

session-size-watcher 从 04:00 一直在告警"trajectory 7.22MB > 阈值 5MB"，但 watcher **不主动通知主会话**（之前救命 1 v3 已经接通了告警转发）。**真正的问题**：watcher **不能自动压缩 trajectory** —— 它只清"陈旧"的 trajectory（mtime 早 300s 以上），**当前活跃 trajectory 永远不会被它清**。

### 调研 trajectory 是什么

- OpenClaw runtime 写的 audit log，每个 agent.run 一组事件
- 7 类型事件：session.started, trace.metadata, context.compiled, prompt.submitted, session.ended, model.completed, trace.artifacts
- `context.compiled` 每个 ~120KB（重复存 tools schema 27KB + 系统 prompt）
- `prompt.submitted` 每个 ~90KB（重复存 prompt + tools）
- `model.completed.messagesSnapshot` 每个 ~97KB（每次 model 调用的完整消息历史）

7.56MB / 157 事件，平均每事件 50KB。

### 写脚本：scripts/compress-trajectory.py

- 默认 dry-run，必须 --execute 才真写
- 策略：保留所有 lifecycle，删 context.compiled/prompt.submitted 的旧版（保留最近 3）
- 先备份 `.pre-compress-<HHMM>`，写新文件用 `.tmp` + rename

### 执行

- 主会话 trajectory: 7.56 MB (157 事件) → **4.28 MB** (117 事件, -48%)
- gateway 进程未持有 trajectory fd（事件驱动），安全
- 备份保留: `.pre-compress-1258`

### 效果（自动验证）

session-size-watcher 之后跑：
- trajectory_mb: 7.22 → **4.28** ✅
- total_dir_mb: 28.47 → **23.25** ✅
- level: CRITICAL → **INFO** ✅
- alerts.json 不再写 trajectory 告警 ✅
- watcher 还自动清理了 16 个旧文件 / 5.26 MB

## 13:01-13:20 联网复查 + 救命 1 v4/v5 改进

点点的"再检查一遍，联网搜索确定方案。复查一次。"要求。

### 联网搜到的关键信息

1. **issue #34919 确认我们做的是对的标准做法** —— "We had to write a custom flush prompt to fully resolve the problem." 官方还没修。

2. **OpenClaw 自带 `session.maintenance` 自动机制** —— mode/pruneAfter/maxEntries/maxDiskBytes 都内置。我们的 mode=enforce, pruneAfter=14d, maxEntries=120 都在跑。但 maxDiskBytes 没配。

3. **官方命令 `openclaw sessions cleanup --enforce`** —— 跑了一次，会清 unreferenced artifacts。dry-run 显示会清 3 个文件 / 24KB。

4. **官方命令 `openclaw sessions export-trajectory`** —— trajectory 是 debug 工具，不是必需保留。我手动压是对的（活跃 trajectory 官方不会清）。

5. **GitHub issue #81631 / cli/cron 文档** —— v6.11 已支持 `--command-cwd` + `--command-argv`！**救命 1 之前用绝对路径是兜底方案，更佳方案是 --command-cwd 显式设 cwd**。

### 救命 1 v4：脱离内联 argv，独立脚本 + --command-cwd

**变更**：
- 新增 `scripts/emergency1.sh`（独立 bash 脚本）
- 新增 `scripts/emergency1-watcher-read.py`（独立 python 解析 alerts.json）
- cron 用 `--command-cwd /home/missyouangeled/.openclaw/workspace` + `--command "bash .../emergency1.sh"`
- timeout 30s → 60s

**实测**：静默+告警分支都通过。

### 复查救命 1 v4 发现 mark read 缺失

**隐患**：watcher 的 `raise_alert` 总是把 alerts.json `unread=True`。救命 1 v4 读 alerts 后不 mark read，导致 watcher 下一轮 raise_alert 又把 unread 设回 True，**标记永远清不掉**。

**修复**：emergency1-watcher-read.py 读完后立刻 mark read（用 `.tmp + rename` 原子写）。

### 救命 1 v5：加 trajectory 监控

**新增**：第 1b 段，检查 `${SESSION_ID}.trajectory.jsonl` 大小 > 5MB 时告警（独立 cooldown 60min）。

**实测**：用 dd 给 trajectory 加 2MB padding → 触发告警 → 注入主会话 → 还原 trajectory。
