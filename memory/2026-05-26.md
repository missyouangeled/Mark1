## 会话摘要

点点早上 8:43 到公司，聊了一会日常。随后让我分析公司 Linux 机上的 OpenClaw 系统是否完善，并给出优化方案。给了最高权限后，我按"稳→准→狠"顺序推进了一整轮系统优化。

## 完成的改动

### 1. 修复 systemd 启动顺序循环依赖（🔧 bug fix）
- **根因**：`openclaw-infos-handle-sidecar.service` 设置了 `After=default.target`，与 gateway 的 `Before=default.target` + unified-proxy 的依赖链形成循环
- **修复**：将 `After=default.target` 改为 `After=network-online.target`
- **效果**：消除启动不确定性，journal 不再报 ordering cycle 错误

### 2. 新增自动化临时文件清理（✨ 新功能）
- **脚本**：`scripts/openclaw-cleanup-temp.sh`
- **规则**：语音 mp3/wav 超过 4h 删除、infos-handle 输出超过 4h 删除、通用 tmp 超过 24h 删除
- **调度**：systemd timer，每 30min
- **首次清理**：27 个文件，34.57MB

### 3. 新增磁盘空间监控（✨ 新功能）
- 在 `scripts/openclaw-local-health-diagnose.py` 新增 `collect_disk_usage()` 函数
- 监控 `/` 和 `/mnt/data`：≥80% warn、≥90% critical、根盘低于 8G 安全线告警
- 当前：根盘 75.8%（11.8G 余量），数据盘 39.1%

### 4. 修复 NO_REPLY 假阳性异常（🔧 bug fix）
- `frontstage-recovery-watch.py` 不再把 NO_REPLY 的 assistant turn 报成 `assistant_turn_missing_visible_text`
- 添加了 `_turn_is_no_reply()` 函数检测静默回应

### 5. Watcher 整合：8 timer → 4 timer（🏗 架构优化）
- **前台保护器** (`frontstage-guardian`)：合并 recovery(15s) + responsiveness(15s) → 每 20s
- **健康采集器** (`health-collector`)：合并 supervisor(30s) + broker(60s) + local-health(5min) → 每 60s 轻量 + 每 5min 完整
- **生命周期维护器** (`lifecycle-maintainer`)：合并 daily-transcript(5min) + cleanup(30min) → 每 5min 转录 + 每 6 次清理
- **resume 恢复**：保持独立，每 60s
- 所有 wrapper 通过子进程调用原脚本，不侵入原逻辑

### 7. 卡住会话自动恢复系统（🛡️ 新功能）
- **根因**：截图发送时 `nvidia/nemotron-nano-12b-v2-vl` 实际 ctx=4096（注册 200k），图片太大撑爆上下文→主会话完全卡死无响应
- **解决方案**：在 `openclaw-health-collector.py` 内置 `AutoRecoveryManager` 类
  - 检测 stuck session（>120s 无新 turn，主会话 >60s 即 critical）
  - 分级恢复：检测 → 二次验证（chat.send）→ 重量恢复（gateway restart）
  - 冷却保护：两次自动重启间隔 ≥ 10 分钟
  - 排除列表：boot- 前缀 sessionId、subagent/cron sessionKey
- **修复 imageModel**：`nvidia/nemotron-nano-12b-v2-vl`（4k ctx）→ `nvidia/google/gemma-4-31b-it`（131k ctx）
- **关闭 boot-md hook**：`hooks.internal.entries.boot-md.enabled = false`
- **实战验证**：11:48:37 自动触发 gateway restart，11:49:29 成功恢复
- **配置文件**：`/home/missyouangeled/.openclaw/openclaw.json`
