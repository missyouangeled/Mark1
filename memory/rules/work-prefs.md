## Git 与变更管理

- 每次做完系统性修改，及时把相关配置提交并推送到 GitHub 备份
- 实际改了功能、打了补丁、修了脚本、接了自动链路 → 默认自动补「变更流水」到 `docs/通用-OpenClaw-补丁变更流水.md`；已构成正式补丁则同步更新补丁注册表/重建清单/升级后自检清单
- 暂时不适合进入正式补丁体系但对后续排查仍重要的修改 → 同步记进 `docs/通用-OpenClaw-非正式修改备忘录.md`

## 后台分身与清理

- 后台分身如果没有任务就及时关闭
- 明显会耗时较长的任务 → 优先自动开分身放到后台继续跑
- 每轮任务做完后默认做收尾清理：关闭没任务的分身、取消/维护卡死或陈旧后台任务、按保留规则清理无用旧会话

## 会话清理

- 「清会话」→ 默认按分层会话清理方案执行：先保留当前正在使用的会话树与主会话，优先清掉陈旧 dashboard/已结束/失败/超时/僵尸 running 的 subagent/旧直聊会话；先移除 sessions 索引和主 jsonl，再清理 trajectory/checkpoint/bak/reset/deleted 残留

## Thinking 推理强度

- 普通问答/简单解释/明确路径的小修改/常规命令 → 默认或低
- 复杂排错/架构取舍/大改前规划/需综合多份上下文再决策/高风险恢复/安全/一致性问题 → 开高
- 任务又长又复杂 → 优先放后台分身并在分身上使用更高推理，不拖慢主会话

## 前后台协作

- 默认「前台只负责正常对话，后台有结果再主动插播进度」
- 主会话必须保持沟通顺畅，不能像没反应
- 除纯日常聊天外，工作型任务默认保留或快速拉起轻量后台分身待命
- 复杂排错时应默认优先开启后台分身
- 持续开发/连续实现/长链路改动 → 默认优先真卸到后台，不只在监工开着但没东西盯

## 监工规则

- 监工 0 或 1 单例：工作型任务期应为 1，无后台任务持续约 2 分钟后可回到 0
- 监工分身与普通任务分身区分对待，保留标识 `main-supervisor-lite`
- 监工唯一标号格式：`main-supervisor-lite@<runtime-host>`
- 不允许出现「只有任务分身、没有监工」的状态
- 任务分身空返回 → 监工先向主会话报告异常并接手检查
- 任务分身异常结束 → 监工先报告「异常结束，正在修复」再排查修复
- 3 分钟无可见产出 → 监工必须补一条简短进度
- 完成后主动汇报：任务做完或达到阶段性完成点都要主动汇报
- 监工不依赖 `thread:true` / `mode:"session"` 的线程绑定会话，优先用一次性 `sessions_spawn(mode:"run", context:"isolated")`
- 上一轮后台任务完成后约 10 分钟等待接续任务窗口，过期自动关闭

## 主会话定义

- 「主会话」= 用户当前正在说话的这个会话（WebChat/Control UI 当前页面）
- 新开 dashboard 前台 → 监工/后台回报的「当前会话」随之前移到新的 dashboard 前台
- 主会话稳定性含媒体能力：附件/音频/视频/图片必须默认可查看可播放可打开

## 设备与地点

- 设备→地点概念：`missyouangeled-VMware-Virtual-Platform` = 公司(Linux)；`TABLET-EH5U3C01` = 掌机(Windows)；`DESKTOP-RQAR1FL` = 公司老电脑(Windows, GTX 1070)
- 设备标识只用于知道自己当前在哪台机器，不拆分规则集
- 共享仓库同步后 `openclaw gateway restart` 作为默认更新流程
- 「同步这台机器」「更新这台机器」→ 默认理解为 `git pull --ff-only`，必要时 `openclaw gateway restart`
- 掌机更新由本地 OpenClaw 自主完成（非手动点更新脚本）
- 新机器自动登记 HOST_CONTEXT.md，高置信度时自动加环境标签，不确定则记为「未归类」
- 系统相关文档必须标设备/系统（如 `公司(Linux)`、`掌机(Windows)`）
- 方案/系统配置/日常工作记录三类在 Git 中绝对分开

## 跨设备文件交付

- 跨设备文件拷给宿主机 → 临时 HTTP 下载服务，发完整 URL
- 宿主机文件拷到当前机器 → 临时上传页（带随机 token）

## 双盘规则（公司 Linux）

- 先预估大小 → 新增占用 ≥1G 或可能把根盘压到 8G 安全线以下时，优先放 `/mnt/data`

## 视频下载工作流

- 优先按 `docs/通用-视频平台下载工作流.md` 执行
- 先试站内直达 → 被验证码/登录墙/作品流异常/懒加载拦住时切到外搜定位公开页面
- 目标位不写死，设计成变量（random/last/index:N/video:ID）
- 作者主页作品流异常时不准误选热点/推荐视频作为作者作品；明确报阻塞并切回公开视频页

## 候选整理器

- `scripts/download-platform-video.py --list-only` 做候选整理并导出结构化 JSON + 候选 URL 文本文件

## 修改类任务规则

- 第一步查已有功能/补丁/入口；第二步检查冲突并及时回报；第三步改完接正式补丁/自动链路

- 四项必过：数据安全、系统稳定性、运行负担、依赖审查 → 详见 `docs/通用-安装前安全审查机制.md`

## 大批量操作防溢出（2026-06-03）

- 先判断总量 → 与模型沟通容量 → 接受不了就分批 → 聊天只放摘要和计数 → 详见 `docs/通用-大批量操作防上下文溢出规则.md`

## Scratch 数据盘临时区域（2026-06-05）

- 大文件操作/临时测试/中间产物默认用 `/mnt/data/openclaw/scratch/`
- 完成后清理 scratch 过期文件 + 内存/缓存清理

## 大工程稳定运行方案（2026-06-05）

- 按 `docs/通用-OpenClaw-大工程稳定运行方案.md` 执行
- 前台只看摘要，后台分身做重活，详细结果落 scratch
- 批量改名执行前必须先做目标名冲突预扫
- `0_PropsGroups/` 文件夹名不做「删除前导下划线」处理

## 会话大小监测事件驱动（2026-06-08）

- 每次收到用户消息后，以 `exec(background=true)` 运行 `scripts/openclaw-session-size-watcher.py --gate-seconds 60 --print-json`
- 门控保证 60 秒内不重复实扫

## 问题追根因方法论（2026-06-09）

- 遇到问题必须追根因，禁止躲避式修复
- 流程：收集数据与日志定位根因链 → 分析并给出具体方案 → 与用户讨论确认 → 实施彻底修复 → 验证不再复现
- 禁止通过调参数、放宽阈值、增加重试等绕开问题

## 问题解决标准流程（2026-06-10）

- 六步标准流程：搜索确定问题 → 分析 → 找解决办法 → 给初步方案 → 自我审核（逻辑悖论/牵连影响/遗漏/bug）→ 有bug修后再审/无bug直接交付
- 禁止跳过自审环节直接给方案
- 详见 `docs/通用-问题解决标准流程.md`
