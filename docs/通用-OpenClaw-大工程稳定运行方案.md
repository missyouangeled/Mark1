# 通用 - OpenClaw 大工程稳定运行方案

- 适用机器:通用
- 系统 / OS:通用
- 文档类型:工作流 / 稳定性治理方案
- 生效日期:2026-06-05

## 目标

当任务具有以下任一特征时,按"大工程"处理:

- 大量文件扫描 / 批量改名 / 大量 `.meta` 同步
- 长时间运行(通常 > 3 分钟)
- 需要大量中间产物、计划文件、差异报告
- 需要分身、验证、回滚预案
- 容易拖慢主会话、触发 watcher/timer 压力、让前台出现"像没反应"体感

本方案目标不是让任务"更快",而是让系统 **持续稳定、前台顺畅、结果可追溯、异常可恢复**。

---

## 一、核心原则

### 1. 主会话轻量化

主会话只负责:
- 收需求
- 看摘要
- 做关键决策
- 接收进度/异常/结果

主会话不负责:
- 长时间同步跑重活
- 承载长清单/大段 diff/大批量文件列表
- 保存大工程的详细中间状态

### 2. 详细信息文件化

- 聊天里只放摘要、计数、样本、结论
- 详细内容一律落地文件
- 文件系统是持久记忆,聊天只是临时窗口

### 3. 临时产物数据盘化

大文件、计划文档、中间日志、差异报告、恢复清单,默认落:

`/mnt/data/openclaw/scratch/`

建议进一步分层:

```text
/mnt/data/openclaw/scratch/
  unity-renames/
    2026-06-05-props/
    2026-06-05-wall/
  reports/
  logs/
  temp/
```

### 4. 重活后台化

- 大工程默认优先用后台分身执行
- 主会话保持可交流
- 详细结果由后台落文件,前台只回摘要

### 5. 冲突前置化

任何批量改名/归并/清理,执行前必须先做"目标名冲突预扫",避免覆盖。

---

## 二、大工程标准流程

### 阶段 A:开工前预检

必须先做这 6 项:

1. **总量预估**
   - 文件数/条目数
   - 预期输出行数
   - 是否超过聊天安全线(200 条)

2. **系统余量检查**
   - `free -h`
   - `df -h / /mnt/data`
   - 当前是否有活跃后台任务 / 高负载 watcher

3. **落点规划**
   - 结果文档、差异报告、候选清单、恢复列表 → scratch
   - 不把大中间产物堆在 workspace 或聊天里

4. **冲突预扫**
   - 原文件名 → 新文件名映射表
   - 检查是否多个文件会映射到同一个目标名
   - 有冲突先停,不直接执行

5. **回滚入口确认**
   - git / 原始副本 / 恢复列表 至少确认一种
   - 不能"先改了再说"

6. **执行模式选择**
   - ≤50 条:可前台轻量执行
   - 50~200 条:分批
   - >200 条或 >3 分钟:后台分身 + 文件落地

### 阶段 B:执行中

1. 前台只保留摘要
2. 后台分身执行扫描、改名、验证、写计划
3. 每批 ≤200 条
4. 每一批都输出:
   - 分类
   - 数量
   - 样本 2~3 条
   - 结果文件路径
5. 出现以下情况立即停下并报异常:
   - 目标名冲突
   - 文件覆盖风险
   - `.meta` 不一致
   - 目录结构与规则冲突
   - watcher/timer 报错频率升高

### 阶段 C:收工后

固定做 7 件事:

1. 生成完整结果文档
2. 备份关键文档到 scratch
3. 更新 daily / MEMORY / 必要规则
4. 清理无任务分身
5. 清理临时缓存与 pyc / tmp
6. 修剪 journald + 释放 kernel cache
7. 做一次系统健康检查

## 三、稳定性分层方案

### 第一层:上下文层

问题:主会话聊久了、文件列表过多、计划过大,模型响应会变钝。

措施:
- 严格执行"聊天只放摘要,详细内容落文件"
- 超过 200 条即分批
- 长链路任务默认后台化
- 同一会话多次大批量操作后,必要时主动建议开新会话续做

### 第二层:文件与 IO 层

问题:大量扫描、改名、写文档、同步 `.meta` 会造成 IO 压力。

措施:
- 大文件/计划/中间产物默认放 scratch
- 工作文件按项目目录分层存放
- 不在 workspace 中长期堆积重型产物
- 完成后清理过期 scratch 内容(但保留用户明确要求保留的项目)

### 第三层:运行时层

问题:分身、timer、watcher、日志收集、systemd oneshot 在高峰期会互相抢资源。

措施:
- 大工程期间优先后台分身,不在主会话同步跑重活
- watcher/timer 频率保持克制,避免高频重叠
- 已知案例:`openclaw-frontstage-guardian.timer` 20s 过密,在 CPU 高峰时会导致前次运行未完全结束,产生 systemd failed 观感
- 建议:高峰期 watcher 以"少而稳"为先,不追求过密检测

### 第四层:scratch 留存与清理层

问题:临时产物如果一直堆着,会持续占用数据盘并增加管理负担;但重要项目目录又不能误删。

措施:
- scratch 采用 `.keep` 保留标记
- 任一顶层项目目录下只要存在 `.keep`,整个顶层目录都视为保留
- 清理前先 dry-run 预览
- 默认只清理超过阈值天数且未标记保留的目录/文件

### 第五层:收尾层

问题:任务做完但分身、缓存、旧日志、旧会话没清,会在下午累积成迟缓体感。

措施:
- 每轮大工程后做固定收尾
- 包含:分身清理、旧会话清理、scratch 过期清理、pyc 清理、tmp 清理、journald 修剪、kernel cache 释放

## 四、批量重命名类任务的硬规则

### 执行顺序(强制)

1. 扫描原始文件
2. 生成"原名 → 目标名"映射表
3. **先检查冲突**
4. 生成 dry-run 报告
5. 用户确认后再正式执行
6. 执行后做文件数 / `.meta` / 目录结构验证
7. 写结果计划文档

### 必须拦截的风险

- 删除后缀后撞名(如 `#PaintWhite` / `#BrickIndustrial_06`)
- 删 `Type_` / `Type-` 后撞名
- 大小写统一后撞名
- 多来源文件合并到同一目录时重名
- `.meta` 没同步

## 五、推荐执行模板

### 大工程开始前

```bash
python3 scripts/openclaw-heavy-task-start.py <目标目录> --task-name <任务名>
python3 scripts/openclaw-system-summary.py --print-human
free -h
df -h / /mnt/data
```

### 大工程完成后

```bash
python3 scripts/openclaw-scratch-cleanup.py --dry-run --print-kept
python3 scripts/openclaw-heavy-task-finish.py
```

## 六、验收标准

满足以下条件,才算"稳定":

- 主会话始终可正常交流
- 大工程详细结果都已落地文件
- scratch 路径明确、可回查
- 没有明显重叠失败的 watcher/timer
- 后台任务结束后完成收尾清理
- 系统摘要检查无新的 critical / failed

---

## 七、当前已确认的本机落地规则

- 数据盘临时区:`/mnt/data/openclaw/scratch/`
- Unity 重命名相关文档默认保留到 scratch,不轻易删除
- 对于大量文件任务,优先后台执行,主会话只看摘要
- watcher 频率需避免高峰期重叠;若出现"Failed to start"但服务实际成功退出,要优先检查 timer 过密与资源争抢

---

## 八、已落地的自动化入口

以下 5 支脚本已全部实现并通过烟测：

1. `scripts/openclaw-heavy-task-start.py` — 大工程统一开工入口
   - 自动建 scratch 子目录、默认写 `.keep`、运行 preflight
   - 若 preflight 失败则中止，防止基于错误状态继续操作
   - 用法：`python3 scripts/openclaw-heavy-task-start.py <目录> --task-name <任务名>`

2. `scripts/openclaw-heavy-task-preflight.py` — 开工前预检
   - 检查总量（文件数）、内存可用、根盘/数据盘余量、user failed units
   - 根据检查结果输出后台化/scratch 路径/冲突预扫建议

3. `scripts/openclaw-heavy-task-finish.py` — 统一收尾（7 步）
   - 系统状态快照 → 临时文件清理 → 会话大小清理 → 子代理检测 → scratch 过期预览 → journald + kernel cache → 健康检查
   - 输出手工收尾提醒（结果文档、daily/MEMORY 更新）

4. `scripts/openclaw-rename-conflict-check.py` — 批量改名前冲突预扫
   - 生成"原名 → 目标名"潜在覆盖报告，不直接执行改名
   - `--strip-hash-suffix`：对删 `#` 后缀的任务提前抓覆盖风险
   - `--prefix RoadSide_`：指定资源前缀（默认 `Props_`，传空字符串 = 无前缀）
   - ⚠️ 预扫仅看主文件（跳过 `.meta`）——执行改名时**必须同步处理 `.meta` 文件**

5. `scripts/openclaw-scratch-cleanup.py` — scratch 过期清理
   - 支持 `.keep` 保留标记、`--dry-run` 预览、按天数清理
   - `--print-kept` 分层显示：🛡️ keep 保护 / 📅 近 N 天内
