# 上下文溢出主动防御方案

> 版本: v1.1
> 生效日期: 2026-06-15
> 适用: Mark1 (公司 Linux) / Mark2
> 设计目标: 在工具循环中模型报 overflow 之前，四层主动防御

---

## 问题背景

OpenClaw 的 auto-compaction 默认仅在「接近上限」或「模型报 overflow」时才触发。工具循环中单轮可消耗 10-30K tokens，compaction 本身也需要 1-3 分钟才能完成——等它跑完时用户可能已经等了很久。

**目标**：从被动反应变为主动防御，在到达上限前 40-60K tokens 就提前干预。

---

## 四层架构

```
┌──────────────────────────────────────────────┐
│  Layer 1: 配置调优 (防线前移)                    │
│  提前在 60-80K 就触发压缩，留 40K+ 缓冲区        │
│  位置: ~/.openclaw/openclaw.json               │
├──────────────────────────────────────────────┤
│  Layer 2: 实时监控 (态势感知)                    │
│  每 5 分钟检查，70%/85%/95% 三级告警             │
│  位置: scripts/openclaw-context-monitor.py     │
├──────────────────────────────────────────────┤
│  Layer 3: Agent 自律 (源头减量)                  │
│  工具循环中主动控制 token 消耗                   │
│  位置: rules/work.md §7                         │
├──────────────────────────────────────────────┤
│  Layer 4: 恢复兜底 (已有)                        │
│  auto-compaction + 3 次重试 + post-guard        │
│  由 Gateway 内置                                │
└──────────────────────────────────────────────┘
```

## Layer 1：配置调优

### 文件

```
~/.openclaw/openclaw.json
workspace/openclaw.json  （同步副本）
```

### 配置项

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "mode": "safeguard",
        "truncateAfterCompaction": true,
        "keepRecentTokens": 25000,
        "notifyUser": true,
        "maxActiveTranscriptBytes": 600000
      }
    }
  }
}
```

| 参数 | 值 | 作用 |
|---|---|---|
| `mode` | `safeguard` | 近上限时触发压缩 |
| `truncateAfterCompaction` | `true` | 压缩后创建新 transcript（不重写旧文件） |
| `keepRecentTokens` | `25000` | 压缩后保留最近 ~25K tokens，主线程不丢上下文 |
| `notifyUser` | `true` | 压缩时通知用户 |
| `maxActiveTranscriptBytes` | `600000` | JSONL 达 600KB 主动压缩（距 131K 上限约 50K+ 缓冲） |

### Mark2 调参指南

| 模型窗口 | `maxActiveTranscriptBytes` 建议 | `keepRecentTokens` 建议 |
|---|---|---|
| 64K | 300000 | 12000 |
| 131K (deepseek-v4-pro) | 600000 | 25000 |
| 200K | 900000 | 40000 |
| 1M+ | 1500000 | 60000 |

## Layer 2：监控脚本

### 文件清单

```
scripts/openclaw-context-monitor.py          # 监控脚本
tools/openclaw-context-monitor/
  openclaw-context-monitor.service            # systemd service
  openclaw-context-monitor.timer              # systemd timer (每 5 分钟)
~/.config/systemd/user/
  openclaw-context-monitor.service            # 安装副本
  openclaw-context-monitor.timer              # 安装副本
~/.local/state/openclaw/context-monitor/
  status.json                                 # 运行状态
```

### 监控逻辑

```
每 5 分钟一次:
  1. 找当前活跃 session JSONL（.lock 文件）
  2. 文件大小 → 校准后 token 估算（14 KB ≈ 1K tokens）
  3. 对比模型上下文窗口
  4. 三级判定:
     - <70%: 静默，仅更新状态文件
     - ≥70% (WARN): 写入 broker 事件
     - ≥85% (ALERT): broker 事件 + frontstage 推送
     - ≥95% (CRIT): 紧急告警
```

### 三级阈值

| 级别 | 阈值 | 环境变量（可覆盖） | 动作 |
|---|---|---|---|
| WARN | 70% | `CONTEXT_MONITOR_WARN_PCT` | 写入 broker 事件 |
| ALERT | 85% | `CONTEXT_MONITOR_ALERT_PCT` | broker 事件 + frontstage 推送 |
| CRIT | 95% | `CONTEXT_MONITOR_CRIT_PCT` | 紧急告警 |

### 去重

同级别事件 10 分钟内不重复发送。

### 安装命令（Mark2 新机）

```bash
# 1. 复制脚本（workspace 已同步则跳过）
cp scripts/openclaw-context-monitor.py /目标/workspace/scripts/

# 2. 安装 systemd 单元
cp tools/openclaw-context-monitor/openclaw-context-monitor.service \
   tools/openclaw-context-monitor/openclaw-context-monitor.timer \
   ~/.config/systemd/user/

# 3. 启用
systemctl --user daemon-reload
systemctl --user enable --now openclaw-context-monitor.timer

# 4. 手动验证
systemctl --user start openclaw-context-monitor.service
python3 scripts/openclaw-context-monitor.py --print-human
cat ~/.local/state/openclaw/context-monitor/status.json | python3 -m json.tool
```

### 手动检查

```bash
# 人类可读
python3 scripts/openclaw-context-monitor.py --print-human

# 静默 check（写入状态文件 + broker）
python3 scripts/openclaw-context-monitor.py --check

# 强制推送到前台（忽略去重）
python3 scripts/openclaw-context-monitor.py --check --force-notify
```

### 校准因子

当前校准因子：**14 KB JSONL ≈ 1K tokens**（来自 2026-06-15 实测 `deepseek-v4-pro`）。

若 Mark2 使用不同模型（不同 tokenizer），可调整：

```bash
# 环境变量方式（写入 service 的 Environment 字段）
CONTEXT_MONITOR_BYTES_PER_KTOKEN=12288  # 12 KB per 1K tokens
```

## Layer 3：Agent 自律规则

### 触发条件（满足任一条即收紧）

1. 单轮工具循环已消耗超过 20K tokens
2. 会话上下文使用率超过 80%
3. 收到 context-monitor 的 ALERT 级别事件

### 收紧措施

1. **分段读取**：停止读大文件全文，改用 `offset/limit`
2. **结构化摘要**：长工具输出做摘要后再传递
3. **落盘接力**：中间结果写 `tmp/`，后续步骤读文件
4. **主动建议**：必要时建议用户 `/compact` 或 `/reset`

### Rule 文件位置

```
rules/work.md §7 「上下文溢出自律规则」
```

## Layer 4：Gateway 内置恢复兜底

此层由 OpenClaw Gateway 内置，无需额外配置：

1. 检测到 overflow → 触发 auto-compaction
2. 重试 3 次
3. 每次重试前自动提醒 agent 写 memory
4. 压缩成功 → 轮转 transcript → 重发 prompt
5. 失败 → post-compaction guard 接管

验证方式：`journalctl --user -u openclaw-gateway | grep -i "compaction\|overflow"`

## 已知限制

1. **Token 估算是近似值**：文件大小 / 14KB 的校准仅在 deepseek-v4-pro 实测，其他模型/tokenizer 会有偏差。可通过 `BYTES_PER_KTOKEN` 环境变量微调。
2. **去重冷却可能延迟**：同级别事件 10 分钟内只发一次，若使用率在冷却期内继续上升可能错过中级告警——但 CRIT 级别变化不受去重限制（severity 变了一定会发）。
3. **依赖 systemd user session**：若用户未登录（systemd user session 不存在），timer 不会运行。
