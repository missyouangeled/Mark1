# 贾维斯中枢安全体系设计

> 设计日期：2026-06-15
> 适用范围：中枢服务器（Ubuntu 24.04）
> 设计原则：纵深防御（Defense in Depth）——每一层单独被攻破都不致命

---

## 〇、威胁模型

在动手设计防御之前，先明确我们在防什么：

| 威胁 | 来源 | 危害 | 概率 |
|------|------|------|------|
| 暴力破解 Web 登录 | 公网扫描器 | 拿到 Control UI / code-server 控制权 | 高 |
| 漏洞利用（已知 CVE） | 自动化扫描 | 提权、数据泄露 | 中 |
| WebSocket 劫持 | 中间人 | 窃听贾维斯对话 | 低（有 TLS） |
| Docker 逃逸 | 恶意容器/配置不当 | 拿到宿主机 root | 低 |
| 凭据泄露 | 误操作/Git 提交 | API key 被盗刷 | 中 |
| DDoS | 恶意流量 | 服务不可用 | 低（CF 挡着） |
| 内部误操作 | 自己 | 删库、配错权限 | 中 |
| 供应链攻击 | npm/pip 恶意包 | 后门 | 低 |

## 一、七层防御体系

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L1 — Cloudflare 边缘层                             │
│                                                                             │
│  Cloudflare Tunnel: 隐藏真实 IP，所有流量经 CF 网络过滤                        │
│  Cloudflare Access: SSO 身份验证（邮箱验证码/GitHub 登录）                    │
│  Cloudflare WAF:    免费 5 条自定义规则                                       │
│  Cloudflare Rate Limiting: 免费 1 条规则（防暴力破解）                        │
│                                                                             │
│  费用: 全部免费 ✅                                                            │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L2 — 网络防火墙层                                  │
│                                                                             │
│  ufw 规则（Ubuntu 默认防火墙）:                                               │
│    ✅ 允许来自 Cloudflare IP 段的 443 端口（但走 Tunnel 其实不需要开端口）       │
│    ✅ 只允许来自 Tailscale 接口的 SSH                                          │
│    ✅ 所有入站流量默认 DROP                                                    │
│    ✅ 出站按需放行（API 调用、Git、apt 更新）                                  │
│                                                                             │
│  关键: 服务器没有任何端口对公网开放 — 全靠 Cloudflare Tunnel 入站               │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L3 — 传输加密层                                    │
│                                                                             │
│  公网段: Cloudflare → 用户 = CF Edge TLS 1.3                                 │
│  内网段: 全部走 Tailscale WireGuard 加密隧道                                  │
│  服务间: localhost 通信不加密（无外部暴露面）                                  │
│  SSH:    ed25519 密钥，禁用密码登录，禁用 root                                │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L4 — 身份认证层                                    │
│                                                                             │
│  服务                      认证方式                      保护级别              │
│  ────────────────────────  ──────────────────────────  ──────────────────    │
│  Control UI 公网入口       CF Access (邮箱验证码)        🔒 双重              │
│  code-server               CF Access + 本地密码         🔒 双重              │
│  Portainer                 CF Access + 本地密码         🔒 双重              │
│  Nextcloud                 应用自身密码 + 2FA app        🔒 双重              │
│  infos-handle API          Bearer token (Gateway key)   🔒 单重              │
│  SSH (Tailscale)           ed25519 密钥                 🔒 单重 + 网络隔离   │
│  Docker socket             仅 localhost + docker group   🔒 文件权限          │
│                                                                             │
│  所有面向公网的服务都通过 CF Access 挡一层                                     │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L5 — 应用加固层                                    │
│                                                                             │
│  Gateway 配置:                                                               │
│    • bind: 127.0.0.1（不监听外网）                                           │
│    • 所有模型 provider API key 放在单独环境变量/文件，600 权限                  │
│                                                                             │
│  SSH 加固:                                                                   │
│    • PermitRootLogin no                                                      │
│    • PasswordAuthentication no                                               │
│    • PubkeyAuthentication yes                                                │
│    • 只允许 jarvis 用户                                                       │
│                                                                             │
│  Docker 加固:                                                                │
│    • 非 root 运行（jarvis ∈ docker group）                                    │
│    • 所有容器: --restart unless-stopped（防意外停止）                          │
│    • 不挂载 docker.sock 到容器（Portainer 例外，仅 localhost 暴露）            │
│    • 容器以非 root 用户运行（user: "1000:1000"）                              │
│    • 敏感目录只读挂载                                                         │
│                                                                             │
│  自动更新:                                                                   │
│    • unattended-upgrades（安全补丁自动安装）                                  │
│    • Docker 镜像定期 docker compose pull（通过 cron）                         │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L6 — 数据保护层                                    │
│                                                                             │
│  凭据文件:                                                                   │
│    credentials/ 目录: chmod 700, gitignored                                  │
│    API key: 环境变量注入，不写死在配置文件                                    │
│    SSH key: 仅 jarvis 用户可读，设密码保护                                     │
│                                                                             │
│  敏感数据:                                                                   │
│    聊天记录 (sessions/): 仅 jarvis 用户可读                                   │
│    MEMORY.md: 不在非主会话中加载（已有规则）                                   │
│    session-backup: /mnt/data，独立数据盘                                      │
│                                                                             │
│  备份保护:                                                                   │
│    数据库备份不包含 API key                                                   │
│    Git 仓库不包含任何凭据                                                     │
│                                                                             │
│  磁盘加密:                                                                   │
│    LUKS 全盘加密（安装系统时勾选）——防物理盗窃                                  │
│    数据盘 /mnt/data 同样加密                                                  │
│    注: 云服务器通常已有供应商层面的磁盘加密                                    │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L7 — 监控与响应层                                   │
│                                                                             │
│  入侵检测:                                                                   │
│    fail2ban: 监控 SSH 登录失败（即使 SSH 只走 Tailscale）                      │
│    auditd: 监控关键文件变更（/etc/caddy/Caddyfile, ~/.ssh/authorized_keys）   │
│                                                                             │
│  容器安全:                                                                   │
│    docker scout / trivy: 定时扫描镜像漏洞                                    │
│    通过 lifecycle-maintainer 每周跑一次                                       │
│                                                                             │
│  安全事件接入 Mark1:                                                         │
│    安全事件 → broker 写入 → infos-handle → Control UI 仪表板「安全」卡片      │
│    fail2ban ban 事件、异常登录、容器漏洞 → 实时告警                            │
│                                                                             │
│  响应预案:                                                                   │
│    🔴 发现未知登录 → CF Access 关闭所有人 + 改密码 + 审计日志                  │
│    🟡 API key 疑似泄露 → 立即轮换 key + 检查账单                               │
│    🟢 容器漏洞 → 更新镜像                                                     │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 二、对照 Mark1 现有安全机制

| Mark1 现有 | 服务器需新增 | 理由 |
|-----------|-------------|------|
| infos-handle localhost 免鉴权 | 不变 | 只监听 127.0.0.1，无外部暴露面 |
| infos-handle 远程需 Bearer token | 保留 + 增加 CF Access | 双重保险 |
| Gateway 绑 0.0.0.0:18789 | **改为** 127.0.0.1:18789 | Caddy 反代即可，Gateway 不应直接暴露 |
| credentials/ gitignored | 不变 + 增加 auditd 监控 | 防止误提交 |
| 无防火墙 | **新增 ufw** | 服务器有公网流量 |
| 无 fail2ban | **新增 fail2ban** | 防暴力破解 |
| SSH 密码登录 | **禁用密码，仅密钥** | 服务器安全基本要求 |
| 无容器扫描 | **新增 trivy 周检** | Docker 服务多了就有风险 |
| session-backup 30min | 不变 + 增加备份加密（可选） | 数据盘已独立 |
| 补丁 systemd 自检 | **扩展** 新增安全项检查 | 不要等出事了才发现 |

## 三、关键配置清单

### 3.1 ufw 防火墙

```bash
# 默认策略
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Cloudflare Tunnel 不需要开放任何入站端口（Tunnel 是出站连接）
# 但如果未来切换到公网 IP 方案：
# sudo ufw allow 443/tcp

# Tailscale 内网（SSH + 服务互通）
sudo ufw allow in on tailscale0

# 启用
sudo ufw enable
sudo ufw status verbose
```

### 3.2 SSH 加固

```bash
# /etc/ssh/sshd_config.d/99-hardening.conf
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
AllowUsers jarvis
MaxAuthTries 3
ClientAliveInterval 300
ClientAliveCountMax 2

sudo systemctl restart sshd
```

### 3.3 fail2ban

```bash
sudo apt install -y fail2ban

# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
bantime = 3600
findtime = 600

# 自定义 action: ban 事件写入 broker 文件 → Control UI 告警
# action = %(action_mwl)s

sudo systemctl enable --now fail2ban
```

### 3.4 auditd 关键文件监控

```bash
sudo apt install -y auditd

# 监控关键文件变更
sudo auditctl -w /etc/caddy/Caddyfile -p wa -k caddy_config
sudo auditctl -w ~/.ssh/authorized_keys -p wa -k ssh_keys
sudo auditctl -w /home/jarvis/.openclaw/workspace/credentials -p wa -k credentials
```

### 3.5 Docker 容器安全基线

```bash
# docker-compose.yml 示例（安全基线）
services:
  nextcloud:
    image: nextcloud:stable
    user: "1000:1000"              # 非 root 运行
    read_only: true                # 只读根文件系统（除了挂载卷）
    restart: unless-stopped
    volumes:
      - nextcloud_data:/var/www/html:rw  # 仅数据目录可写
    security_opt:
      - no-new-privileges:true     # 禁止提权
    cap_drop:
      - ALL                         # 丢弃所有 capabilities
    cap_add:
      - CHOWN                       # 仅加必要的
      - SETGID
      - SETUID

  # 不安全的容器将被 Portainer 标记
```

### 3.6 容器漏洞扫描（cron 周检）

```bash
# 在 lifecycle-maintainer 中新增：
# 每周日凌晨 4 点跑一次 trivy 扫描
docker run --rm \
  -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image --severity HIGH,CRITICAL \
  $(docker images --format '{{.Repository}}:{{.Tag}}' | grep -v '<none>') \
  > /tmp/trivy-report.txt

# 有 HIGH/CRITICAL 漏洞 → 写入 broker 事件 → Control UI 告警
```

### 3.7 安全事件接入 Control UI

```
安全事件源                →  事件类型          →  仪表板卡片
─────────────────────────────────────────────────────────
fail2ban ban             →  ssh_ban            →  🔴 安全告警
auditd 配置变更          →  config_change      →  🟡 配置变更
trivy HIGH/CRITICAL      →  container_vuln     →  🟡 容器漏洞
ufw 拦截                 →  firewall_block     →  ℹ️ 防火墙日志
CF Access 异常登录        →  auth_failure       →  🔴 安全告警
```

## 四、安全审查——发现的问题

对照 Mark1 现有链路和 v3 方案，逐项审查安全面：

### ✅ 已有/已修正

| 项 | 状态 |
|----|------|
| Gateway 不直接暴露公网 | ✅ v3 已改为 Caddy 反代 |
| code-server 需要密码 | ✅ v3 已加 |
| CF Access 挡在 Web 服务前 | ✅ 本设计新增 |
| SSH 禁密码 | ✅ 本设计新增 |
| ufw 防火墙 | ✅ 本设计新增 |

### ⚠️ 需要额外注意

| 项 | 风险 | 修正 |
|----|------|------|
| Portainer 暴露 docker.sock | 如果 Portainer Web UI 被攻破 = 宿主机 root | Portainer 加 CF Access + 强密码 + 可考虑不用时关掉容器 |
| 微信桥接（Windows 侧） | Windows 被攻破 → 微信数据泄露 | 掌机 Windows 也装上 Tailscale + Windows Defender |
| 备份未加密 | 备份文件含聊天记录 | 可选：tar 前 gpg 加密 |

### 🔴 一个底线建议

> **任何面向公网的服务，都加一层 Cloudflare Access。**
> 这意味着：访问者必须先通过 CF 的身份验证（邮箱验证码 / GitHub 登录），才能到达你的 Caddy/服务。
> 这是免费且最有效的第一道防线——在流量到达你的服务器之前，CF 已经挡掉了 99% 的恶意扫描。

## 五、安全检查清单（日常使用）

迁移完成后，每月跑一次：

```bash
# 一键安全体检脚本
sudo ufw status verbose                    # 防火墙规则
sudo fail2ban-client status sshd           # SSH 封禁记录
sudo docker scout quickview 2>/dev/null || \
  docker run --rm aquasec/trivy image --severity HIGH,CRITICAL --no-progress $(docker images -q | head -5)  # 镜像漏洞
sudo ausearch -k caddy_config -k ssh_keys --start today  # 配置变更审计
systemctl --user status openclaw-gateway   # Gateway 绑 127.0.0.1 确认
grep "PasswordAuthentication" /etc/ssh/sshd_config.d/*.conf  # SSH 密码禁用确认
```
