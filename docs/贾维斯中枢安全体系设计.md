# 贾维斯中枢安全体系设计

> 设计日期：2026-06-15 | 审查日期：2026-06-15（联网验证 + Mark1 对照）
> 适用范围：中枢服务器（Ubuntu 24.04）
> 设计原则：纵深防御（Defense in Depth）——每一层单独被攻破都不致命
> 🔍 审查来源：Cloudflare 官方文档 / NVD CVE 数据库 / Docker 安全最佳实践 2026 / Reddit r/selfhosted / Ubuntu 24.04 加固指南

---

## 〇、威胁模型

在动手设计防御之前，先明确我们在防什么：

| 威胁 | 来源 | 危害 | 概率 |
|------|------|------|------|
| 暴力破解 Web 登录 | 公网扫描器 | 拿到 Control UI / code-server 控制权 | 高 |
| 漏洞利用（已知 CVE） | 自动化扫描 | 提权、数据泄露 | 中 |
| CF TLS 终止处的流量可见 | Cloudflare 自身 | CF 可解密所有 HTTPS 流量 | 中（信任风险） |
| WebSocket 劫持 | 中间人 | 窃听贾维斯对话 | 低（有 TLS） |
| Docker 逃逸 | 恶意容器/配置不当 | 拿到宿主机 root | 低 |
| docker.sock 滥用 | 被攻破的容器 | 控制所有容器 / 宿主机 | 中（Portainer 面） |
| 凭据泄露 | 误操作/Git 提交 | API key 被盗刷 | 中 |
| DDoS | 恶意流量 | 服务不可用 | 低（CF 挡着） |
| 内部误操作 | 自己 | 删库、配错权限 | 中 |
| 供应链攻击 | npm/pip 恶意包 | 后门 | 低 |
| Docker Engine 漏洞 | 未及时更新 | 容器逃逸/提权 | 中（实测 CVE-2025-68121） |

## 一、七层防御体系

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L1 — Cloudflare 边缘层                             │
│                                                                             │
│  Cloudflare Tunnel: 隐藏真实 IP，所有流量经 CF 网络过滤                        │
│  Cloudflare Access: SSO 身份验证（邮箱验证码/GitHub 登录）                    │
│  Cloudflare WAF:    免费 5 条自定义规则                                       │
│  Cloudflare Rate Limiting: 免费 1 条规则（防暴力破解）                        │
│                                                                             │
│  费用: 全部免费 ✅（Access 免费 50 用户，个人使用足够）                         │
│  ⚠️ 已知取舍: CF 在边缘终止 TLS，技术上可解密流量。信任边界 = 相信 CF 不会滥用   │
│     缓解: 极度敏感数据不应通过 CF Tunnel 传输；日常聊天/代码无此顾虑           │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L2 — 网络防火墙层                                  │
│                                                                             │
│  ufw 规则（Ubuntu 默认防火墙）:                                               │
│    ✅ 允许来自 Cloudflare IP 段的 443 端口（但走 Tunnel 其实不需要开端口）       │
│    ✅ 只允许来自 Tailscale 接口的 SSH                                          │
│    ✅ 所有入站流量默认 DROP                                                    │
│    ✅ 出站按需放行（API 调用、Git、apt 更新）                                  │
│                                                                             │
│  关键: 服务器没有任何端口对公网开放 — 全靠 Cloudflare Tunnel 入站               │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L3 — 传输加密层                                    │
│                                                                             │
│  公网段: Cloudflare → 用户 = CF Edge TLS 1.3                                 │
│  内网段: 全部走 Tailscale WireGuard 加密隧道                                  │
│  服务间: localhost 通信不加密（无外部暴露面）                                  │
│  SSH:    ed25519 密钥，禁用密码登录，禁用 root                                │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L4 — 身份认证层                                    │
│                                                                             │
│  服务                      认证方式                      保护级别              │
│  ────────────────────────  ──────────────────────────  ──────────────────    │
│  Control UI 公网入口       CF Access (邮箱验证码)        🔒 双重              │
│  code-server               CF Access + 本地密码         🔒 双重              │
│  Portainer                 CF Access + 本地密码         🔒 双重              │
│  Nextcloud                 应用自身密码 + 2FA app        🔒 双重              │
│  infos-handle API          Bearer token (Gateway key)   🔒 单重              │
│  SSH (Tailscale)           ed25519 密钥                 🔒 单重 + 网络隔离   │
│  Docker socket             仅 localhost + docker group   🔒 文件权限          │
│                                                                             │
│  所有面向公网的服务都通过 CF Access 挡一层                                     │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L5 — 应用加固层                                    │
│                                                                             │
│  Gateway 配置:                                                               │
│    • bind: 127.0.0.1（不监听外网）                                           │
│    • 所有模型 provider API key 放在单独环境变量/文件，600 权限                  │
│                                                                             │
│  SSH 加固:                                                                   │
│    • PermitRootLogin no                                                      │
│    • PasswordAuthentication no                                               │
│    • PubkeyAuthentication yes                                                │
│    • 只允许 jarvis 用户                                                       │
│                                                                             │
│  Docker 加固:                                                                │
│    • 非 root 运行（jarvis ∈ docker group）                                    │
│    • 所有容器: --restart unless-stopped（防意外停止）                          │
│    • 容器以非 root 用户运行（user: "1000:1000"）                              │
│    • 敏感目录只读挂载                                                         │
│                                                                             │
│  Docker Socket 安全（⚠️ 审查重点修正）:                                        │
│    • 不直接挂载 /var/run/docker.sock 到任何面向公网的容器                      │
│    • 推荐方案: docker-socket-proxy 代理（限制 API 调用权限）                    │
│      → 只暴露 Portainer 需要的 API（containers/list, images/list 等）         │
│      → 拒绝 exec/create/run 等高危操作                                       │
│    • 降级方案: Portainer 可直挂 docker.sock，但必须 CF Access + 强密码         │
│      + 不在公网开放 Portainer 端口 + 不用时 docker stop portainer             │
│                                                                             │
│  AppArmor:                                                                  │
│    • Ubuntu 24.04 自带 AppArmor 4.0                                          │
│    • Docker 容器默认启用 AppArmor 策略（docker-default profile）               │
│    • 检查: sudo aa-status | grep docker                                      │
│                                                                             │
│  自动更新:                                                                   │
│    • unattended-upgrades（安全补丁自动安装）                                  │
│    • ⚠️ Docker Engine 不在 unattended-upgrades 范围（来自 Docker 官方 repo）   │
│    • Docker Engine: 每月手动 sudo apt upgrade docker-ce                     │
│    • Docker 镜像更新通知：使用 Diun（crazymax/diun，替代已归档的 Watchtower）    │
│      → 只通知不自动更新，避免半夜自动升级把服务搞挂                            │
│      → 每周日检查一次，通知接入 broker → Control UI 显示「有 N 个容器可更新」      │
│      → 手动确认后再 docker compose pull && docker compose up -d             │
│    • code-server: 每月检查更新（已有 CVE-2025-47269 CVSS 8.3 先例）           │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L6 — 数据保护层                                    │
│                                                                             │
│  凭据文件:                                                                   │
│    credentials/ 目录: chmod 700, gitignored                                  │
│    API key: 环境变量注入，不写死在配置文件                                    │
│    SSH key: 仅 jarvis 用户可读，设密码保护                                     │
│                                                                             │
│  敏感数据:                                                                   │
│    聊天记录 (sessions/): 仅 jarvis 用户可读                                   │
│    MEMORY.md: 不在非主会话中加载（已有规则）                                   │
│    session-backup: /mnt/data，独立数据盘                                      │
│                                                                             │
│  备份保护:                                                                   │
│    数据库备份不包含 API key                                                   │
│    Git 仓库不包含任何凭据                                                     │
│                                                                             │
│  磁盘加密:                                                                   │
│    LUKS 全盘加密（安装系统时勾选）——防物理盗窃                                  │
│    数据盘 /mnt/data 同样加密                                                  │
│    注: 云服务器通常已有供应商层面的磁盘加密                                    │
│                                                                             │
│  镜像供应链安全:                                                               │
│    Docker 镜像固定 digest（不使用 :latest 标签）                                │
│    示例: image: portainer/portainer-ce:2.27.1@sha256:abc123...               │
│    所有 docker-compose.yml 中的镜像均需锁定版本 + digest                        │
│    获取 digest: docker inspect <image> --format '{{.RepoDigests}}'   │
│    定期 trivy 扫描锁定版本（L7），配合 Diun 通知新版本可用时手动升级             │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         🛡️ L7 — 监控与响应层                                   │
│                                                                             │
│  入侵检测:                                                                   │
│    fail2ban: 监控 SSH 登录失败（即使 SSH 只走 Tailscale）                      │
│    auditd: 监控关键文件变更（/etc/caddy/Caddyfile, ~/.ssh/authorized_keys）   │
│                                                                             │
│  容器安全:                                                                   │
│    docker scout / trivy: 定时扫描镜像漏洞                                    │
│    通过 lifecycle-maintainer 每周跑一次                                       │
│                                                                             │
│  安全事件接入 Mark1:                                                         │
│    安全事件 → broker 写入 → infos-handle → Control UI 仪表板「安全」卡片      │
│    fail2ban ban 事件、异常登录、容器漏洞 → 实时告警                            │
│                                                                             │
│  响应预案:                                                                   │
│    🔴 发现未知登录 → CF Access 关闭所有人 + 改密码 + 审计日志                  │
│    🟡 API key 疑似泄露 → 立即轮换 key + 检查账单                               │
│    🟢 容器漏洞 → 更新镜像                                                     │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 二、对照 Mark1 现有安全机制

| Mark1 现有 | 服务器需新增 | 理由 |
|-----------|-------------|------|
| infos-handle localhost 免鉴权 | 不变 | 只监听 127.0.0.1，无外部暴露面 |
| infos-handle 远程需 Bearer token | 保留 + 增加 CF Access | 双重保险 |
| Gateway 绑 0.0.0.0:18789 | **改为** 127.0.0.1:18789 | Caddy 反代即可，Gateway 不应直接暴露 |
| credentials/ gitignored | 不变 + 增加 auditd 监控 | 防止误提交 |
| 无防火墙 | **新增 ufw** | 服务器有公网流量 |
| 无 fail2ban | **新增 fail2ban** | 防暴力破解 |
| SSH 密码登录 | **禁用密码，仅密钥** | 服务器安全基本要求 |
| 无容器扫描 | **新增 trivy 周检** | Docker 服务多了就有风险 |
| session-backup 30min | 不变 + 增加备份加密（可选） | 数据盘已独立 |
| 补丁 systemd 自检 | **扩展** 新增安全项检查 | 不要等出事了才发现 |
| 无 AppArmor 检查 | **新增** aa-status 确认 | Ubuntu 24.04 自带，应确认启用 |
| 无 Docker Engine 更新策略 | **新增** 每月 apt upgrade docker-ce | unattended-upgrades 不覆盖 Docker repo |
| 无 docker.sock 隔离 | **新增** docker-socket-proxy | 限制容器对 Docker API 的访问权限 |
| 无 code-server 版本管理 | **新增** 安装 ≥v4.99.4 + 月检 | CVE-2025-47269 CVSS 8.3 |

## 三、关键配置清单

### 3.1 ufw 防火墙

```bash
# 默认策略
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Cloudflare Tunnel 不需要开放任何入站端口（Tunnel 是出站连接）
# 但如果未来切换到公网 IP 方案：
# sudo ufw allow 443/tcp

# Tailscale 内网（SSH + 服务互通）
sudo ufw allow in on tailscale0

# 启用
sudo ufw enable
sudo ufw status verbose
```

### 3.2 SSH 加固

```bash
# 首先生成 ed25519 密钥（带密码保护）
ssh-keygen -t ed25519 -C "jarvis@hub" -f ~/.ssh/id_ed25519
# 提示输入 passphrase 时务必设一个强密码

# /etc/ssh/sshd_config.d/99-hardening.conf
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
AllowUsers jarvis
MaxAuthTries 3
ClientAliveInterval 300
ClientAliveCountMax 2

sudo systemctl restart sshd
```

### 3.3 unattended-upgrades（安全补丁自动安装）

```bash
sudo apt install -y unattended-upgrades

# 启用安全更新自动安装
sudo tee /etc/apt/apt.conf.d/50unattended-upgrades << 'EOF'
Unattended-Upgrade::Allowed-Origins {
    "${distro_id}:${distro_codename}-security";
    "${distro_id}ESMApps:${distro_codename}-apps-security";
    "${distro_id}ESM:${distro_codename}-infra-security";
};
Unattended-Upgrade::Automatic-Reboot "false";
Unattended-Upgrade::Remove-Unused-Dependencies "true";
EOF

# 启用定时器
sudo systemctl enable --now unattended-upgrades

# ⚠️ 注意：Docker Engine 来自 Docker 官方 repo，不在此范围内
#     需要手动: sudo apt upgrade docker-ce（每月一次）
```

### 3.4 AppArmor 状态确认

```bash
# AppArmor 是 Ubuntu 24.04 默认启用的强制访问控制系统
# Docker 容器默认受 AppArmor docker-default 策略保护

sudo aa-status | grep docker
# 应显示: docker-default (enforce)

# 查看当前加载的策略
sudo aa-status | head -5
# 应显示: apparmor module is loaded.
```

### 3.5 fail2ban

```bash
sudo apt install -y fail2ban

# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
bantime = 3600
findtime = 600

sudo systemctl enable --now fail2ban
```

### 3.6 auditd 关键文件监控

```bash
sudo apt install -y auditd

# 监控关键文件变更
sudo auditctl -w /etc/caddy/Caddyfile -p wa -k caddy_config
sudo auditctl -w ~/.ssh/authorized_keys -p wa -k ssh_keys
sudo auditctl -w /home/jarvis/.openclaw/workspace/credentials -p wa -k credentials
```

### 3.7 Docker 容器安全基线 + Socket Proxy

#### 3.7.1 docker-socket-proxy（优先推荐）

```bash
# 替代直接挂载 docker.sock，用代理限制 Portainer 的 API 权限
# 创建 docker-compose.yml for socket-proxy
docker run -d \
  --name docker-socket-proxy \
  --restart unless-stopped \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  -p 127.0.0.1:2375:2375 \
  -e CONTAINERS=1 \
  -e IMAGES=1 \
  -e NETWORKS=1 \
  -e VOLUMES=1 \
  -e INFO=1 \
  -e EXEC=0 \
  -e POST=0 \
  -e AUTH=0 \
  -e SECRETS=0 \
  tecnativa/docker-socket-proxy

# Portainer 改为连接 tcp://docker-socket-proxy:2375
# 不再需要挂载 docker.sock
docker run -d \
  -p 127.0.0.1:9000:9000 \
  --name portainer \
  --restart unless-stopped \
  -v portainer_data:/data \
  portainer/portainer-ce --http-port 9000
# Portainer 初始化时，Docker endpoint 填 tcp://docker-socket-proxy:2375
```

#### 3.7.2 容器安全基线

```bash
# docker-compose.yml 示例（安全基线）
services:
  nextcloud:
    image: nextcloud:stable
    user: "1000:1000"              # 非 root 运行
    read_only: true                # 只读根文件系统（除了挂载卷）
    restart: unless-stopped
    volumes:
      - nextcloud_data:/var/www/html:rw  # 仅数据目录可写
    security_opt:
      - no-new-privileges:true     # 禁止提权
    cap_drop:
      - ALL                         # 丢弃所有 capabilities
    cap_add:
      - CHOWN                       # 仅加必要的
      - SETGID
      - SETUID

  # 不安全的容器将被 Portainer 标记
```

### 3.9 容器漏洞扫描（cron 周检）

```bash
# 在 lifecycle-maintainer 中新增：
# 每周日凌晨 4 点跑一次 trivy 扫描
docker run --rm \
  -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image --severity HIGH,CRITICAL \
  $(docker images --format '{{.Repository}}:{{.Tag}}' | grep -v '<none>') \
  > /tmp/trivy-report.txt

# 有 HIGH/CRITICAL 漏洞 → 写入 broker 事件 → Control UI 告警
```

### 3.10 安全事件接入 Control UI

```
安全事件源                →  事件类型          →  仪表板卡片
─────────────────────────────────────────────────────────
fail2ban ban             →  ssh_ban            →  🔴 安全告警
auditd 配置变更          →  config_change      →  🟡 配置变更
trivy HIGH/CRITICAL      →  container_vuln     →  🟡 容器漏洞
ufw 拦截                 →  firewall_block     →  ℹ️ 防火墙日志
CF Access 异常登录        →  auth_failure       →  🔴 安全告警
```

## 联网审查日志（2026-06-15）

对照 Mark1 实际链路 + 搜索最新安全实践，发现以下问题并已修正：

| # | 发现 | 来源 | 严重度 | 修正 |
|---|------|------|--------|------|
| 1 | CF Tunnel 在边缘终止 TLS，CF 技术上可解密流量 | Hacker News / Reddit 讨论 | ⚠️ 信任边界 | 新增威胁模型条目 + L1 已知取舍说明 |
| 2 | docker.sock 直接挂载是社区公认的高风险模式 | Reddit r/docker / Docker 论坛 2025 | 🔴 高危 | 新增 docker-socket-proxy 方案，限制 API 调用权限 |
| 3 | code-server CVE-2025-47269（CVSS 8.3） | NVD 官方数据库 | 🔴 高危 | 要求安装版本 ≥4.99.4 + 月检更新 |
| 4 | Docker Engine CVE-2025-68121（Go TLS bug CVSS 9+） | Portainer 官方博客 / NVD | 🔴 严重 | 新增手动升级策略（unattended-upgrades 不覆盖 Docker repo） |
| 5 | AppArmor 未明确提及确认步骤 | Ubuntu 24.04 官方安全文档 | 🟡 中等 | 新增 AppArmor 状态确认命令 |
| 6 | unattended-upgrades 未作为独立步骤 | Ubuntu 加固指南 2025/2026 | 🟡 中等 | 新增独立步骤 + 明确 Docker Engine 不在此范围的说明 |
| 7 | SSH 私钥未要求密码保护（passphrase） | SSH 加固最佳实践 | 🟡 中等 | 新增 ssh-keygen + passphrase 提示 |
| 8 | Portainer CVE-2025-49593 存在 | NVD 数据库 | 🟡 中等 | L4 身份认证层已双重防护，月检更新即可覆盖 |

**审查结论**：7 层结构本身设计合理，修正的都是执行层面的细节和风险认知空白。最关键的修正：#2（docker-socket-proxy）和 #4（Docker Engine 手动升级）是这次审查中发现的设计盲区。

| AppArmor 确认启用 | ✅ 本次审查新增 |

### ⚠️ 需要额外注意

| 项 | 风险 | 修正 |
|----|------|------|
| Portainer + docker.sock | 即使走 proxy，Portainer 仍有管理权限 | 加 CF Access + 强密码 + 不用时关掉容器 |
| 微信桥接（Windows 侧） | Windows 被攻破 → 微信数据泄露 | 掌机 Windows 也装上 Tailscale + Windows Defender |
| 备份未加密 | 备份文件含聊天记录 | 可选：tar 前 gpg 加密 |
| CF TLS 终止 | CF 技术上可看到明文流量 | 信任边界清晰即可，日常聊天无顾虑 |
| Docker Engine 非自动更新 | CVE-2025-68121（Go TLS bug CVSS 9+）影响所有旧版 | 每月 `sudo apt upgrade docker-ce` |
| code-server CVE | CVE-2025-47269 CVSS 8.3 已被修复（≥v4.99.4） | 每月检查更新 + 安装时确认版本 ≥4.99.4 |

### 🔴 一个底线建议

> **任何面向公网的服务，都加一层 Cloudflare Access。**
> 这意味着：访问者必须先通过 CF 的身份验证（邮箱验证码 / GitHub 登录），才能到达你的 Caddy/服务。
> 这是免费且最有效的第一道防线——在流量到达你的服务器之前，CF 已经挡掉了 99% 的恶意扫描。

## 五、安全检查清单（日常使用）

迁移完成后，每月跑一次：

```bash
# 一键安全体检脚本
sudo ufw status verbose                    # 防火墙规则
sudo fail2ban-client status sshd           # SSH 封禁记录
sudo aa-status | grep docker               # AppArmor Docker 策略确认

# Docker Engine 版本检查（CVE-2025-68121 修复需 ≥29.3.0）
docker version --format '{{.Server.Version}}'

# 容器镜像漏洞扫描
docker run --rm aquasec/trivy image --severity HIGH,CRITICAL --no-progress $(docker images -q | head -5)

# 配置变更审计
sudo ausearch -k caddy_config -k ssh_keys --start today

# 关键服务绑定检查
ss -tlnp | grep -E "18789|18790|18792"     # 是否全绑 127.0.0.1
systemctl --user status openclaw-gateway

# SSH 密码禁用确认
grep "PasswordAuthentication" /etc/ssh/sshd_config.d/*.conf

# 检查是否有 Docker Engine 安全更新可用
apt list --upgradable 2>/dev/null | grep docker-ce
```
