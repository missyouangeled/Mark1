# 通用-视频平台下载工作流

- 适用机器：通用
- 系统 / OS：通用（其中浏览器启动参数需按当前机器环境调整）
- 文档类型：跨机器共享工作流 / 工具运行笔记

## 用途

当用户提出“去抖音/其他视频平台搜索并下载某条视频”这类需求时，优先按本流程执行，目标是：

1. **先快速拿到可验证的目标页面**
2. **尽量绕开站内搜索、验证码、登录墙造成的第一层阻塞**
3. **尽快落到真实 mp4 / m3u8 / 可下载媒体地址**
4. **完成下载、校验、交付**
5. **如果链路不完整，明确记录卡点与缺口，便于后续补齐**

---

## 总原则

### 1. 先完成用户目标，不被非关键缺口拖死

如果某个平台的“理想路线”缺少依赖（例如未装浏览器库、验证码方案未打通、官方 API 无法用），不要先卡死在“补完所有基础设施”上。

优先顺序：

- **能直接达成用户目标的可运行链路** > 完美但未打通的理想链路
- **公开页面 / 公开视频 / 页面内真实媒体地址** > 死磕站内搜索入口
- **最小成功路径** > 大重构 / 大安装

### 2. 先验证页面真实状态，再决定路线

不要凭印象说“这个页面应该能看到列表”。

每次都先实际检查：

- 页面标题
- 当前 URL
- 可交互快照
- DOM 中是否真有作品列表
- 网络请求里是否真返回作品数据
- 是否出现验证码 / 登录弹层 / 服务异常

### 3. 对下载类任务，交付必须可验证

交付前至少做一项校验：

- 文件存在
- 文件大小非 0
- `ffprobe` / 媒体信息可读
- 时长正常

---

## 通用执行流程

## 第 0 步：容量预检

下载前先预估体积并做空间判断。

- 小文件（几十 MB 以内）可直接落在工作区 `tmp/`
- 大文件按 `scripts/storage-preflight.sh` 先做预检
- 预计新增占用 >= 1G，或可能压到根盘安全线以下时，优先放 `/mnt/data`

推荐：

```bash
bash scripts/storage-preflight.sh 200M video-download
```

---

## 第 1 步：优先走平台内直接路径

默认顺序：

1. 打开平台搜索页
2. 搜索目标关键词
3. 进入目标作者主页
4. 打开视频列表第一个视频
5. 提取真实媒体地址并下载

如果这一条线畅通，直接完成，不必绕路。

---

## 第 2 步：一旦平台内入口被拦，立刻切到“外搜定位”

常见拦截：

- 验证码中间页
- 登录墙
- 搜索结果空白 / 前端懒加载失败
- 主页作品流返回“服务异常”

此时不要只停在“被拦住了”。

应立刻改走：

1. 用外部搜索定位公开页面
2. 先拿到以下任一可验证入口：
   - 作者主页 URL
   - 公开视频页 URL
   - 作者抖音号 / sec_user_id / 视频 ID
3. 再从公开视频页反查作者主页，或从作者主页反查公开视频页

适用检索关键词模板：

- `site:平台域名 关键词`
- `"作者名" site:平台域名/video`
- `"作者名" 平台名 账号`
- `"抖音号/用户号" 平台名`

---

## 第 3 步：到达页面后，优先找“作者链接”和“视频真实地址”

### A. 如果已经在公开视频页

优先拿：

- 作者主页链接
- 视频标题
- 视频 ID
- 视频标签内 `currentSrc`

### B. 如果已经在作者主页

优先检查：

- 作品区是否正常加载
- 页面内是否存在 `/video/` 链接
- 网络里是否确实发出了作品列表 API
- API 返回体是否为空 / 0 长度 / 错误

如果作者主页能进但作品流异常，不要原地等：

- 回退到公开视频页链路
- 用公开索引到的公开视频作为“当前可验证入口”继续下载

---

## 第 4 步：提取真实媒体地址

优先顺序：

1. 页面内 `<video>` 的 `currentSrc`
2. 页面内脚本状态（JSON / 内嵌数据）
3. 浏览器网络请求中的 mp4 / m3u8 / play API
4. 作者页 / 作品页 API 返回的播放地址字段

原则：

- **先抓页面已经解出来的真实媒体地址**
- 能拿到 mp4 就先拿 mp4
- 如果只有 m3u8，再考虑转存 / ffmpeg

---

## 第 5 步：下载并校验

推荐最小校验：

```bash
ls -lh <file>
ffprobe -v error -show_entries format=duration,size -of json <file>
```

至少确认：

- 文件存在
- 体积正常
- 时长正常
- 不是 0 字节 / HTML 错页

### 当前最小脚本入口（已落地）

当你已经拿到**抖音公开视频页 URL** 时，可直接用：

```bash
python3 scripts/download-platform-video.py 'https://www.douyin.com/video/<视频ID>?source=Googlebot' --browser-args=--no-sandbox
```

当你手里已经有多个候选视频页 URL 时，也可以把“目标位”作为变量传进去：

```bash
python3 scripts/download-platform-video.py \
  --pick=index:2 \
  'https://www.douyin.com/video/第1条?source=Googlebot' \
  'https://www.douyin.com/video/第2条?source=Googlebot' \
  --browser-args=--no-sandbox
```

它现在也支持直接吃**混合文本 / 搜索片段 / 分享文案**，自动从里面提取受支持的 URL：

```bash
python3 scripts/download-platform-video.py \
  --pick=last \
  --browser-args=--no-sandbox \
  '这里有两个候选： https://www.douyin.com/video/第1条?source=Googlebot 和 https://www.douyin.com/video/第2条?source=Googlebot'
```

如果候选内容已经被整理进一个文本文件，也可以：

```bash
python3 scripts/download-platform-video.py \
  dummy \
  --input-file ./candidates.txt \
  --pick=index:1 \
  --browser-args=--no-sandbox
```

说明：`dummy` 只是占位输入；真正会用的是 `--input-file` 里自动提取到的受支持 URL。

如果你此时还不想直接下载，只想先把候选整理成统一格式，也可以：

```bash
python3 scripts/download-platform-video.py \
  --list-only \
  --pick=last \
  '这里有两个候选 https://www.douyin.com/video/第1条?source=Googlebot 和 https://www.douyin.com/video/第2条?source=Googlebot'
```

如果你还想顺手把候选 URL 落成文本文件，便于下一步直接复用：

```bash
python3 scripts/download-platform-video.py \
  --list-only \
  --pick=last \
  --candidates-out tmp/video-downloads/candidates.txt \
  '这里有两个候选 https://www.douyin.com/video/第1条?source=Googlebot 和 https://www.douyin.com/video/第2条?source=Googlebot'
```

如果你还想把这次候选整理结果固定成结构化 JSON 文件，便于 agent 或后续脚本接力：

```bash
python3 scripts/download-platform-video.py \
  --list-only \
  --pick=last \
  --candidates-out tmp/video-downloads/candidates.txt \
  --report-out tmp/video-downloads/candidates.json \
  '这里有两个候选 https://www.douyin.com/video/第1条?source=Googlebot 和 https://www.douyin.com/video/第2条?source=Googlebot'
```

它会输出：

- `generated_at`：生成时间
- `selected`：按当前 `--pick` 选中的目标
- `selected_from`：去重后的原始候选 URL 列表
- `candidates[]`：标准化候选（含 `index / platform / id / url`）
- `candidates_out`：候选导出文件路径（如果传了 `--candidates-out`）
- `next_download_command`：基于这份候选结果直接继续下载的推荐命令

当前支持：

- `first`
- `last`
- `random`
- `index:N`（1-based）
- `video:<id>`

当前脚本已完成：

- 空间预检
- 打开页面
- 对抖音公开视频页提取 `<video>.currentSrc`
- 下载 mp4
- `ffprobe` 校验
- 输出 JSON 摘要
- 下载结果里附带 `replay_download_command`，便于复跑
- 当输入是抖音作者主页 URL 时，尝试解析候选视频 URL；若检测到主页作品流返回“服务异常”，则**停止自动选取**，避免误把热点/推荐视频当成作者作品

注意：这不是完整的“关键词搜索器”；它目前**还不负责**从“关键词搜索”里自动生成候选列表，但已经支持：

1. 上游拿到多个候选视频页 URL 时，用 `--pick` 选择目标位
2. 上游给的是混合文本 / 搜索片段 / 分享文案时，脚本可先自动抽取受支持 URL 再进入选择
3. 上游若只想先做“候选整理”，可用 `--list-only` 输出标准化候选 JSON
4. 输入作者主页 URL 时，在主页正常的情况下提取候选；若主页异常则明确拒绝误选

---

## 第 6 步：交付给用户

默认优先级：

1. 聊天内直接附件 / `MEDIA:`
2. 如果用户明确说“给我浏览器地址我自己下”，则起临时下载服务
3. 对 `mp4/mp3/pdf` 等可能被浏览器直接预览的文件，优先带 `Content-Disposition: attachment`

## 当前缺口 / 尚未补齐

### A. 缺少稳定的“账号态 / 登录态”浏览器路径

当前没有一个已验证的、可长期复用的抖音登录态浏览器配置。

影响：

- 搜索入口容易触发验证码
- 作者作品流可能异常
- 一些仅登录后可见的数据读不到

### B. 公开搜索引擎也不稳定，不能直接当脚本默认上游

这次继续尝试把“关键词 → 候选列表”也脚本化时，浏览器自动化实际碰到：

- 百度：安全验证
- 搜狗：反爬页
- DuckDuckGo：当前环境不稳定 / 无有效结果
- Bing：中文相关结果质量低，容易偏题

结论：

- 当前不能把“公开搜索引擎抓候选”直接写成默认脚本主路径
- 这一步暂时仍以 **agent 的外部搜索工具 + 人工规则** 为主，而不是在脚本里盲做

### C. 缺少稳定可审计的验证码解决方案

现有候选要么：

- 依赖外部 API key
- 依赖第三方打码服务
- 权限面偏大
- 安全审查未通过“可直接安装”标准

结论：

- 当前不能默认依赖“自动过验证码”作为主路径

### D. `nodriver-kit` 依赖来源未确认

本次抖音 skill 里引用了 `nodriver-kit`，但公开包源里没有直接找到可安装发布。

当前结论：

- **不要在依赖来源不清时盲装**
- 用户目标可先用已可运行浏览器链路完成

### E. 统一脚本入口已落地，但仍是最小版

当前已新增：

- `scripts/download-platform-video.py`

它已经能处理：

- 抖音公开视频页 URL → 真实 mp4 → 下载 → 校验

但仍未补齐：

- 关键词搜索 → 自动定位作者 / 第一条作品
- 作者主页异常页下的自动回退选择
- 其他平台适配器（Bilibili / Xiaohongshu / Kuaishou 等）

### E. “列表第一个视频”在异常页下可能存在语义歧义

当主页作品流本身报错时，“第一个视频”无法再完全依赖主页 DOM 直接读取。

此时必须明确区分：

- **理想定义**：主页作品流的第一条
- **当前可验证替代定义**：公开索引里当前最靠前、且能被作者主页 / 公开视频页双向验证的那条

后续若补齐登录态 / 作品流解析，应优先恢复到理想定义。

## 以后执行时的默认反应模板

当用户说：

- “去抖音下载这个人第一个视频”
- “去某平台搜一下然后下视频”
- “帮我把这个作者最新视频拿下来”

默认按下面反应：

1. 先容量预检（如有下载）
2. 先试平台内直接路径
3. 一旦被验证码 / 登录墙 / 服务异常拦住，立刻外搜定位
4. 先拿作者主页 / 公开视频其中任一可验证入口
5. 把媒体地址拿出来
6. 下载并校验
7. 给用户附件或下载地址
8. 若存在缺口，顺手记录进文档 / TOOLS / learnings

## 当前机器的额外注意事项

### 公司（Linux）这台 VM

- 浏览器自动化若出现：`No usable sandbox!`
- 对 `agent-browser` 需优先考虑：

```bash
agent-browser --args "--no-sandbox" ...
```

- 但如果已有 daemon 在跑，改参数前要先：

```bash
agent-browser close --all
```

否则新的 `--args` 会被忽略。
