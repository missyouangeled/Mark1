# 大批量操作防上下文溢出规则

> 生效日期：2026-06-03
> 触发条件：任何需要遍历/处理/列出大量文件的任务（修改类、查找类、重命名类等）

## 核心原则

**聊天里只放摘要，详细内容落地文件。**

---

## 执行流程

### 第一步：预判总量

在动手之前，先估算任务涉及的规模：

- 涉及多少文件/条目？
- 预计产生多少行输出？
- 输出内容平均行长多少？

> 例如：`find` 扫目录 → 数文件数 → 报告「约 800 个文件待处理」

### 第二步：与模型沟通容量

根据预估总量，判断当前模型能不能一口吞下：

| 预估值 | 策略 |
|--------|------|
| ≤ 50 条 | 可直接在聊天里展示全部 |
| 50 ~ 200 条 | 精简展示：只给分类计数 + 代表性样本（每类 2-3 个） |
| > 200 条 | **必须分批**，每批 ≤ 200，且只汇报计数不刷屏 |

> 宁可多问一句「这个量模型可能吃不下，我分批处理？」也​​不要硬塞。

### 第三步：分批执行

```
┌─ 判断总量 ──→ 模型能接受？ ──→ 一次执行
│                    │
│                    └── 接受不了 ──→ 拆成 N 批
│                                         │
│                          每批 ≤ 200 条，只汇报：
│                          • 分类名
│                          • 改名/命中数
│                          • 代表性样本 2-3 个
│                          • 结果落地文件路径
```

### 第四步：结果落地文件，不堆在聊天里

所有详细信息写进磁盘文件：

- **单次任务**：在对应目录下生成 `rename_log.txt` / `search_result.txt` 等
- **汇总文档**：任务完成后合并到 `docs/` 下形成总对照文档
- 聊天里只说：「✅ 完成，详情见 `docs/xxx.md`（N 条）」

## 输出规范

### ✅ 好的输出

```
【Military】改名 17 个
  样本：Roadblock_Type-03_01 → Roadblock_3_01
        Palatka_A_Grounp → Palatka_A_Group
        gaoshejiqiangyanti → Gaoshejiqiangyanti
  详情 → rename_log.txt
```

### ❌ 坏的输出

```
[DRY] Roadblock_Type-03_01.prefab → Roadblock_3_01.prefab
[DRY] Palatka_A_Grounp.prefab → Palatka_A_Group.prefab
[DRY] props_grounp_defense line01 (1).prefab → Props_Group_Defense_Line01_(1).prefab
... (800 行全打出来，撑爆上下文)
```

## Script 内置防溢出

所有批量脚本应默认内置防溢出能力：

1. `--dry-run` 默认只输出计数 + 样本，不输出全文
2. `--verbose` 显式开启全文输出（用户主动要求时）
3. 改名日志自动写入 `rename_log.txt`，不依赖聊天输出做记录

---

## 长时间任务的后台卸出

如果任务不只是输出量大、而且执行时间长：

- 用 `sessions_spawn` 卸到后台分身
- 完成后只回报摘要到主会话
- 主会话始终保持可对话状态

---

## 上下文健康度自检

当同一会话内累积多次大批量操作后，主动检查：

1. 本轮对话是否已经包含大量文件列表或预览输出？
2. 是否接近模型上下文上限？
3. 如果是 → 建议开 `/new` 新会话继续，所有成果已在文件中，新会话可无缝衔接

---

## 记忆要点

- **预判总量 → 沟通容量 → 分批落地 → 聊天只放摘要**
- 200 条是安全线，超过即分批
- 文件系统是持久记忆，聊天是临时窗口
- 炸上下文 = 浪费时间重做，预防比修复划算

## Unity 重命名计划文件标准格式

> 适用于 Props / Wall / 后续 Unity 资产文件夹的重命名计划文档

### 文件头部（HEADER）

```markdown
# <文件夹名> 重命名计划

> 生成时间: YYYY-MM-DD HH:MM
> 源目录: Assets/AssetScene/SceneModels/<文件夹名>/
> 总文件数: <N> | Prefab:<N> Material:<N> FBX:<N> 贴图:<N> .meta:<N> | <N> 个顶层文件夹
> 本计划 = V3 基础规则 + <专项规则>，合并为一份完整计划
```

### 规则章节

依次放：前缀规则 / 清理规则 / 目录结构规则 / 排除规则 / 特殊处理

### 六、文件清单

按顶层文件夹分组，每张表格固定 6 列：

```markdown
## <顶层文件夹名>/

| 原始地址 | 原始文件名 | 新地址 | 新文件名 | 类型 | 备注 |
|---|---|---|---|---|---|
```

### 列说明

| 列 | 含义 | 示例 |
|---|---|---|
| 原始地址 | 改名前的相对路径（来自原始文件系统或规则反推） | `Gate_double/Materials/Wall_Gate_Double.mat` |
| 原始文件名 | 改名前的文件名 | `Wall_Gate_Double.mat` |
| 新地址 | 改名后（含目录调整）的相对路径 | `Gate_double/Materials/Mat_Wall_Gate_Double.mat` |
| 新文件名 | 改名后的文件名 | `Mat_Wall_Gate_Double.mat` |
| 类型 | Prefab / Material / FBX / 贴图 / .meta | `Material` |
| 备注 | 补充说明，如"已确认""按规则反推""已确认（All 扁平化）" | `已确认（All 扁平化）` |

### .meta 文件处理

- `.meta` 文件始终跟随主文件，**不单独命名**
- 表格中 .meta 行单独列出，原始 / 新文件名为对应的 .meta 文件
- 根目录下的 `.meta` 在最后单开一个 `## <文件夹名>/（根目录 .meta）` 小节
