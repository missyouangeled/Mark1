# 方案与系统工作隔离规则

- 适用机器：通用
- 系统 / OS：通用
- 文档类型：跨机器共享规则

## 用途

这份文档用于明确区分以下三类内容，避免它们在 GitHub 仓库里互相混杂、互相干涉：

1. **方案 / 决策 / 调研结论**
2. **系统配置 / 脚本 / 维护说明 / 修复记录**
3. **日常工作记录 / 当天聊天摘要 / 临时执行痕迹**

## 第一类：方案

### 定义

“方案”指的是：

- 技术路线
- 架构取舍
- 调研结论
- 长期决策
- 尚未实施或部分实施的思路
- 项目级设计草案

### 归档位置

统一写入：

- `PLANS.md`

### 规则

- `PLANS.md` 只存放“方案内容”本身
- 不在 `PLANS.md` 里记录系统现状、脚本细节、机器专用维护步骤、临时修复命令
- 如果某个方案后来真正落地了，实施细节应写回对应的系统文档、项目文档或维护说明，而不是继续堆在 `PLANS.md`

## 第二类：系统配置 / 维护内容

### 定义

“系统配置 / 维护内容”指的是：

- OpenClaw gateway 配置
- watchdog / systemd / Scheduled Task
- 本机维护说明
- 系统修复记录
- 脚本说明
- 机器专用运行规则
- 本机部署、更新、恢复方法

### 归档位置

统一写入：

- `docs/` 下对应说明文档
- `TOOLS.md` 的环境说明条目
- `HOST_CONTEXT.md`
- 对应脚本文件头部说明

### 规则

- 机器专用内容必须明确写出 `适用机器` 和 `系统 / OS`
- 系统内容不能只写在 `PLANS.md` 里
- 如果是本机维护说明，优先写入对应维护文档，例如：
  - `docs/掌机-Windows-OpenClaw-维护说明.md`
  - `docs/公司-Linux-OpenClaw-维护说明.md`

## 第三类：日常工作 / 临时记录

### 定义

“日常工作 / 临时记录”指的是：

- 当天做了什么
- 用户今天新增了什么要求
- 某次会话里达成了什么共识
- 临时执行过哪些动作
- 需要沉淀进记忆的当天摘要

### 归档位置

统一写入：

- `memory/daily/YYYY-MM-DD.md`

### 规则

- daily 文件用于“当天发生了什么”
- 不拿 daily 文件代替正式方案文档
- 不拿 daily 文件代替正式系统维护说明

## 总隔离原则

一句话规则：

> **方案进 `PLANS.md`，系统维护进 `docs/` / `TOOLS.md` / 脚本说明，日常执行痕迹进 `memory/daily/`；三类内容必须分开，不能互相代替。**

## GitHub 归档要求

凡是准备推送到 GitHub 的内容，提交前都必须先判断属于哪一类：

- 属于方案 → 写到 `PLANS.md`
- 属于系统维护 → 写到对应维护文档 / 脚本说明
- 属于日常记录 → 写到 `memory/daily/`

禁止出现以下混用：

- 在 `PLANS.md` 里堆系统脚本使用说明
- 在系统维护文档里夹杂大量未定稿方案讨论
- 在 daily 记录里代替正式维护文档或正式方案存档
