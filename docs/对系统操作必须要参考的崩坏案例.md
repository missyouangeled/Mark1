## CASE-20260706-004 — MiniMax-M3 大上下文下 stopReason=length 导致工具调用死锁

### 项目
2026-06-22 路由大修后把 MiniMax-M3（MiniMax M3，1M 上下文稀疏注意力模型）作为主会话 / 子 agent / cron 的默认模型。7/2 和 7/6 上午先后两次出现"主会话工具调用完全失能"——表现为任何 `exec echo hi` 都不返回，工具调用像被卡在队列里。

### 机器 / 环境
- 机器：公司 Linux（VMware 客体）
- OpenClaw 版本：v2026.6.11
- 主会话模型：`minimax/MiniMax-M3`
- session jsonl 大小（截断前）：约 21k 行（497MB jsonl + 1.78MB trajectory）→ 上下文已逼近"长度截断窗口"
- 时间窗：7/2 上午、7/6 上午 10:40（连续两次）

### 触发动作
1. 用户和主会话正常聊天，session jsonl 持续累积
2. 单次请求的提示词 token 数超过 MiniMax-M3 在该次路由上的实际可用长度窗口
3. M3 服务端返回 `stopReason=length`（不是 `end_turn`，是"输出被截断"）
4. OpenClaw 客户端把后续 tool_use 全部塞进等待队列，**但 M3 已经 end_of_turn**，客户端死锁
5. 表现：所有 `exec`、`read`、`web_search` 都"挂起"不返回，主会话实质瘫痪
6. 重启 gateway 也无法恢复（重启后第一轮新请求又踩同一个雷）

### 结果
- 主会话工具调用完全失能，**echo hi 都不返回**
- 唯一能恢复的手段：**截断 session jsonl**（保留最近 50 行）+ 移除 .lock 文件，不重启 gateway
- 截断后主会话立即恢复——但**只要上下文再次涨上去，问题会原样复现**
- 7/2 和 7/6 两次都是同一根因

### 已知技术线索
1. **MiniMax-M3 是 1M 上下文、稀疏注意力（MiniMax Sparse Attention, MSA）模型**。MSA 架构特性："at long context, you do not need to look at the whole past to predict the next token"——即**上下文越长，sparse attention 漏读概率非线性上升**
2. M3 在 NVIDIA Build 平台也有过类似症状：Playground 出现"125+ Queue Requests"卡死（外部论坛报告）
3. OpenClaw 客户端的默认行为：**模型返回 stopReason=length 时，会把后续 tool_use 全部塞入等待队列**，期待模型"继续说话"——但 length 截断后模型已经 end_of_turn，客户端死锁
4. 用户原配置 `compaction.memoryFlush.softThresholdTokens = 32000` + `keepRecentTokens = 25000`——**触发时机太晚**：等真到 32k 时，M3 已经在 length 边缘抖动，safeguard 模式来不及收
5. 7/2 第一次发病前，用户使用 M3 约 10 天；7/6 第二次发病时 session 累积到 21k 行。这印证"**不是模型突然坏了，是上下文逼近 MSA 窗口边界**"

### 最可能原因
**MiniMax-M3 稀疏注意力架构 + OpenClaw 客户端对 stopReason=length 的默认处理行为**的组合 bug：
- 触发条件：单次请求的提示词 token 数逼近 M3 的"实际可用长度窗口"（不一定是 1M 标称值，MSA 在长上下文下输出会被强制截断）
- 直接原因：M3 返回 `stopReason=length` + OpenClaw 客户端把后续 tool_use 塞入永不释放的等待队列
- 为什么"用了一段时间"才出问题：早期会话上下文小（< 30k token），MSA 还能 cover；用户累计对话越来越多，单次请求的提示词长度超过 MSA 的"覆盖窗口边界"，length 抖动开始频繁

### 为什么会扩大成系统级事故
1. **主会话实质瘫痪**——所有需要工具调用的操作（exec、read、cron run、模型切换）全部卡死
2. 唯一恢复手段是**手动截断 session jsonl + 删 lock**——这需要从外部终端操作，**主会话自己救不了自己**
3. 如果用户在生产时段（早上 10:40 到公司、开始一天工作）发病，**直接卡死一上午**
4. 重启 gateway 没用——重启后第一轮新请求又踩同一个雷
5. 用户的 cron（早安问候 8:15、午餐提醒 12:00）也用 MiniMax-M3，**理论上 cron 触发时如果 session 大也可能踩**——但目前 cron 触发的是 isolated session（小上下文），未观测到踩雷

### 结论
**MiniMax-M3 作为默认模型时，OpenClaw 没有"自动避免 length 死锁"的保护**——主会话的 compaction 阈值必须设得比"M3 length 抖动窗口"**早**很多，不能等真到边缘才收。

### 后续默认规则
1. **不再让主会话靠"不截断"硬扛**。`compaction.memoryFlush.softThresholdTokens` 必须 ≤ 15000（远小于 M3 length 抖动窗口），触发即收
2. **`compaction.keepRecentTokens` ≤ 12000**——截断后只留近 12k，宁可牺牲长尾历史也不要踩雷
3. **`compaction.maxHistoryShare` ≤ 0.4**——历史最多占上下文 40%，给生成留够 headroom
4. **`compaction.mode` 保持 `safeguard`**（OpenClaw 当前版本只有 `default` / `safeguard` 两档，没有 `enforce`）
5. **`truncateAfterCompaction: true`** 必须显式开启，确保截断真的发生
6. **不动 `agents.defaults.model.primary` 路由顺序**——用户明确选择保持 MiniMax-M3 为主，不接受在 length 时切到 agnes 兜底（接受质量可能下降的风险）
7. 任何对 compaction 参数的修改，**必须先备份 `openclaw.json`**、`config validate` 通过、**不 restart gateway**（按 CASE-20260706-003）
8. 一旦观察到 M3 再次 length 抖动（任何一次 `stopReason=length` 出现），**立即**：
   - 截断当前 session jsonl 到 ≤ 50 行
   - 删除 `.lock` 文件
   - 不 restart gateway
9. **不要为了"在主会话内自救"重启 gateway**——按 CASE-20260706-003，从主会话内 restart 必崩
10. 监控 cron：早安 / 午餐虽然用 M3，但 isolated session 上下文小，**目前未观测踩雷**。如果未来 cron 触发后看到 "tool_use queue" 类症状，同 8 处理

### 复盘要求
- 此案例列入「系统操作前必须参考的崩坏案例」
- 每次 MiniMax-M3 出现 length 抖动或工具调用卡死，**必查本案例**
- 每次修改 `agents.defaults.compaction.*` 参数前必查本案例，确认改动方向一致
- 每次主会话出现"工具调用失能"症状，**先看 session jsonl 行数**（> 10k 行基本就是 M3 length 雷），**不要先 restart gateway**
- 如果未来 M3 服务端修了这个 length 抖动问题，可以把 `softThresholdTokens` 调回 32k；但**先观测一周确认无 length 抖动再调**

## CASE-20260706-003 — 从 OpenClaw 主会话内执行 `openclaw gateway restart` 会打断主会话

### 项目
全系统体检 + 维护。用户明确要求"只保稳定"，所以做了 3 项小改：cron timeout 90s→180s、cron fallback ollama→litellm/agnes-2.0-flash、全局模型 fallback 链补全。改完 config validate 显示正常，提示 "Restart the gateway to apply"。

### 机器 / 环境
- 机器：公司 Linux（VMware 客体）
- OpenClaw 版本：v2026.6.11
- Gateway 启动方式：systemd user service（`openclaw-gateway.service`）
- 当前会话：`agent:main:main`，通过本地 127.0.0.1:18789 连 Gateway

### 触发动作
1. 修改 `agents.defaults.model.fallbacks` 后，`openclaw config set` 输出 "Updated agents.defaults.model.fallbacks. Restart the gateway to apply."
2. 在当前 main session（正在执行 tool 调用）内运行 `openclaw gateway restart`
3. main session 被 SIGTERM 中断，本轮 tool 调用全部丢失
4. systemd 显示 gateway 状态 `deactivating (stop-sigterm)`，但不知道是因为自己把自己杀了

### 结果
- main session 连接被强制断开（`[System] Your previous turn was interrupted by a gateway restart` 系统提示）
- 当前轮的所有未完成 exec、cron、模型验证结果都没保存
- gateway 进程在 systemd 服务里被 kill 后要重新拉起
- 因为 systemd 单元本身完整，几秒后 systemd 自动重启了 gateway（pid 从 1996 变成 8869），无需人工干预，但用户能看到 5-15 秒 gateway 不可达

### 已知技术线索
1. `openclaw gateway restart` 等价于 `systemctl --user restart openclaw-gateway`
2. systemd restart 杀的是 main PID（节点进程），所有活动 WebSocket 连接都断
3. 从 main session 内触发 restart = 自杀式操作，因为命令本身就是通过断掉的连接发起的
4. systemd unit 有 `Restart=always` 或类似机制会自动拉起新的 gateway（这是这次没崩的原因）
5. `openclaw config set` 修改需要 restart 才生效——这是设计如此

### 最可能原因
当 `openclaw config set` 输出的 "Restart the gateway to apply" 时，正确做法是从外部触发 restart（另一个 session、cron job 里、systemctl 直接），而不是从当前正在运行的 session 里。回滚改动不需要 restart；只想验证改动是否落盘，应该用 `openclaw config get` 而不是触发 gateway 重启。

### 为什么会扩大成事故
1. main session 被打断 → 当前轮的所有诊断信息丢了一半
2. 用户看到 `[System] Your previous turn was interrupted by a gateway restart`，会怀疑系统不稳
3. 如果 systemd 没有自动 restart 兜底，会变成 CASE-20260616-002 类似的 gateway 长时间不可达
4. 在生产时间（早上 10:14，用户刚到公司）打断会话，可能错过早安问候 cron 的下一次注入

### 结论
**从 OpenClaw 主会话内**不要**主动跑 `openclaw gateway restart`。**

### 后续默认规则
1. 修改 `openclaw.json` 后，**先验收改动**：`openclaw config get <path>`、`openclaw config validate`，不放心可以 `openclaw cron get <id>` 看具体 job
2. **不要为了"让改动生效"在当前 session 跑 gateway restart**。如果只是小改 fallback 链、timeout 等，Gateway 在下次 chat 请求时也会读到新配置（fork 进程或 reload 热加载）
3. 真要 restart：
   - 选项 A：让 cron job 自己触发（设个一次性 cron）
   - 选项 B：起一个新的 isolated session 跑 restart
   - 选项 C：保留 main session 不动，**手动从终端**（不是 OpenClaw 内）跑 `systemctl --user restart openclaw-gateway`
4. restart 前必先确认：
   - `systemctl --user status openclaw-gateway` 是 active
   - 当前 main session 没有未提交的 tool 调用 / 用户输入
   - 备份已经做了（cp openclaw.json openclaw.json.bak.<date>）
5. 这次 case 必须加进启动协议：每次启动先读崩坏案例，读到这里

### 复盘要求
- 此案例列入「系统操作前必须参考的崩坏案例」
- 每次涉及 `openclaw gateway restart` / `systemctl --user restart openclaw-gateway` 前必查本案例
- 如果只是改 cron / agents / models 配置、且 config validate 已通过，可以**不 restart**——Gateway 在下个 chat 请求时会读最新配置

## CASE-20260616-002 — Python 脚本直写 openclaw.json 导致 Gateway 启动失败（配置重复键）

### 项目

为 Mark42 配置多 Agent 协作（researcher + coder + agentToAgent），用 Python 脚本直接 `json.dump` 写入 `openclaw.json`。

### 机器 / 环境

- 机器：公司 Linux（VMware 客体）
- OpenClaw 版本：v2026.6.6
- 重启方式：`systemctl --user restart openclaw-gateway`

### 触发动作

1. 用 Python `json.dump()` 直接追加 `agents.list`（researcher + coder）和 `agents.subagents.allowAgents` 和 `tools.agentToAgent`
2. 写入后重启 Gateway
3. Gateway 启动失败，报 JSON 解析错误

### 结果

- `openclaw.json` 中出现了两个重复的 `"agents"` 顶层键
- Python `json.load` 加载时第二个 `agents` 覆盖第一个，导致 `defaults` 段丢失（只剩下 `agents.list`）
- `openclaw gateway restart` 直接报错，前台连不上
- 用户需要手动从终端重启并看到报错

### 已知技术线索

1. Python `json.dump` 本身不会对重复键报错
2. `json.load` 时重复键以最后一个胜出
3. OpenClaw Gateway 启动时解析 JSON 遇到重复键会直接报配置错误
4. 用 `openclaw config set` 逐条追加则不会产生重复键，因为 OpenClaw 自己在做合并

### 最可能原因

直接写 JSON 文件绕过了 OpenClaw 的配置合并逻辑。脚本写入了第二个 `agents` 对象块（只含 `list`），但文件中已有完整的 `agents` 块（含 `defaults`）。结果 JSON 里出现两个 `agents` 键。

### 为什么会扩大成事故

1. JSON 重复键在 Python 里静默通过（不报错）
2. 只在 Gateway 启动时才暴露，修复窗口在下次重启
3. 如果手动恢复不及时，Gateway 会反复重启失败

### 结论

**以后修改 `openclaw.json` 一律用 `openclaw config set`，不直接写文件。**

### 后续默认规则

1. 任何 `openclaw.json` 修改：先用 `openclaw config validate` 确认当前配置干净
2. 增量追加用 `openclaw config set <path> <value> --json`
3. 修改前备份：`cp openclaw.json openclaw.json.bak.$(date +%Y%m%d-%H%M)`
4. 修改后验证：`openclaw config validate`
5. 严禁用 Python/`write`/`edit` 直接改写 `openclaw.json` 的 agents 段

### 复盘要求

- 此案例列入「系统操作前必须参考的崩坏案例」
- 每次改 `openclaw.json` 前先查本案例
- 如果已用正确方式（`openclaw config set`）写入，不再需要额外核对

## CASE-20260513-001 — Windows 宿主机温度读取导致系统崩溃

### 项目

为 OpenClaw 本地健康监督接入 Windows 宿主机温度读取能力，希望把宿主机真实 CPU 温度桥接到 Linux 客体的本地健康页。

### 机器 / 环境

- 机器角色：Windows 宿主机（承载当前工作环境）
- 系统：Windows 10（从远程会话里看到 PowerShell 版本 `5.1.18362.145`，属于较老系统链路）
- 连接关系：Linux VMware 客体通过 SSH 直连宿主机 `192.168.233.1:22`
- 风险上下文：该机器承载工作环境，恢复成本高，不适合把底层硬件监控当成低风险试验

### 触发动作

1. 为读取宿主机温度，先在宿主机侧准备了温度桥接方案
2. 由于 Windows 自带 WMI / ACPI 温度源拿不到可用 CPU 温度，转而引入第三方硬件监控工具 `LibreHardwareMonitor`
3. 工具被落到：
   - `E:\DESK\OpenClawBridgeTools\LibreHardwareMonitor\`
4. 用户在宿主机上手动打开 `LibreHardwareMonitor.exe`

### 结果

- 宿主机随后出现严重系统级崩溃
- 用户描述为：系统直接崩溃，且恢复成本高，最终需要重装 Windows，并花了较长时间抢修工作环境与工作资料
- 该事故已经超过“普通软件崩溃 / 闪退”范围，应视为**系统稳定性事故**

### 已知技术线索

#### 1) 不是单纯“普通用户态读温度”

对下载包做静态检查后，发现其中明确包含与底层硬件访问相关的符号/组件痕迹，例如：

- `InstallPawnIO`
- `PawnIO`
- `IEmbeddedControllerIO`
- `WindowsEmbeddedControllerIO`
- `Embedded Controller`

这说明该工具为了拿到更多硬件传感器信息，可能会下探到更底层的 I/O / EC / 传感器访问链路，而不只是安全地调用普通公开 API。

#### 2) 社区存在相似公开报告

本次调研中已看到外部公开讨论/问题报告与此模式相似，例如：

- `pawnIO installation causes BSOD – LibreHardwareMonitor cannot ...`
  - `https://github.com/LibreHardwareMonitor/LibreHardwareMonitor/issues/2258`
- `Use LibreHardwareMonitor without using PawnIO just for monitoring ...`
  - `https://github.com/LibreHardwareMonitor/LibreHardwareMonitor/discussions/2149`
- 相关生态中也有类似反馈：某些机器在 PawnIO / 类似底层监控驱动链路下会蓝屏或系统崩溃

> 注意：这些公开案例并不等于“已经精确证明本机就是同一个驱动点爆炸”，但足以证明这类工具链并非低风险。

### 最可能原因（高概率解释，不装作精确法医结论）

最可能的链路是：

1. 用户想读取 CPU 温度
2. Windows 自带接口无法稳定给出真实温度
3. 第三方硬件监控工具为了拿到温度，走了更底层的驱动 / EC / 直接 I/O 链路
4. 该链路与当前宿主机的系统版本、驱动栈、主板/EC/显卡/芯片组环境不兼容
5. 结果不是“应用崩溃”，而是系统级崩溃
6. 崩溃进一步导致高恢复成本，最终逼近“只能重装 / 大量抢修”的后果

### 为什么会严重到“几乎不可修复”

因为一旦问题发生在驱动/内核级，代价就不再是普通应用闪退，而可能扩展为：

- 蓝屏 / 死机 / 强制重启
- 系统状态损坏
- 文件系统或注册表状态异常
- 工作环境恢复成本极高
- 即便理论上“可修”，实际也可能不如重装来得快

对承载工作环境的机器来说，这种风险是不可接受的。

### 结论

这次事故说明：

> 真正危险的不是“读取温度”这件事本身，而是为了读取温度而触发的底层硬件访问/驱动链路。

因此以后在工作机器、宿主机、第三方电脑、朋友电脑上：

- **不要再把硬件监控 / 温度读取 / 传感器工具当作低风险动作默认尝试**

### 后续默认规则

对类似任务，默认改为：

1. 先停在低风险方案
   - 只看已有日志 / 事件查看器 / 崩溃转储
   - 不主动再装新的硬件监控工具
2. 如果用户仍坚持继续
   - 必须先再次确认
   - 必须明确说明“有系统崩溃风险”
   - 优先隔离到非工作环境、可回滚、可重装、无第三方重要数据的机器
3. 对朋友/第三方电脑
   - 默认更保守
   - 不主动尝试此类底层工具

### 复盘要求

如果未来要继续分析这类事故，优先做：

- 被动分析：事件查看器、可靠性监视器、崩溃转储、系统日志
- 禁止默认做：再次主动拉起相同硬件监控工具复现

## CASE-20260706-005 — 救命 1 自测把自己救没了

### 项目
长记忆项目（自动锚定方案）的救命 1 cron：每 5 分钟扫 main session jsonl 行数

### 机器 / 环境
- 公司 Linux（missyouangeled-VMware-Virtual-Platform）
- 主 agent session: `68e35bdd-4acd-4a57-80ed-a1fda8b8102e.jsonl`
- 时间：2026-07-06 ~11:53

### 触发动作
为验证救命 1 cron 触发后能成功调用 `openclaw-proactive-inject.py` 注入消息到主会话：

1. 用 `openclaw cron edit --command` 把救命 1 的阈值从 12000 改成 **100**
2. 备份主 session 为 `.pre-test-backup`（11:53，638KB）
3. `openclaw cron run 7f82f74c-c8a3-41a4-870b-a2943b8f13ff` 手动触发

### 结果
- 主 session 从 245 行被截断到 55 行（638KB → 95KB）
- 自动产生两个 corrupt 备份：
  - `.corrupt-2026-07-06T03-53-42-398Z-7a9f4848.jsonl`（11:53）
  - `.corrupt-2026-07-06T04-25-46-124Z-aa846093.jsonl`（12:25）
- 系统报告 `missing tool result in session history; inserted synthetic error result for transcript repair`
- 主会话被迫中断 ~30 分钟（11:53 → 12:25）
- **数据未丢失**（两次备份齐全：`.pre-test-backup` + `.bak.1155.TEST`，各 638KB）

### 已知技术线索
- 救命 1 的 `if LINES > 阈值` 分支执行了：
  ```sh
  cp "$SESS" "$BAK"
  tail -50 "$SESS" > /tmp/sess-tail.jsonl
  mv /tmp/sess-tail.jsonl "$SESS"
  rm -f "$SESS.lock"
  ```
- OpenClaw 在 `mv` 后检测到 session 文件被外部覆盖，自动把它备份为 `*.corrupt-*.jsonl`
- `rm -f "$SESS.lock"` 删的是 OpenClaw 自己的会话锁文件 —— **额外破坏了主会话的锁状态**

### 最可能原因
救命 1 的"截断动作"是**直接覆盖 main session jsonl 文件**。但 main session 是 main agent 当前正在使用的活文件：

1. main agent 正在读这个 jsonl（用于当前会话上下文）
2. 救命 1 在另一进程里直接 `cp + tail + mv` 覆盖它
3. main agent 的读句柄指向旧 inode → 读到的全是旧数据直到 EOF
4. OpenClaw 检测到文件被外部改动 → 标记为 corrupt → 自动备份
5. main session 因为锁文件被 `rm -f` + 文件被覆盖 → **当前会话被迫中断**

**这是设计层面的根本问题**，不是补丁能修的。

### 为什么会扩大成系统级事故
- 我（main agent）**自己就在主会话里**
- 救命 1 触发 = **把我自己截断 = 救命 1 把救命对象干掉了**
- 救命 1 唯一的设计目的就是"在 session 失控时紧急保护"
- 但它**最危险的时候就是它最该触发的时候**（session 失控）—— 那时候救命 1 跑得动吗？能跑动 = 它在破坏自己；跑不动 = 它失效
- 阈值改 100 测试 = 把"应该救命的时候"提前到现在 = **当前就被救没了**

### 结论
> 救命 1 的"自动截断 session"设计是**自残式救命**，本质上不可用。

### 后续默认规则

1. **救命 1 重新设计**：
   - ✅ 只监控 + 告警 + 不动 session
   - ❌ 不允许自动截断、覆盖、删除任何 session 文件
   - 截断必须由用户在**新 session** 手动处理

2. **测试救命逻辑时绝对不能拿主 session 当目标**：
   - 必须用**测试 session**（独立 jsonl 文件，放 sessions 目录但不是主会话）
   - 或者用**沙箱副本**（`/tmp/test-sess.jsonl` + 修改 grep 路径）

3. **救命 1 cron 命令本身不能再改阈值**：
   - 阈值必须写死在文件里，不能用 `--command` 临时改
   - 改阈值要走"新建 cron + 删旧 cron"的完整流程

4. **救命类脚本的"破坏性操作"必须先 dry-run**：
   - 任何动 session 文件的动作（cp / tail / mv / rm）必须有 `--dry-run` 默认开关
   - 真跑要 `--execute-now` 显式触发

5. **systemd 服务 / OpenClaw gateway 配置热加载**会触发 gateway 自动 restart —— **不要相信"我还没 restart gateway"**。任何改配置的动作都要假设 gateway 随时会重启。

### 复盘要求
未来任何"自动化救命脚本"：

- 必须明确写出"危险对象清单"（它会动哪些文件、哪些是活文件）
- 必须明确写出"测试时的隔离方案"
- 必须先 dry-run 跑通，再加自动触发
- 任何会动 `~/.openclaw/agents/*/sessions/*.jsonl` 的脚本，默认就是高风险
