## 使用规则

- 硬件监控工具
- 传感器读取
- 驱动层 / 内核级辅助
- Embedded Controller / EC / SMBus / Ring0 / 直接 I/O
- 可能影响系统稳定性的清理、修复、调参、第三方底层工具

若案例中已经出现过“直接导致系统崩溃 / 蓝屏 / 无法正常恢复”的相似模式，则默认：

- 不主动继续尝试
- 先明确说明风险
- 需要用户再次确认
- 优先退回低风险、可回滚方案

## CASE-20260513-001 — Windows 宿主机温度读取导致系统崩溃

### 项目

为 OpenClaw 本地健康监督接入 Windows 宿主机温度读取能力，希望把宿主机真实 CPU 温度桥接到 Linux 客体的本地健康页。

### 机器 / 环境

- 机器角色：Windows 宿主机（承载当前工作环境）
- 系统：Windows 10（从远程会话里看到 PowerShell 版本 `5.1.18362.145`，属于较老系统链路）
- 连接关系：Linux VMware 客体通过 SSH 直连宿主机 `192.168.233.1:22`
- 风险上下文：该机器承载工作环境，恢复成本高，不适合把底层硬件监控当成低风险试验

### 触发动作

1. 为读取宿主机温度，先在宿主机侧准备了温度桥接方案
2. 由于 Windows 自带 WMI / ACPI 温度源拿不到可用 CPU 温度，转而引入第三方硬件监控工具 `LibreHardwareMonitor`
3. 工具被落到：
   - `E:\DESK\OpenClawBridgeTools\LibreHardwareMonitor\`
4. 用户在宿主机上手动打开 `LibreHardwareMonitor.exe`

### 结果

- 宿主机随后出现严重系统级崩溃
- 用户描述为：系统直接崩溃，且恢复成本高，最终需要重装 Windows，并花了较长时间抢修工作环境与工作资料
- 该事故已经超过“普通软件崩溃 / 闪退”范围，应视为**系统稳定性事故**

### 已知技术线索

#### 1) 不是单纯“普通用户态读温度”

对下载包做静态检查后，发现其中明确包含与底层硬件访问相关的符号/组件痕迹，例如：

- `InstallPawnIO`
- `PawnIO`
- `IEmbeddedControllerIO`
- `WindowsEmbeddedControllerIO`
- `Embedded Controller`

这说明该工具为了拿到更多硬件传感器信息，可能会下探到更底层的 I/O / EC / 传感器访问链路，而不只是安全地调用普通公开 API。

#### 2) 社区存在相似公开报告

本次调研中已看到外部公开讨论/问题报告与此模式相似，例如：

- `pawnIO installation causes BSOD – LibreHardwareMonitor cannot ...`
  - `https://github.com/LibreHardwareMonitor/LibreHardwareMonitor/issues/2258`
- `Use LibreHardwareMonitor without using PawnIO just for monitoring ...`
  - `https://github.com/LibreHardwareMonitor/LibreHardwareMonitor/discussions/2149`
- 相关生态中也有类似反馈：某些机器在 PawnIO / 类似底层监控驱动链路下会蓝屏或系统崩溃

> 注意：这些公开案例并不等于“已经精确证明本机就是同一个驱动点爆炸”，但足以证明这类工具链并非低风险。

### 最可能原因（高概率解释，不装作精确法医结论）

最可能的链路是：

1. 用户想读取 CPU 温度
2. Windows 自带接口无法稳定给出真实温度
3. 第三方硬件监控工具为了拿到温度，走了更底层的驱动 / EC / 直接 I/O 链路
4. 该链路与当前宿主机的系统版本、驱动栈、主板/EC/显卡/芯片组环境不兼容
5. 结果不是“应用崩溃”，而是系统级崩溃
6. 崩溃进一步导致高恢复成本，最终逼近“只能重装 / 大量抢修”的后果

### 为什么会严重到“几乎不可修复”

因为一旦问题发生在驱动/内核级，代价就不再是普通应用闪退，而可能扩展为：

- 蓝屏 / 死机 / 强制重启
- 系统状态损坏
- 文件系统或注册表状态异常
- 工作环境恢复成本极高
- 即便理论上“可修”，实际也可能不如重装来得快

对承载工作环境的机器来说，这种风险是不可接受的。

### 结论

这次事故说明：

> 真正危险的不是“读取温度”这件事本身，而是为了读取温度而触发的底层硬件访问/驱动链路。

因此以后在工作机器、宿主机、第三方电脑、朋友电脑上：

- **不要再把硬件监控 / 温度读取 / 传感器工具当作低风险动作默认尝试**

### 后续默认规则

对类似任务，默认改为：

1. 先停在低风险方案
   - 只看已有日志 / 事件查看器 / 崩溃转储
   - 不主动再装新的硬件监控工具
2. 如果用户仍坚持继续
   - 必须先再次确认
   - 必须明确说明“有系统崩溃风险”
   - 优先隔离到非工作环境、可回滚、可重装、无第三方重要数据的机器
3. 对朋友/第三方电脑
   - 默认更保守
   - 不主动尝试此类底层工具

### 复盘要求

如果未来要继续分析这类事故，优先做：

- 被动分析：事件查看器、可靠性监视器、崩溃转储、系统日志
- 禁止默认做：再次主动拉起相同硬件监控工具复现
