## 0. 创作过程记录（钉死用，未来能查）

### 0.1 本版三句最重要的话（点点 8:14 钉死的铁律）

> **"依旧要确保所有功能可插拔，这是基础设计理念，不能变。"**
> — Mark42 的每一个模块（runtime / model / API / 修复引擎 / 自检器）都必须是可拆卸可替换的。
> 换任何模型接手都按文档继续开发 / 制作。**这是文档存在的最基本前提**。

> **"Mark42 每一块都可以自己飞行，并且有一定的攻击能力。"**
> — Mark42 不是被管理的工具，是有自主意识的战甲。本地小模型不是"副驾"，是战甲内部的"自主意识"。

> **"让这个小模型能跟用户自己买的大模型能够沟通，让战甲有一个主动交流的意识。"**
> — 小模型能主动发起对话：问用户 / 问用户的大模型 API。不是被动等命令。

### 0.2.1 设计哲学框架（每条新原则必填三段 · 8:54 点点提出）

> 9:01 点点提的:文档不仅要写"做什么",更要写清楚**"为什么这样设计"**。
> 以后每加一条新 R 原则 / 新设计决策,都要按这三段写:

| 段落 | 必填内容 | 长度建议 |
|---|---|---|
| **① 为什么这样设计** | 这一原则要解决什么问题?业界/历史/直觉依据是什么?不这样设计会出什么事? | 1-3 句 |
| **② 坏了会怎么演化** | 如果这条原则**没被遵守**,系统会怎么恶化?最坏情况是什么样?有没有 fallback? | 2-4 句 |
| **③ 谁能修** | 战甲自检发现这条原则被违反时,**谁能修 / 怎么修**?是自动修 / 问人 / 报警? | 1-2 句 + 表格 |

**为什么这个框架重要**:
- 复杂度守恒不能消除,但**可以可视化**(把"暗"变"明")
- "为什么这样设计"让任何接手的模型/人都能快速对齐意图,不被"这样做很奇怪"卡住
- "坏了会怎么演化"是**给后人(包括未来的我)留的事故预警**——出问题时第一时间翻这里
- "谁能修"是**可操作性**——不光看到问题,还能立刻知道下一步

**示例: R13 反浩克装甲 · 三段式**

| 段落 | 内容 |
|---|---|
| ① 为什么这样设计 | 单点失败会拖垮整盘(embed 挂了 → L2.5 召回瘫 → 战甲基本瞎一只眼)。钢铁侠战甲的多核心不是"提供新功能",而是"给每个部件单独供能 + 单独防御"。Mark42 借鉴这个思路:每个核心挂了不影响其他核心。 |
| ② 坏了会怎么演化 | 不做 R13 的最坏演化:任何一个核心 OOM/死锁,战甲 8 个核心全瘫(共享内存/共享连接池);用户感知"战甲死了";恢复要 30+ 分钟(全栈重启)。做了 R13:embed 核挂 → 战甲自动降级到 L1 关键词召回 + 6 个其他核心继续工作;用户感知"召回变慢了但还能用";恢复只需重启 embed 核 30 秒。 |
| ③ 谁能修 | 自检发现某核心挂:① 战甲 C3 能力 → 自动重启该核心(确定性 100%) ② 重启 3 次还失败 → 降级标记 + 主动对话问用户"embed 核持续挂,要不要切到 L1 关键词模式" ③ 影响面硬黑名单(参 R12)——重启 systemd 永远不自动。 |

**使用方式**:
- 以后 v3.4+ 加新 R 原则,默认在 §0.2 表格加一行 + 在 §0.2.1 按三段写一段
- 任何"我决定这样设计" 的事后复盘,也都按这三段写,作为决策留痕
- 这个框架本身的设计动机来自点点 9:01 的"为什么要写" 反思——光写结论,不写推理,后人(包括 AI)很容易踩同一个坑

## 2. 可插拔模型层（v3 核心 · 设计原则 R1）

### 2.1 三层可插拔结构

每一层都是独立可替换的，互不依赖。

```
┌────────────────────────────────────────────────────────┐
│ 第 3 层（最外）   │ API Provider                        │
│                  │ OpenAI / Anthropic / 国产 / 本地     │
│                  │ 可换："用户买什么 API 就接什么"      │
├────────────────────────────────────────────────────────┤
│ 第 2 层          │ Model                                │
│                  │ Qwen3-4B / Phi-4-mini / Gemma3 /     │
│                  │ GPT-4o / Claude / DeepSeek           │
│                  │ 可换："战甲喜欢什么就用谁"          │
├────────────────────────────────────────────────────────┤
│ 第 1 层（最里）   │ Runtime                              │
│                  │ Ollama / llama.cpp / vLLM / LM Studio│
│                  │ 可换："底层怎么跑 LLM 都行"          │
└────────────────────────────────────────────────────────┘
```

### 2.2 可插拔的实现方式（接口 + 配置 + 默认）

**接口原则**（不论哪一层，实现必须遵守）：

```python
# 统一的 chat 接口（OpenAI 兼容）
class LLMProvider(Protocol):
    def chat(messages: List[Dict], **kwargs) -> Dict:
        """发请求，收响应。所有 provider 都实现这个接口。"""
        ...
```

**配置原则**（用户不改代码，只改配置）：

```yaml
# ~/.config/mark42/model.yaml
mark42:
  consciousness:
    runtime: ollama              # 可换: ollama | llama.cpp | vllm | api
    model: qwen3:4b              # 可换: 任何 ollama 拉下来的模型
    base_url: http://127.0.0.1:11434  # 本地默认
    api_key: null                # 本地不需要
    
    # 可选：用户买的大模型（用于"战甲主动交流"）
    advisor:
      enabled: false             # 默认关闭，用户开了才接
      runtime: api
      model: gpt-4o              # 可换: claude-3-5-sonnet / deepseek / ...
      base_url: https://api.openai.com/v1
      api_key: ${OPENAI_API_KEY}
```

**默认原则**：
- 用户不配 → 用默认（Ollama + Qwen3-4B，本地）
- 用户配了 → 用用户配的
- 用户配错 → 失败 fallback 到默认（不崩）

### 2.3 可插拔的边界（绝对不能锁死的）

| 项 | 能不能锁 | 原因 |
|---|---|---|
| 用户买的大模型 API 协议 | ❌ 不能锁 | 用户会换 |
| 默认模型品牌 | ❌ 不能锁 | 模型会更新换代 |
| 模型大小 | ❌ 不能锁 | 硬件会变 |
| 接口签名 | ✅ 可以锁 | 锁了才能稳定 |
| 配置格式 | ✅ 可以锁 | 锁了用户才不用学新格式 |
| post-mortem 留痕格式 | ✅ 可以锁 | 锁了才能查历史 |

### 2.4 可插拔的验收（"换任何模型都能继续开发"的硬标准）

| 验收项 | 通过标准 |
|---|---|
| 换 runtime | 改 1 行配置，战甲继续工作 |
| 换 model | 改 1 行配置，战甲继续工作 |
| 换 API provider | 改 1 行配置 + 1 行 api_key，战甲继续工作 |
| 卸载本地小模型 | 改 1 行配置 `enabled: false`，战甲退化到 v2 状态（不崩） |
| 接入外部大模型 | 改 1 行配置 `advisor.enabled: true`，战甲获得主动交流能力 |

**为什么叫"硬标准"**：这一条不通过，文档就不能算完。

### 2.5 强制读协议（设计原则 R9 · 8:26 钉死）

**问题**：可插拔 ≠ 随便换。如果用户换了个新模型（比如把 Qwen3-4B 换成 Gemma-3-4B），新模型根本不知道 Mark42 是什么 → 它会按自己的理解瞎跑 → 跑偏。

**风险**：和你 6/30 钉死的"AI 跑偏了用户也不知道"是同源问题。可插拔本身是为了方便，但**必须配一个"强制读协议"防止新模型无证上岗**。

**三种"强制"强度**：

| 强度 | 实现方式 | 优点 | 缺点 |
|---|---|---|---|
| **L1 弱强迫 · 启动自检** | Mark42 启动时跑 `consciousness.py`，第一步就是读设计手册。如果模型答不出 Mark42 是什么（自测题），启动失败 | 用户不用记 | 模型可能装样懂 |
| **L2 中强迫 · 协议签名** | Mark42 给小模型的所有 prompt 第一行都是："你必须读过 `Mark42-OS化深化方案-v3.md`，并在响应里输出你的模型 ID + 你对 Mark42 的理解（≤200 字）"。输出不对，战甲拒绝后续调用 | 模型必须真读，否则收不到回复 | 部分模型会忽略 prompt |
| **L3 强强迫 · 验证问答** | 启动时战甲出 5-10 道关于 Mark42 的判断题（比如 "Mark42 的三层问不动边界是什么？"），模型答对 ≥ 8 道才放行 | 真能验证 | 出题 + 判题要写一套 |

**v3 采用 L2 + L3 组合**：L2 是软约束（prompt 引导），L3 是硬约束（答不对不放行）。

**配置开关**：
```yaml
mark42:
  consciousness:
    read_protocol:
      required: true              # 默认 true
      min_correct: 8              # L3 通过分数（默认 8/10）
      questions_pool: 20          # 战甲从题池里随机抽 10 道
      cooldown_hours: 24          # 同模型验证通过后，24h 内不再重考
```

## 3. 战甲自主意识层（v3 核心 · 设计原则 R2/R3/R5）

### 3.1 为什么是"意识"而不是"工具"

| 工具视角 | 意识视角 |
|---|---|
| 战甲检测 → 把信号发给模型 → 模型给回复 → 战甲执行 | 战甲主动观察 → 自己判断 → 自己决定要不要问人 |
| 模型是被调用的函数 | 模型是战甲内部的一个"声音" |
| 响应是被动的 | 行动是主动的（主动发起对话） |

**关键区别**：意识有**意图性**。战甲不是被通知"喂你修一下这个"，是**自己发现**"这个坏了，我自己评估一下，100% 能修我就修，不能我就问人"。

### 3.2 战甲意识的 5 个能力（按确定性原则分层）

| 能力 | 确定性 | 战甲能不能自主决定 | 实现 |
|---|---|---|---|
| **C1. 自检** | 100% | ✅ 完全自主 | 读取 watchdog / heartbeat / status.json |
| **C2. 评估确定性** | 100%（基于规则） | ✅ 完全自主 | 规则表 + 历史 post-mortem 匹配 |
| **C3. 100% 确定的修复** | 100% | ✅ 完全自主 | auto_remediate 引擎执行 |
| **C4. 不确定时主动发起对话** | 0% | ⚠️ 主动发起，但**不自主决定修法** | 调 advisor API / 问用户 |
| **C5. 学过的错误直接走档案** | 高（基于历史） | ✅ 自主（但有上限） | 错误档案查询 + 复用方案 |

### 3.3 战甲主动交流意识 · 4 类对话场景

**场景 A：自检后，不确定是不是真问题**
```
战甲：[C1] heartbeat 25 分钟没动
战甲：[C2 评估] 我不确定 25 分钟算不算 stale。
        历史：99% 情况下 15 分钟就动。但今天 cron 任务多，可能延迟。

战甲：[C4 主动对话] → advisor:
        "我看到 heartbeat 25 分钟没动。阈值是 5 分钟。
         历史上这个值偶尔会到 20 分钟（cron 任务堆积时）。
         要不要：(a) 等 5 分钟再看？(b) 现在就 restart？
         你的判断是什么？"
```

**场景 B：自检后，知道坏了但修法不确定**
```
战甲：[C1] 检测到 L2.5 embed 索引丢失
战甲：[C2 评估] 重建方案我有（命令 A），但我不确定：
        - 重建耗时 180 秒，会不会影响 context-guard loop？
        - 重建完要不要 restart sidecar？

战甲：[C4 主动对话] → advisor:
        "我准备执行 memory-embed-index.py --force。
         历史耗时 180 秒。context-guard 当前每 300 秒跑一次。
         我的方案：执行重建 + 同时暂停 context-guard 240 秒 + 重建后 restart sidecar。
         这个方案安全吗？"
```

**场景 C：新类型异常，第一次见**
```
战甲：[C1] scratch 目录里出现 .tmp-xyz123 未知格式文件
战甲：[C2 评估] 我不知道这是什么。我的规则表里没有。
        这条命中 R5（文档/代码口径不一致 → 问贾维斯）

战甲：[C4 主动对话] → 用户（点点）/ 贾维斯:
        "我看到一种新异常。
         类型：scratch 出现未知 .tmp 文件
         影响：未知
         风险：如果是用户数据，删了会出事
         我不敢动。请确认这是什么？要不要清理？"

战甲：[写错误档案] 不管用户怎么回答，这条事件都要进错误档案。
        下次再遇到类似异常，战甲有据可查。
```

**场景 D：学过的错误，直接走档案**
```
战甲：[C1] 检测到某类型异常
战甲：[C5 查错误档案] 命中 #ERR-007
        档案记录：上次怎么处理、用户怎么回答、结果如何

战甲：[C2 评估] 这次情况和上次 95% 相似。

战甲：[C4 主动对话] → advisor / 用户:
        "这和我 6/15 处理过的 ERR-007 很像（索引文件被截断）。
         上次方案是：停 sidecar → 删索引 → 重建 → 启 sidecar。
         我按上次方案走？还是要先看你一眼？"
```

**场景 D 的进阶**：如果上次方案已经被用户标记为"approved-no-confirm"（用户授权自动复用），战甲**直接执行不问了**。这是"学习"的具体含义。

### 3.4 主动交流协议（用户大模型 API 怎么接）

**核心约束**：
- 用户没开 advisor → 战甲所有不确定都问用户（不静默执行）
- 用户开了 advisor → 战甲按"确定性原则"分流：
  - 100% 确定 → 不问
  - 不能 100% 确定 → 问 advisor
  - advisor 也不确定 → 问用户

**协议形态**（OpenAI Chat Completions 兼容）：
```http
POST {advisor.base_url}/chat/completions
Authorization: Bearer {advisor.api_key}
Content-Type: application/json

{
  "model": "{advisor.model}",
  "messages": [
    {
      "role": "system",
      "content": "你是 Mark42 战甲的外部顾问。战甲会带着上下文问你技术方案..."
    },
    {
      "role": "user",
      "content": "我准备执行 XXX 方案。我的环境是 X、Y、Z。这个方案安全吗？"
    }
  ],
  "response_format": { "type": "json_object" }
}
```

**响应契约**（advisor 必须返回的结构）：
```json
{
  "verdict": "approve" | "reject" | "modify",
  "confidence": 0.95,
  "reasoning": "因为 ... 所以 ...",
  "modified_plan": {           // verdict=modify 时才有
    "steps": ["...", "..."],
    "changes": "把第二步从 X 改成 Y，原因是 ..."
  }
}
```

### 3.5 主动交流的边界（v3 不做的）

| 不做 | 原因 |
|---|---|
| ❌ 战甲不问用户就修改业务代码 | R5 钉死 |
| ❌ 战甲不问用户就清理用户数据 | R5 钉死 |
| ❌ 战甲调用外部大模型 API 改 Mark42 自己的代码 | R5 钉死 |
| ❌ 战甲主动发邮件 / 通知第三方 | 不在故障恢复范畴 |
| ❌ 战甲在不必要时（如一切正常）主动聊天 | 太吵 |

### 3.6 多核心架构 · 反浩克装甲（设计原则 R10 · 8:26 钉死）

### 3.6.1 R10 多核心的上限说明(9:14 联网验证后补 · Tian Pan 2026-04)

> **钉死的硬数据**(联网验证 9:14):
> - **Anthropic 研究**:简单地在工作流中增加 Agent **往往降低性能而不是提升**
> - **失败点数**:N 核产生 N(N-1)/2 个潜在失败点
>   - 4 核 = 6 个失败点
>   - 8 核 = 28 个失败点
>   - 16 核 = 120 个失败点
> - **业界数据**:生产级多 Agent 系统的协调成本**指数级**增长

**为什么 8 核是上限**:
- Mark42 当前 8 核 = 28 个潜在失败点
- 这已经是"能力与复杂度"的**平衡点**
- 加 1 核 = 加 7 个新失败点(从 28 → 36)
- **加核的边际收益 < 边际成本**

**硬规则(钉死)**:
- ❌ **禁止** 8 核 → 9 核,除非先减一个核
- ✅ 加核流程:① 评估能不能合并现有核 ② 合并后释放位 ③ 再加新核
- ✅ 加核前必须做"复杂度守恒"评估:加这个核能减少多少用户打扰?
- ❌ **不** 为"听起来酷" 加核(比如 LLM 调用次数 ≠ 核数)

**反例(避免的事)**:
- "我要让战甲支持更多模型" → 加 1 核 → 失败点 +7 → 维护成本激增
- "我要让战甲支持更多语言" → 加 1 核 → 同上
- 正确做法:**先看现有 8 核里有没有"语言支持" 这件事的位置**(向量化/分类/脱敏都可以做),实在不行才加

**激活条件**:
- 任何"加核" 提案,必须先在 daily 写一段"为什么要加 + 边际收益/成本分析"
- 点点拍板才能加
- 9:14 钉死

### 3.6.2 R13 反浩克装甲的 2 个实现细节（9:14 联网验证 Tian Pan 后补）

> **背景**:9:14 联网验证发现 R13 概念对(和大厂共识一致),但**缺 2 个实现细节**。
> 这 2 个细节是 R13 从"原则" 变成"代码" 的关键。

#### 细节 1 · 工具关键性分类（Critical / Optional / Degradable）

**为什么需要**:不是所有 8 个核心都同等重要。embed 核挂了 → 战甲瞎一眼;但 PII 脱敏核挂了 → 战甲还能用(只是日志里可能出现敏感信息)。

**3 级分类**(参考 Tian Pan "按关键程度分类工具"):

| 级别 | 定义 | 失败时行为 | 例子(Mark42) |
|---|---|---|---|
| **Critical** | 战甲核心能力,挂了用户感知强 | 立即报警 + 自动重启 + 重启失败立刻问用户 | 核心意识(主 LLM)、核心 1 综合推理 |
| **Optional** | 增强能力,挂了不影响主体 | 降级标记 + 静默重启 + 不打扰用户 | PII 脱敏核、异常检测核 |
| **Degradable** | 核心但有 fallback | 切到 L1 关键词召回 / 切到主 LLM 兜底 | embed 核 → L1 关键词;分类核 → 关键词分类 |

**8 核初步分类**(v3.3 草案,等 R13 实施时细化):

| 核心 | 当前状态 | 关键性 | 失败时 fallback |
|---|---|---|---|
| 核心 1 · 核心意识(主 LLM) | ✅ | Critical | 报警 + 问用户切到备用 LLM |
| 核心 2 · 战甲自主意识(小模型) | ⏳ | Degradable | 切到主 LLM 兜底 |
| 核心 3 · 记忆向量引擎 | ✅ | Degradable | 切到 L1 关键词召回 |
| 核心 4 · 文本压缩 | ⏳ | Optional | 降级为截断 + 简单摘要 |
| 核心 5 · 代码理解 | ⏳ | Optional | 跳过代码分析,主 LLM 兜底 |
| 核心 6 · 日志分类 | ⏳ | Optional | 降级为按来源分类 |
| 核心 7 · PII 脱敏 | ⏳ | Optional | 降级为正则脱敏 |
| 核心 8 · 异常检测 | ⏳ | Optional | 降级为简单阈值 |

#### 细节 2 · 降级响应契约（FAILURE.md 规范）

**为什么需要**:战甲降级时,需要给用户/上游传**结构化信息**,而不仅是"我挂了" 一个字符串。

**FAILURE.md 规范**(参考 Tian Pan 2026-04):

```json
{
  "status": "degraded",
  "core_id": "embed-sidecar",
  "core_name": "记忆向量引擎",
  "criticality": "degradable",
  "missing_capabilities": [
    "L2.5 语义召回",
    "L2.5 跨文档关联"
  ],
  "fallback_active": "L1 关键词召回",
  "auto_recovery_in_progress": true,
  "estimated_recovery": "30 秒",
  "user_action_required": false,
  "reason": "embed-sidecar process died (signal 9)"
}
```

**契约字段说明**:

| 字段 | 类型 | 用途 |
|---|---|---|
| `status` | enum: ok / degraded / failed / unknown | 上层第一眼看到的健康度 |
| `core_id` | string | 哪个核挂了 |
| `criticality` | enum: critical / optional / degradable | 上层决策(报警?静默?) |
| `missing_capabilities` | string[] | **结构化列出来**挂了影响什么 |
| `fallback_active` | string | 当前在用什么兜底 |
| `auto_recovery_in_progress` | bool | 战甲在自动恢复吗 |
| `estimated_recovery` | string | 预计多久恢复 |
| `user_action_required` | bool | 用户需要做什么 |
| `reason` | string | 为什么挂(技术原因) |

**为什么这 9 个字段是必须的**:
- `status` + `criticality` → 上层 LLM 决定"要不要问用户"
- `missing_capabilities` → 用户**看得见丢了什么**,不是"战甲挂了" 一句话
- `fallback_active` → 用户**看得见在用什么顶**,心里有底
- `auto_recovery_in_progress` + `estimated_recovery` → 用户决定"等还是动手"
- `user_action_required` → **避免误报**(optional 失败时不打扰用户)
- `reason` → 出事后看 log 一眼就懂

**反例(避免的事)**:
```json
// ❌ 错:一个 string 打发
"error": "embed-sidecar is down"
// 战甲上层看到只能"猜",不知道 criticality、不知道 fallback、不知道要不要问用户
```

**激活条件**:
- R13 实施时,所有 8 核必须按这个契约写
- 战甲上层(主 LLM / 用户感知层)按这个契约决定行为
- v4 用户感知层 HUD 直接消费这个契约的字段

### 3.6.3 R14 集群思维 + 修复边界（9:25 点点提出 · 趁热打铁钉死）

> **为什么必须有这一节**:9:25 点点指出 R10-U + R13 看似矛盾(R10-U 钉死复杂度,8 核上限;R13 钉死降级,核心可挂)。**解法是"集群思维"**——让复杂度在物理上自生自灭。这一节是把"脑子里" 的设计正式钉到文档,避免两天后忘光。

#### 4 个核心原则(为什么这样设计)

| 原则 | 内容 | 业界对应 |
|---|---|---|
| **集群化** | 8 核 = 8 个独立小集群(配置 + 二进制 + 启动脚本 = 一体) | K8s Pod / Service Mesh sidecar |
| **复杂度收敛** | 集群内可复杂,集群间必须简单(契约 9 字段) | Istio / 微软 Dapr |
| **坏了就换** | 集群挂 → 整体替换,不修复 | Netflix cattle vs pets(2011 伯尔曼提出) |
| **修复有边界** | 集群内自动修,集群挂人参与,战甲自身坏了 → 人修 | 承认"无限递归问题"无完美解,划线 |

#### 8 核 → 8 集群 命名规范

| 旧名 | 新集群名 | 集群内组件(初稿) | 关键性(R13-D) |
|---|---|---|---|
| 核心 1 · 核心意识(主 LLM) | `cluster-consciousness` | 主 LLM 客户端 + 配置 + 契约校验器 | Critical |
| 核心 2 · 战甲自主意识(小模型) | `cluster-auto-consciousness` | Qwen3-4B + 路由 + 配置 | Degradable |
| 核心 3 · 记忆向量引擎 | `cluster-memory-vector` | embed-sidecar + 索引 + 配置 | Degradable |
| 核心 4 · 文本压缩 | `cluster-text-compress` | T5-base + 启发式 + 配置 | Optional |
| 核心 5 · 代码理解 | `cluster-code-understand` | Qwen2.5-Coder + 配置 | Optional |
| 核心 6 · 日志分类 | `cluster-log-classify` | BGE-M3 + 关键词 + 配置 | Optional |
| 核心 7 · PII 脱敏 | `cluster-pii-redact` | Presidio + 正则 + 配置 | Optional |
| 核心 8 · 异常检测 | `cluster-anomaly-detect` | Isolation Forest + 阈值 + 配置 | Optional |

#### "坏了就换" 的 6 个工程关键词

1. **集群配置目录化** —— 每个集群 `clusters/<name>/` 目录(配置 + 二进制 + 启动脚本)
2. **契约 schema 一份** —— `schema/failure-contract.yaml`,所有集群共享(R13-D 9 字段)
3. **集群打包** —— `clusters/<name>.tar.gz` = 配置 + 二进制 + 启动脚本 = 一体
4. **集群健康度 3 指标** —— ① 进程在 ② 端口通 ③ 契约校验通过
5. **替换流程命令** —— `mark42 cluster replace <name> [--from-backup|--from-git-tag]` 一键换
6. **替换审计** —— 每次替换写 `actions.jsonl`(R7 自动行为必有据可查)

#### "换" 的 2 个来源(初稿)

| 来源 | 何时用 | 风险 |
|---|---|---|
| **代码备份**(`.bak-yyyymmdd-hhmmss`) | 集群配置坏(改坏了) | 低,纯回滚 |
| **Git tag**(集群上次稳定版本) | 集群代码坏(bug) | 中,需 git 配合 |

#### 修复边界(钉死)

| 层级 | 修法 | 谁修 |
|---|---|---|
| **集群内组件挂** | 自动重启 3 次,3 次失败 → 整体替换集群 | 战甲 C3 + C5 能力 |
| **集群挂 3 次** | 主动对话问用户"要不要切回上一版 / 切到 L1 兜底" | 战甲 C4 主动对话 |
| **战甲自身(systemd)挂** | **不自动化,人修** | 点点 + 贾维斯(避免无限递归) |

#### R14 与已有原则的关系

```
R10-U 钉死 "8 核上限"  →  防止复杂度爆炸
R13   钉死 "降级而非崩溃"  →  防止单点失败
R13-D 钉死 "契约 9 字段"  →  防止复杂度外泄
R14   钉死 "集群思维 + 修复边界"  →  防止无限递归  ← 9:25 点点补最后一环
```

**这 4 条加起来,Mark42 v3.4 在设计层面"接近成熟"** —— 不是"完美",是"体系成立"(像 K8s 1.0 那种感觉,有缺陷但跑得通)。

#### 业界对应(联网验证 9:25)

| 哲学 | 大厂实践 | Mark42 R14 对应 |
|---|---|---|
| **Cattle vs Pets**(伯尔曼 2011) | Netflix / Amazon 整机替换 | "集群挂 → 整体换" |
| **Service Mesh** | Istio / Linkerd / 微软 Dapr | "8 核 = 8 个独立小集群" |
| **Immutable Infrastructure** | Netflix Spinnaker / K8s Deployment | "配置目录化 + 集群打包" |
| **承认无限递归** | 所有分布式系统 | "战甲自身坏了 → 人修" |

**R14 不是新概念,是已经赢了的概念;但 Mark42 单机场景下,这种思路不常见**——大多数单机工具还在用"Pets" 哲学(修修补补)。R14 让 Mark42 在**单机场景下也有"集群级" 抗复杂度能力**。

#### 激活条件

- R14 当前是**设计原则**,不强制实施
- v3.5 阶段首次落地 1 个集群(`cluster-memory-vector`,因为已经有真实代码),跑通"换" 流程
- 跑通后再铺开其他 7 集群
- 9:25 钉死

#### 反例(避免的事)

- ❌ "我修一下这个集群" → **不修,换**。修的成本 > 换的成本
- ❌ "集群挂 10 次还在重启" → 超过 3 次主动问人,**避免战甲进入死循环**
- ❌ "战甲自身挂了让战甲自己修" → **绝对不**,这层永远人修
- ❌ "加 1 个新集群" → **禁止**,R10-U 钉死 8 核上限,加必须先减

## 4. 错误档案系统（v3 核心 · 设计原则 R5 落地）

### 4.1 为什么需要"档案"而不是"日志"

| 日志 | 档案 |
|---|---|
| 一次性记录事件 | 累积式学习（同类事件能聚类） |
| 文本流 | 结构化条目（可查询可复用） |
| 给运维看的 | 给战甲意识层用的（直接喂回决策） |
| 看完就丢 | 战甲查档案"上次怎么处理的" |

**档案 ≠ 日志**。日志是事件流，档案是**带反馈的学习条目**。

### 4.3 错误档案的"学习"机制（v3 的核心创新）

**学习的三层含义**：

| 学习层级 | 含义 | 实现 |
|---|---|---|
| **L1 · 记录** | 战甲记得见过这种异常 | 写入 entries.jsonl |
| **L2 · 复用** | 战甲下次遇到同类异常，会查档案、按上次方案走 | 战甲意识层 C5 能力 |
| **L3 · 自动批准** | 用户授权后，战甲不再问，直接执行上次方案 | `auto_approval` 字段 |

- ✅ 可以用于"清理已知垃圾"（如已确认的 tmp 目录）
- ✅ 可以用于"已知类型故障的修复"

#### L3 自动批准的防护机制（设计原则 R12 · 8:37 钉死）

**动机**：L3 是 v3 的核心创新但也是最大风险。如果不加护栏，可能出现"破坏性学习"——模型学到一个错误方案，反复自动执行，把系统越修越坏。

**两条铁护栏**：

| 护栏 | 规则 | 实现 |
|---|---|---|
| **Cooldown 计数** | 同一个错误档案 ID 连续自动批准 ≤ 5 次 | 第 6 次自动批准时强制用户重新确认（不能跳过） |
| **影响面硬黑名单** | 以下四类方案 **永远不能 L3 自动批准**，即使档案里说可以 | 战甲在 `auto_approval` 写入时校验，命中即拒绝 |

**L3 影响面硬黑名单**（永久）：

| 类别 | 为什么永久禁 | 例子 |
|---|---|---|
| 修改用户数据 | R5 钉死 | 删 `.md` 文件 / 清空 scratch 用户文件 |
| 修改业务逻辑 | R5 钉死 | 改 `mark42.py` 业务参数 / 改 `model.yaml` 默认模型 |
| 修改 systemd service / 守护进程配置 | 6/30 "AI 跑偏了用户也不知道"原则 | 改 `mark42-*.service` 文件 / restart 守护进程 |
| 删除任何目录（哪怕是 tmp） | 影响面不可逆 | `rm -rf` 任何形式 |

**配置文件**（`mark42.yaml`）：

```yaml
mark42:
  error_archive:
    l3_protection:
      cooldown_max: 5            # 连续自动批准上限
      cooldown_window_days: 30   # 30 天内的连续计数
      hard_blacklist_categories:
        - user_data_modification
        - business_logic_modification
        - systemd_service_modification
        - directory_deletion
```

**用户可见的提示示例**：

```
$ mark42 archive approve ERR-2026-07-14-001 --scope similar_match
⚠️ 已连续自动批准 5 次（同 ERR-001，30 天内）。下次类似异常将强制要求重新确认。
✅ 已授权

$ mark42 archive approve ERR-2026-07-14-005 --scope similar_match
❌ 拒绝：ERR-005 涉及 "scratch 文件清理"，命中 hard_blacklist (directory_deletion)。
   即使用户授权 L3 自动批准，战甲也不会自动执行。请改成 manual_only 模式。
```

**为什么这是 v3.1 的"补丁"而不是"新增"**：因为 v3 已经有 L3 学习机制，R12 是给这个机制加护栏，不破坏原设计。

### 4.4 错误档案的 4 种状态机

```
                ┌──────────┐
                │  NEW     │ ← 第一次见
                └────┬─────┘
                     │ 战甲写档案 + 问用户/advisor

                   ▼

              └────┬─────────────┘

                   ▼
              ┌──────────────┐
              │  REJECTED     │ ← 用户说"以后别这样"
              └──────────────┘
```

### 4.5 错误档案与"战甲意识层"的协作流程

```
故障信号
   │
   ▼
C1. 自检（读取 watchdog / status.json）
   │
   ▼
C2. 评估确定性（规则表 + 错误档案匹配）
   │
   ├── 命中错误档案 + auto_approved = true → 直接执行（C3）
   │
   ├── 命中错误档案 + auto_approved = false → 按上次方案问一遍 → 执行
   │
   ├── 未命中但 100% 确定（规则表内） → 直接执行（C3）
   │
   ├── 未命中且不能 100% 确定 → C4 主动对话（advisor / user）
   │       │
   │       ├── advisor 给 approve → 执行
   │       ├── advisor 给 modify → 按修改执行
   │       └── advisor 给 reject / 超时 → 问用户
   │
   └── 完全没见过的新类型 → C4 问用户 → 写错误档案（NEW 状态）
```

```json
      "total_failures": 1
    }
  },
  "updated_at": "2026-07-14T08:14:23+08:00"
}
```

**核心位状态机**：
```
UNKNOWN（启动未探） → healthy（探活成功） 
                       → degraded（探活失败 1 次，但有 fallback）
                       → down（探活失败 ≥3 次 / 无 fallback）
                       → quarantined（用户手动隔离）
```

**`mark42 cores` 命令**：
```
$ mark42 cores list

  --------------------------------+-----------------------+-----------------------------+-------------+--------------
  core_1_main_consciousness       | main_consciousness    | minimax/MiniMax-M3          | healthy     | 2 分钟前
  core_2_armor_consciousness      | armor_consciousness   | qwen3:4b                    | healthy     | 启动未使用
  core_3_memory_vector_engine     | memory_vector         | MiniLM-L12-v2               | healthy     | 28 分钟前
  core_4_text_compressor          | text_compression      | not_loaded                  | down        | -

$ mark42 cores verify core_2_armor_consciousness

  随机抽 10 题（从 20 题题池里）
  Q001 ✅ Q002 ✅ Q003 ❌ Q004 ✅ Q005 ✅
  Q006 ✅ Q007 ✅ Q008 ✅ Q009 ✅ Q010 ✅
得分：9/10 ✅ 通过

✅ 已隔离 core_2。战甲将退到 v2 行为，直到用户恢复。
```

**为什么状态机要分 `degraded` 和 `down`**：
- `degraded` = 还能用，但有 fallback 顶上。比如向量引擎挂了，退回 BM25 关键词搜索，**能搜但不准**。
- `down` = 完全废了，没有 fallback，战甲必须有提示（不能假装没事）。

## 9. v3 阶段结论

**v3 把 Mark42 从"能自己修"升级到"能自己修 + 能主动问 + 能学习"。**

核心新增三件套：

| 组件 | 解决什么 | 一句话 |
|---|---|---|
| **可插拔模型层** | 锁死风险 + 商品化阻塞 | 用户想换什么换什么，战甲照样工作 |
| **战甲意识层** | "看得见但不敢动"的死板 | 战甲有判断、有意图、能主动问 |
| **错误档案** | 每次新异常都从头问的重复劳动 | 学过的错误直接走档案，越用越聪明 |

**v3 守住的边界**：
- R5 三层问不动的铁律（文档口径 / 新类型 / 业务修改）
- v2 全部既有边界（自动修不碰用户数据 / 不跨机器联动 / N 圈上限）

**v3 落地的不变前提**：
- 稳大于强（v1 起的总原则）
- 所有自动行为必有据可查
- 所有自动修必须先 dry-run
- 可插拔设计（runtime / model / api 三层都可换）
