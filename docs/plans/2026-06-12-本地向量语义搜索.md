# 本地向量语义搜索 — 设计方案

**日期**: 2026-06-12
**状态**: 待确认

---

## 背景

当前记忆搜索架构是 4 层（L1 grep → L2 BM25 → L3 云端 → L4 备份），缺少一个关键的本地语义搜索层。BM25 是关键词匹配，搜「点点难过」找不到「点点哭了」/「点点眼泪」。

## 目标

在现有 L2（BM25）和 L3（云端）之间加一层 **L2.5 本地向量语义搜索**，使用 `paraphrase-multilingual-MiniLM-L12-v2` 对所有记忆文件做 embedding，实现中文友好的语义搜索。

---

## 方案对比

### 方案 A：离线索引 + 磁盘持久化（推荐）

**怎么做**：
1. 安装 `sentence-transformers` + `torch`（用 uv 建独立 venv，放 /mnt/data）
2. 写一个索引脚本：遍历 `memory/` 所有 .md 文件 → 按段落切 → 生成 384 维向量 → 存为 NumPy `.npy` + 文本映射 `.json`
3. 写一个搜索脚本：输入 query → 生成向量 → 余弦相似度排序 → top-K 结果
4. 接入现有路由：`memory-search-router.py` 加 L2.5 层调用
5. 索引重建策略：每次对话结束后如果 memory 文件有变化就重建（增几十 KB 很快）；或定时每小时重建

**优点**：
- 搜索速度极快（毫秒级，纯 NumPy 向量运算）
- 不引入额外数据库依赖（chromadb/faiss/milvus 等）
- 文件体积 58 个 memory 文件 × 每个 10-30 段落 → 索引约 500KB-2MB
- 模型只加载一次，推理时复用

**缺点**：
- 需要重建索引（但数据量小，几秒）
- 纯 CPU 推理首次加载 3-8 秒

### 方案 B：轻量向量数据库（chromadb/faiss）

**怎么做**：
安装 chromadb 或 faiss，把 embedding 存进向量 DB，搜的时候用内置索引。

**优点**：
- 不需要自己写索引逻辑
- chromadb 支持增量更新

**缺点**：
- 引入额外依赖（chroma 那堆包几百 MB）
- 功能过剩——才不到 2000 段落的数据量，纯 NumPy 完全够
- 增加了维护复杂度

### 方案 C：直接用云端 memory_search（现状）

不引入本地模型，继续依赖 OpenClaw 自带的云端语义搜索（L3）。

**优点**：零安装
**缺点**：依赖网络、慢（4-10 秒）、有 API 额度限制

---

## 推荐方案：A

**理由**：
- 你的 memory/ 只有 2.2MB、58 个文件，段落级索引最多 1-2MB，纯 NumPy 够用
- 公司 Linux 7.7GB 内存 + mnt/data 32GB 空间充裕
- 不引入新数据库，维护成本低
- 可以和现有 BM25 搜索融合（语义 + 关键词双通道结果合并）
