## 一、这一阶段做了什么

Phase 1.5 让 Mark42 多Agent 有了：

- 聚合状态（status --all-agents）
- 结构化协调建议（recommendations）
- 趋势回看（broker snapshot + recommendationTrend）
- dry-run 可执行建议（suggestedActions）

Phase 2 在这层之上，加了一个最小但安全的“确认后执行”骨架：

**`mark42.py actions` 子命令 + actions_runner 模块**

---

## 二、核心设计

### 1. 默认永远 dry-run

不带 `--yes` 时：

- 命令只打印预览
- **不会调用任何会改状态/重启进程的真实接口**
- 同样会写 broker 留痕（`actions.previewed`）

### 2. 必须 `--yes` 才真执行

`--yes` 是显式确认：

- 缺省时一切不可执行
- 即便加了 `--yes`，未在白名单的动作仍被安全策略拒绝
- 执行后会写 broker 留痕（`actions.executed`）

### 3. 动作白名单 / dry-run-only 分级

当前阶段支持的 action id 分两类：

**可执行白名单：**
- `restart-assemble`
  - 加 `--yes` 后才会真调 `assemble_restart(agent=...)`

**dry-run-only（当前阶段不执行）：**
- `rebalance-default-agent`
- `rebalance-heavy-tasks`
- 加 `--yes` 也仍然只返回"refusedBySafetyPolicy=true"的 dry-run 视图
- 这是故意保守，避免这一阶段就把“自动调度”搬上生产

## 三、命令形态

### 列出当前可执行动作
```
mark42.py actions --list
```
输出 JSON：
```json
{
  "actions": [
    {
      "actionId": "restart-assemble",
      "title": "重启 Mark42 assemble 守护",
      "reason": "assemble 全部停止",
      "commandPreview": "mark42.py assemble",
      "priority": "high",
      "executableNow": true,
      "dryRunOnly": false,
      "agent": null
    }
  ]
}
```

### 预览某个动作（默认行为）
```
mark42.py actions --run restart-assemble --agent main
```
输出：
```json
{
  "actionId": "restart-assemble",
  "executed": false,
  "level": "info",
  "summary": "DRY-RUN: 将重启 assemble (agent=main)",
  "commandPreview": "mark42.py assemble (agent=main)",
  "agent": "main",
  "metadata": {}
}
```

### 真执行某个动作
```
mark42.py actions --run restart-assemble --agent main --yes
```
输出：
```json
{
  "actionId": "restart-assemble",
  "executed": true,
  "level": "info",
  "summary": "已重启 assemble: pid=<N>",
  "commandPreview": "mark42.py assemble (agent=main)",
  "agent": "main",
  "metadata": { "pid": <N>, "log": "..." }
}
```

## 四、与 Phase 1.5 的衔接

### pending 桥接
- `mark42.py status --all-agents` 每次生成 aggregate 后，会把当前 `suggestedActions` 桥接到：
  - `~/.local/state/openclaw/mark42/actions-pending.json`
- 这让 `actions --list` 能看到“刚刚建议过什么”

### broker 留痕
- 每次“预览”或“执行”都通过 `_append_broker(...)` 写一条事件
- 事件类型：
  - `actions.previewed`
  - `actions.executed`

这样 Phase 1.5 的“历史留痕体系”被无缝继承，不会因为半自动骨架的引入丢追溯。

## 七、明确不做的事

为了满足“写下来 > 记着”与“自动行为有据可查”，这一阶段明确不做：

### 1. 不调度多 agent 任务
- 只是“半自动执行”入口
- 不做 agent 间任务迁移
- 不做 loop / heavy 自动重新分配

### 2. 不静默执行任何东西
- 没有 `--auto` / `--quiet-execute`
- 即便加 `--yes`，不在白名单内的仍被拦下

### 3. 不写复杂的策略引擎
- 没有“如果 X 则自动 Y”之类联动
- 推荐 ↔ 执行 是两件事，决策仍由人判断

## 八、下一步建议

如果继续 Phase 2 的下一阶段，最自然的两个方向：

1. **给已白名单动作加更多样例**
   - 例如“清理某 agent 的过期 heavy 任务”
   - 但仍走 dry-run → --yes 这个流程

2. **加建议动作品质统计**
   - 例如一段时间内：
     - 多少条被预览
     - 多少条被实际执行
     - 执行失败的分布
   - 这样可以发现：
     - 被忽略的建议（也许规则不实用）
     - 经常失败的动作（也许设计缺陷）

3. **Phase 3 候选**
   - 仅对真正安全的低风险动作（如清理 scratch）打开一次性“推荐即执行”
   - 但需要前置：
     - 用户级白名单
     - 可关闭的总开关
