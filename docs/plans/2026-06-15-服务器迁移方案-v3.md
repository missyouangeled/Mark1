## 逐项审查

### 问题 1：双 Caddy 嵌套 → 跳数冗余

**Mark1 现状**：
```
Internet → unified-proxy (Caddy :18788) → Gateway/Infos-handle
```

**v2 设计**：
```
Internet → Cloudflare Tunnel → 外层 Caddy (:443) → unified-proxy (Caddy :18788) → Gateway
```

**问题**：两层 Caddy 背靠背，增加一跳延迟，配置分散在两处，排查故障时要在两个 Caddy 日志之间跳。

**修正**：把外层 Caddy 和 unified-proxy 的 Caddy 合并为一个。unified-proxy 不再独立监听 :18788，改为由唯一的 Caddy 实例同时承担「公网入口路由」和「内部 /v1/ 路由分发」。

修正后架构：
```
Internet → Cloudflare Tunnel → Caddy (:443)
  ├─ jarvis.xxx.com  → Gateway :18789       (Control UI / WebChat)
  ├─ /v1/query/*     → infos-handle :18790  (健康数据 API)
  ├─ /v1/handle/*    → infos-handle :18790
  ├─ /healthz        → infos-handle :18790
  ├─ code.xxx.com    → code-server :8080
  ├─ port.xxx.com    → Portainer :9443
  └─ svc.xxx.com     → Docker 服务
```

**影响**：unified-proxy.service 可以退役。新增一个 `jarvis-gateway-proxy.service`（替代 unified-proxy），Caddyfile 集中管理所有路由。

### 问题 2：鉴权链断裂

**Mark1 现状**：
```
infos-handle sidecar 鉴权规则：
  localhost 请求 → 免鉴权
  远程/LAN 请求 → 需 Bearer token
```
判断依据是 `X-Forwarded-For` / `X-Real-IP`。

**v2 问题**：外层 Caddy 反代到 unified-proxy 时写了 X-Real-IP，但 unified-proxy 再反代到 infos-handle 时会不会覆盖？如果两层 Caddy 各自写一次 X-Forwarded-For，可能导致 IP 列表不正确。

**修正**：合并 Caddy（问题 1 的修正）后，只需一层代理，X-Forwarded-For 只写一次，不会出错。Caddyfile 中显式配置：
```
header_down X-Real-IP {remote_host}
```
（`header_down` 是发给后端的头，即 Caddy → infos-handle 方向）

### 问题 3：WebSocket 代理链

**Mark1 核心依赖**：Control UI 通过 WebSocket 与 Gateway 实时通信。当前链路：
```
浏览器 ←WebSocket→ unified-proxy (:18788) ←WebSocket→ Gateway (:18789)
```

**v2 新增**：Cloudflare Tunnel + 外层 Caddy。每增加一层，WebSocket 握手都可能断开。

**验证结果**：
- Cloudflare Tunnel 默认支持 WebSocket（无需配置）
- Caddy 默认支持 WebSocket（自动检测 Upgrade 头）
- 两层叠加：已验证可行 ✅

**修正**：在 Caddyfile 中显式声明 WebSocket 支持（防止未来默认行为变化）：
```
@ws {
    header Connection *Upgrade*
    header Upgrade websocket
}
reverse_proxy @ws localhost:18789
```

### 问题 4：code-server 无认证 → 安全漏洞

**风险等级**：🔴 严重

**v2 遗漏**：code-server 如果部署到公网（通过 CF Tunnel），但没有配置密码认证，任何知道域名的人都能在你的服务器上开一个 VS Code 终端——等同于拿到服务器 shell 权限。

**修正**：三步锁定 code-server：
```bash
# 1. 设置强密码
echo "your-strong-password" > ~/.config/code-server/config.yaml

# 2. 或更好：用 Cloudflare Access 做 SSO 登录
#    （在 CF Dashboard 中为 code.xxx.com 添加 Access 策略）
#    这样访问 code.xxx.com 前必须先通过 CF 身份验证

# 3. code-server 只监听 127.0.0.1，完全依靠 Caddy 反代
code-server --bind-addr 127.0.0.1:8080
```

**推荐**：Cloudflare Access（免费，支持邮箱验证码/GitHub/Google 登录），比密码更安全，而且不用记密码。

### 问题 5：Docker 服务监控盲区

**Mark1 的 watcher 覆盖范围**：
- ✅ Gateway 进程
- ✅ infos-handle sidecar
- ✅ embed-sidecar
- ✅ session/traffic/transcript 异常
- ❌ Docker 容器（新增）

**问题**：如果 Nextcloud 的 MySQL 挂了、微信桥接崩了，Mark1 的 6 个 watcher 都不会告警。

**修正**：在 `health-collector` 中新增一层 Docker 健康检查：
```
每 5 分钟（与完整层一起跑）：
  docker ps --format '{{.Names}} {{.Status}}' 
  → 检查是否有 unhealthy / exited 容器
  → 写入 broker 事件 → Control UI 仪表板新增「Docker 服务」卡片
```

不需要另起一套监控（如 Uptime Kuma），在 Mark1 现有 broker 管道里加一层即可。Uptime Kuma 作为**可选**的人类可视面板，不与 Mark1 自动化管道耦合。

### 问题 6：告警体系割裂

延展问题 5。如果 Docker 服务挂了，有两种通知路径：
- Mark1 broker → Control UI 仪表板（需要你打开 Control UI 才看到）
- 主动推送（目前 Mark1 的 watcher 状态变化会通过 frontstage broker 注入 Control UI）

**修正**：Docker 异常事件统一走 Mark1 的 `frontstage-broker` → `chat.inject` 管道。你在 Control UI 聊天界面里就能看到「⚠️ nextcloud 容器已停止，尝试重启中...」。

### 问题 7：微信方案过于乐观 → 需要现实路线

**技术现实**：微信没有官方 Linux 客户端。所有开源方案都是在 Windows 上 Hook 微信进程。

| 你能做什么 | 怎么实现 | 安全度 |
|-----------|---------|--------|
| **聊天记录备份 + 多端查阅** | WeChatMsg 解析 Windows 上的微信数据库 → 导出到服务器 → 通过 Web 界面浏览 | 🟢 安全（只读） |
| **微信消息桥接到服务器** | 在 Windows 机器（掌机/公司PC）上跑 WeChatFerry → 通过 Tailscale 把消息推送到服务器 → 服务器转发到其他设备 | 🟡 中等（Hook 进程） |
| **在服务器上直接登录微信** | ❌ 不可行（微信无 Linux 客户端，Wine 模拟极不稳定且封号风险极高） | 🔴 不推荐 |

**修正后的推荐路线**：
1. **阶段 1（零风险）**：部署 WeChatMsg，定期从 Windows 电脑导出聊天记录到服务器。服务器提供 Web 界面浏览历史。
2. **阶段 2（需要时）**：在掌机（Windows）上跑 WeChatFerry，消息通过 Tailscale 推送到服务器。服务器作为消息中枢，你从任何设备问贾维斯「最近微信有什么消息」就能看到。
3. **不做**：在 Linux 服务器上原生跑微信。

### 问题 8：备份缺口

**Mark1 备份范围**：
- ✅ session-backup（每 30min）
- ✅ workspace（Git push）

**v2 新增但未覆盖**：
- ❌ Docker volumes（数据库：Postgres/MySQL 数据文件）
- ❌ /srv/projects/（开发代码）
- ❌ 自部署服务的配置和数据

**修正**：
```
# 新增备份策略
/mnt/data/backups/
  ├── docker-volumes/   # 每日 tar 备份 Docker volumes
  ├── projects/         # Git 自动推送（每个项目独立 repo）
  └── configs/          # Caddyfile / docker-compose.yml / env files

# 在 lifecycle-maintainer 中新增：
#   - 每日凌晨 3 点 tar 备份所有 Docker volumes
#   - 保留最近 7 天
#   - 超过 7 天的自动删除
```

> 📌 Mark1 回收机制现状：lifecycle-maintainer（每5min）+ session-size-watcher（事件+每10min）
> + cleanup-temp / cleanup-old-audio / scratch-cleanup / flush-memory-sync 等脚本。
> **迁移到 Mark2 后，全部升级为回收设计 v2.0**（独立文档 `docs/贾维斯中枢-Mark2-回收机制设计.md`）。

### 问题 9：工作区边界模糊

**Mark1 workspace 路径**：`~/.openclaw/workspace/`
- 这是一个 Git mono-repo（test-git.git）
- 存放：SOUL.md / MEMORY.md / scripts/ / docs/ / skills/ / memory/

**v2 开发项目路径**：`/srv/projects/`
- 这应该是多个独立 Git 仓库

**边界规则**：
```
~/.openclaw/workspace/  → Git repo: test-git（贾维斯自身）
    贾维斯的配置、记忆、脚本、文档
    不存放用户开发项目代码

/srv/projects/          → 每个项目独立 git init / clone
    网页、小程序、API、图片处理脚本
    贾维斯通过 exec 工具帮你管理这些项目
```

**修正**：在方案中明确这一边界，避免未来把开发项目代码混进 workspace repo。

### 问题 10：Python venv 容器化冲突

**Mark1 依赖**：
- `~/.local/share/openclaw-voice-venv311` → infos-handle sidecar、ChatTTS、lifecycle-maintainer
- `~/.local/share/openclaw-embed-venv311` → embed-sidecar、lifecycle-maintainer

**v2 提了「可以容器化」但没说怎么过渡。**

**修正**：迁移第一阶段**不容器化 Python 服务**。保持和 Mark1 一模一样的 venv + systemd 方式。容器化是未来优化项，不是迁移前提。原因：
1. Python venv 已经在 Mark1 上跑了几个月，稳定
2. 容器化需要额外测试 GPU/CPU 推理、音频设备访问等
3. 先迁移后优化，不要在一次迁移中改变太多变量

### 问题 11：Tailscale 与 Cloudflare Tunnel 访问路径

**两种访问方式**：
- 公网：`https://jarvis.xxx.com` → Cloudflare Tunnel → Caddy → Gateway
- 内网：`http://100.x.x.1:18789` → 直连 Gateway（经 Tailscale）

**问题**：用哪个？延迟不同、鉴权不同。

**修正规则**：
```
日常使用（你在外面/手机上）：
  → jarvis.xxx.com（Cloudflare Tunnel）
  → 有 SSL，有 CF Access 保护

你在家/公司，且设备装了 Tailscale：
  → 也建议用 jarvis.xxx.com
  → 不要直接连 Tailscale IP 裸端口
  → 统一入口，避免出现"为什么 A 路径能用 B 路径不行"的诡异 bug

远程管理（SSH/文件传输）：
  → 用 Tailscale 私网 IP
  → ssh 100.x.x.2
  → scp/rsync
```

### 问题 12：SSL 证书责任不清

| 网络方案 | 谁管 SSL |
|----------|----------|
| Cloudflare Tunnel | CF 自动签发边缘证书，后端走 HTTP 即可 |
| 公网 IP + Caddy | Caddy 自动 Let's Encrypt |

**修正**：如果选 Cloudflare Tunnel，后端 Caddy 不需要配 SSL，Caddyfile 写 `http://` 即可。如果未来切换到公网 IP，再开 Caddy 的自动 HTTPS。

当前 Caddyfile 建议：
```
# Cloudflare Tunnel 方案（推荐）
jarvis.yourdomain.com {
    reverse_proxy localhost:18789
}
```
Caddy 会自动检测 CF Tunnel 的协议并适配。

### 问题 14：Caddy → Portainer 反代协议

**v2 写的是**：`reverse_proxy localhost:9443`

**问题**：Portainer 默认 9443 是 HTTPS。Caddy 的 `reverse_proxy` 默认用 HTTP 去连后端。需要显式声明：
```
port.yourdomain.com {
    reverse_proxy https://localhost:9443 {
        transport http {
            tls_insecure_skip_verify
        }
    }
}
```

**或者更简单**：Portainer 也配成 HTTP 模式（`--http-enabled`），然后让外层 Caddy 处理 HTTPS。这样后端全走 HTTP，Caddy 统一管 SSL。

### 步骤 10: 🛡️ 安全加固

```bash
# 10.1 ufw 防火墙
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow in on tailscale0
sudo ufw enable

# 10.2 SSH 加固
sudo tee /etc/ssh/sshd_config.d/99-hardening.conf << 'EOF'
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers jarvis
MaxAuthTries 3
EOF
sudo systemctl restart sshd

# 10.3 fail2ban（防 SSH 爆破）
sudo apt install -y fail2ban
sudo tee /etc/fail2ban/jail.local << 'EOF'
[sshd]
enabled = true
maxretry = 3
bantime = 3600
findtime = 600
EOF
sudo systemctl enable --now fail2ban

# 10.4 auditd（监控关键文件变更）
sudo apt install -y auditd
sudo auditctl -w /etc/caddy/Caddyfile -p wa -k caddy_config
sudo auditctl -w ~/.ssh/authorized_keys -p wa -k ssh_keys
sudo auditctl -w ~/.openclaw/workspace/credentials -p wa -k credentials
```

## Mark1 安全对照

| Mark1 现有 | 服务器需新增 | 理由 |
|-----------|-------------|------|
| infos-handle localhost 免鉴权 | 不变 | 只监听 127.0.0.1 |
| Gateway 绑 0.0.0.0:18789 | 改为 127.0.0.1 | Caddy 反代即可 |
| 无防火墙 | 新增 ufw | 服务器有公网流量 |
| 无 fail2ban | 新增 fail2ban | 防暴力破解 |
| SSH 密码登录 | 禁密码仅密钥 | 安全基本要求 |
| 无容器扫描 | 新增 trivy 周检 | Docker 多了就有风险 |
| 补丁 systemd 自检 | 扩展安全项 | 不要等出事了才发现 |
