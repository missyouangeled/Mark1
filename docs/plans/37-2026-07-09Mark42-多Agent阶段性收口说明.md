# 2026-07-09：Mark42 多Agent 阶段性收口说明

> 状态：✅ 已收口（阶段性）  
> 范围：Mark42 多Agent 的状态聚合、协调建议、历史趋势、dry-run 可执行建议、broker 留痕闭环  
> 备注：本轮目标不是“自动执行”，而是先把“看得见 / 说得清 / 留得住 / 可继续扩”做完整。

---

## 一、这次收口解决了什么

在本轮之前，Mark42 的多 Agent 能力已经有基础隔离，但还不够“系统化”：

- 能按 agent 分开读状态，但缺少统一聚合视角
- 能看单点状态，但不容易一眼判断谁忙、谁告警、谁空闲
- 有结论，但缺少历史留痕与趋势回看
- 有建议方向，但没有结构化的“下一步动作建议”

这次收口后，多Agent 这一块已经形成闭环：

**多Agent 状态 → 聚合摘要 → 协调建议 → dry-run 可执行建议 → broker 历史留痕 → 趋势回看**

## 二、当前已完成能力

### 1. Agent 隔离基础

已具备：

- `managedAgents` / `defaultAgent` 读取与解析
- agent 白名单校验
- agent 作用域的 armor / engine / heavy 状态读取
- assemble pid 文件按 agent 隔离
- `assemble status --all-agents` 聚合查看

这意味着 Mark42 已经不是“只有 main 的单体结构”，而是有了最小多 agent 运行骨架。

---

### 2. `status --all-agents` 聚合摘要

已支持：

- `mark42.py status --all-agents`
- `mark42.py status --all-agents --json`

聚合 `summary` 目前包含：

- `agentCount`
- `warningAgents`
- `runningLoops`
- `totalLoops`
- `activeHeavyTasks`
- `assembleAliveAgents`
- `armorPeak`
- `armorAverage`
- `busiestAgent`
- `idleAgents`
- `recommendationTrend`

human 输出也已同步增强：

- 默认 agent 标识
- 告警 agent 前置
- severity 图标
- 最忙 agent
- 空闲 agent
- 顶部一屏聚合摘要

---

### 3. 协调建议（recommendations）

当前已落地 5 条协调建议规则：

1. **默认 agent 是最忙且有负载**
   - 有空闲 agent → 建议分流
   - 无空闲 agent → 建议注册更多空闲 agent 或拆 loop

2. **存在空闲 agent 且不是默认 agent**
   - 提示其可作为故障接管候选

3. **多个 agent 同时告警**
   - 建议人工介入
   - 单个非默认 agent 告警时，提示优先确认默认 agent 是否接管

4. **assemble 全部停止**
   - 提示 Mark42 守护已整体下线

5. **heavy 任务集中在单一 agent**
   - 提示负载不均

说明：

- `recommendations` 已同时出现在 JSON 与 human 输出中
- 目的不是替代人判断，而是让 status 从“只报事实”变成“也给方向”

---

### 4. 历史留痕与趋势统计

这次最关键的增强之一，是把 recommendations 从“即时输出”升级成“可回看历史”。

已新增 broker 事件：

- `status.recommendations`

每次执行 `status_dashboard(all_agents=True)` 时，都会写入一份快照，包含：

- `defaultAgent`
- `summary`
- `recommendations`
- `suggestedActions`

并新增历史读取与趋势构造能力：

- `read_mark42_recommendation_history(limit=20)`
- `_build_recommendation_trend(...)`

当前趋势字段：

- `recentSnapshots`
- `recentWithRecommendations`
- `sameAsPrevious`
- `lastChangedAt`
- `topRecurringRecommendations`

human 输出也会显示：

- 最近快照数
- 其中多少次出现建议
- 近期高频建议

这满足了点点一贯强调的原则：

> 自动 / 不可见行为要有据可查；写下来 > 记着

虽然 status 本身不是自动执行，但它现在输出的“建议”已经可以被历史追踪，不会只存在于一瞬间的终端里。

---

### 5. dry-run 可执行建议（suggestedActions）

为避免过早进入“自动执行”带来的副作用，本轮先只做结构化建议动作，不直接执行。

新增聚合字段：

- `suggestedActions`

当前支持的动作类型：

1. `restart-assemble`
   - 用于 assemble 全停场景
   - 给出命令预览：`mark42.py assemble`

2. `rebalance-default-agent`
   - 用于默认 agent 过忙但其他 agent 空闲时
   - 提示把新 loop / heavy 任务优先下发到空闲 agent

3. `rebalance-heavy-tasks`
   - 用于 heavy 任务明显集中时
   - 给出后续任务分配方向

这些动作当前统一约束为：

- `dryRunOnly = true`
- 只展示、不自动执行
- 仅作为“可操作方向”交给人判断

这相当于先搭出“自动化前一层”的结构：

- recommendation 是自然语言建议
- suggestedAction 是结构化动作建议

后续若要做半自动或确认后执行，不需要重构输出协议。

## 五、边界与未做项

这次是“阶段性收口”，不是“最终态”。以下内容明确**还没做**：

### 1. 不自动执行 suggestedActions

当前所有建议动作都只是：

- 展示
- 留痕
- dry-run

并不会真正去：

- 自动重启 assemble
- 自动迁移 loop
- 自动改派 heavy 任务

这是故意保守的设计，避免多Agent 刚成型就引入静默副作用。

---

### 2. 不做复杂调度器

当前只是：

- 识别谁忙 / 谁闲 / 谁告警
- 给出建议

还没有：

- 动态负载均衡器
- 自动抢占
- 任务抢救迁移
- 真实 broker 驱动的 agent 调度器

这些应属于后续 phase，而不是本轮收口范围。

---

### 3. 趋势仍是轻量版

当前 recommendationTrend 已够实用，但还不是完整分析层。暂未提供：

- 持续时长统计
- 按 agent 聚类的长期趋势
- 建议变化 diff
- 建议消失 / 新增的显式分类

现阶段目标是“够稳、够轻、够能回看”，而不是做成重分析系统。

---

## 六、下一阶段建议

如果后续继续推进，多Agent 最自然的下一步是：

### Phase 2：确认后执行 / 半自动化

建议先做：

1. `suggestedActions` 增加稳定 schema 与 action type 约定
2. CLI 增加“显示建议动作详情”的入口
3. 对少数低风险动作支持：
   - 先 dry-run
   - 再显式确认执行
4. 所有执行动作继续写 broker 留痕

推荐优先顺序：

1. `restart-assemble` 的确认后执行
2. 新任务优先路由到空闲 agent 的建议式分配
3. heavy 任务后续分流策略

不建议直接跳到“全自动执行”，因为那会绕过当前这轮刚建立起来的可见性与可追踪性原则。

## 七、阶段结论

本轮可以定义为：

### Mark42 多Agent Phase 1.5：已收口

它的意义不是“所有事都自动完成”，而是：

- 多Agent 已经能被可靠观察
- 状态不再只是散点，而是有聚合摘要
- 建议不再只是口头说明，而是结构化输出
- 建议与动作方向都有历史留痕
- 后续要自动化时，已经有稳定承接层

换句话说：

**现在的 Mark42 多Agent，已经从“有多个 agent”进化到“像一个最小可运转的多 agent 系统”。**
