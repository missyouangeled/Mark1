- `chat.history` 是显示归一化的投影，不是原始日志（`docs/web/webchat.md:57`）。

## 4. 排查与恢复方案（不动代码）

> **原则**：先把"后端是否正常"与"前端是否异常"拆成两个独立问题；再按从外到内（前端 → Gateway → 工具）的顺序排查。

### 阶段 1：分清"后端 vs 前端"（先做这个）

| 步骤 | 操作 | 预期现象 | 判定 |
|------|------|----------|------|
| 1.1 | 打开宿主 TTY 终端，直接运行 `openclaw status` | 看到 Gateway / 会话状态 | 输出可读 → 后端工具链存活 |
| 1.2 | 在 TTY 直接执行一组代表性命令：短输出 / 长输出 / 多行 / 含路径与冒号 | stdout / stderr 都正常 | 后端工具执行无问题 |
| 1.3 | 在 TTY 用 `openclaw gateway stability --bundle latest` 看最近异常 | 应无 oversized payload / 异常退出 | 后端稳定 |
| 1.4 | 用 TTY 看 Gateway 日志（`openclaw logs tail` 或直接 `tail` 文件）| 日志可读 | 后端日志正常 |

- **如果 1.1–1.4 全正常**：根因几乎确定在 WebChat / Control UI 前端渲染层（对应 Q1 / Q2 / Q3 / Q4）。
- **如果 TTY 端日志/输出本身不可读**：才考虑后端问题；否则不要往下走。

### 阶段 2：分清"前端哪一层"（最小破坏性）

| 步骤 | 操作 | 预期 / 判定 |
|------|------|-------------|
| 2.1 | 用**全新隐私窗口**打开 Control UI / WebChat | 如果正常 → 老窗口/老 session 是缓存/历史污染（Q3 / Q4）|
| 2.2 | 在新窗口里**新建一个全新 session**（不沿用旧 session）| 如果正常 → 旧 session 历史里有触发渲染误判的污染内容（Q3）|
| 2.3 | 执行 `openclaw doctor --fix` 清理遗留配置（按文档 `docs/web/webchat.md:86` 建议）| 自动修复常见的投影 / 配置不一致 |
| 2.4 | 重启 Gateway，并对 Control UI **硬刷新**（Ctrl+Shift+R）+ 清站点数据 | 处理 service worker / PWA 缓存（Q4）|
| 2.5 | 在新 session 内重测：单行短输出 / 多行输出 / 含路径输出 | 若仍异常 → 确认是 OpenClaw 渲染层回归（Q1 / Q2 / Q8）|

### 阶段 3：定位"渲染层回归的形态"

按文档与公开 issue 提示，重点收集以下现象：

1. **长度阈值**：是否只有超过某个字符数才坏？（裁剪占位 → 误分类）
2. **多行阈值**：是否单行可读，多行不可读？
3. **特殊字符**：是否含 `:` / `/` / 路径 / JSON / 看起来像附件指令的行才坏？
4. **类型相关性**：是否 `exec` / `process` / `read` 都坏？`image_generate` 是否也坏（公开 issue 提示）？
5. **新 vs 老 session**：新 session 是否同样坏？

把以上观察写进本方案文件（追加章节），作为后续向 OpenClaw 反馈 / 自己定位源码的依据。

### 阶段 4：临时绕过（确认是渲染 bug 后）

> 这是**绕过**而不是**修复**，目的是让点点在 bug 解决前能继续工作。

- **工作流调整**：
  - 代码编辑：继续在 OpenClaw / WebChat 发起；
  - 验证输出、日志、`read` 内容：统一改在 TTY 终端看（`cat` / `less` / `tail`）；
  - WebChat 只负责发指令和接收简短结论，**不承担**"看长 stdout / stderr"的责任。
- **结构化输出建议**（仅在被 OpenClaw 误判成附件时考虑，且要明确告知点点"这是绕过不是修复"）：
  - 把结果落到文件 → TTY `cat`；
  - 在 WebChat 里只回"已写入 /path/to/file，请到 TTY 查看"这种短结论；
  - 避免在 WebChat 内回长文本 / 多行文本 / 含路径的文本。
- **严禁**：为了绕过而再次去改 `engine.py` 等业务文件——这会让排查目标丢失、且违反本方案"不动代码"原则。

### 阶段 5：根治路径（仅在阶段 3 确认为 OpenClaw 渲染 bug 后讨论）

> 当前**不做**；先把阶段 1–4 走完，再决定要不要改 OpenClaw。

可能的根治方向（待阶段 3 数据回来后细化）：

1. **升级 OpenClaw**到包含相关修复的版本（先查 CHANGELOG 中是否提到 WebChat / Control UI 渲染修复）。
2. **在源码层排查渲染分支**：重点看 tool output card 渲染、`chat.history` display-normalized、media attachments 识别、大消息裁剪后占位逻辑（对应文档段落：`docs/web/control-ui.md:105`、`docs/web/control-ui.md:175`、`docs/web/control-ui.md:186`、`docs/web/webchat.md:33`）。
3. **回滚 OpenClaw 版本**到回归前一个稳定版（如果升级后发现是新版引入的）。
