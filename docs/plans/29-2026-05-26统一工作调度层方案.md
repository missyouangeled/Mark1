## 2026-05-26：统一工作调度层方案

### 背景
当前后台任务、subagent 管理、监工监控三类能力分散在不同脚本和手动决策中。用户需要每次判断"要不要开监工""要不要 spawn""什么时候清理"，决策成本累积且容易状态不一致。

### 核心思路
把三件事——后台任务创建、subagent 管理、监工监控——收进一个统一的工作调度层：

```
用户: "帮我分析 X"
       ↓
调度层:
  1. 自动判断复杂度 → 决定前台/后台
  2. 自动选 model/thinking
  3. 自动拉起监工（如果需要后台）
  4. 排队或并行执行
  5. 监控进度，按需回报
  6. 完成后自动清理监工和分身
```

### 5 个维度收益

1. **心智模型简化**：用户只需说"帮我做 X"，调度判断、跟踪、清理全自动
2. **任务生命周期自动化**：创建→排队→运行→监控→完成→自动清理，失败自动重试
3. **资源感知调度**：检查 active 任务数，超阈值排队；大任务高 thinking+后台，小任务前台+默认
4. **统一状态日志**：一条线看所有任务全貌，不需要翻 4 个状态文件
5. **减少协议开销**：把 AGENTS.md 里 2000+ 字的手动规则消化成自动调度引擎，100% 强制执行

### 实施建议
- 分层走：先做任务队列+自动清理 → 再做智能调度判断 → 最后替换监工手动开关
- 排在"三层架构"的工作层之后，待 broker 数据层完全稳定再动
- 当前暂不实施——用户当前任务量不需要系统自动判断

### 状态
📐 方案已设计，存档备用。待换设备或任务量增大时考虑实施。

### 触发条件（未来）
- 换更强设备后
- 日常任务量明显增大，手动管理成本过高
- 三层架构中工作层需要落地时
