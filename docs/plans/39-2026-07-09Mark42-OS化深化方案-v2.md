## 2. 联网搜出来的关键参考（不抄，按 Mark42 现状挑着用）

我搜了 AI self-healing / agent loop / systemd supervision 三个方向，核心结论提炼如下：

### 2.1 AI Agent Self-Healing Patterns（Taskade / Union.ai / DEV 等）
提炼出来的"恢复阶梯"：

```
分类故障 → 退避重试 → 熔断 → 优雅降级 → 检查点续跑 → 升级到人 → 学习
```

要点：
- **先分类再决策**：transient（抖动） / permanent（坏了） / critical（崩了） 三种处理方式完全不同
- **退避必带 jitter**：10 个 agent 同时撞 rate limit，不加 jitter 会"雪崩重试"
- **必须有熔断**：某个东西一直坏就别再试，先告警
- **必须能续跑**：checkpoint + resume，不能每次失败从头来
- **实在不行升级给人**：这是兜底，不丢人

### 2.2 systemd 的 supervision pattern
这是工业级最朴素的"循环兜底"，提炼：

```
Restart=on-failure / always
RestartSec=（带退避）
StartLimitBurst + StartLimitIntervalSec（防"狂重启"）
WatchdogSec + sd_notify（heartbeat，进程活着但卡死也能拉起）
OnFailure=（兜底 service）
```

Mark42 当前**没用 systemd 守护**（守护都是 Python 进程组），所以这套要翻译成 Mark42 自己的实现。

### 2.3 OODA / Diag-Mitigate-Repair-Verify-Document
两类经典自愈循环，Mark42 选 **Diag-Mitigate-Repair-Verify-Document**，因为：
- OODA 偏军事决策，不适合 OS 层
- 5 段式（DMRVD）每一步都有"产出物"，**正好符合点点一直强调的"留痕 > 记着"原则**

## 3. v2 新增：Mark42 问题自愈总架构 ⭐

### 3.1 通俗版解释（给点点看的）

我把整个自愈体系想成一个**医院**：

- **总 loop** 就像医院的"急诊分诊台"——不管哪儿出毛病，先到它这报到
- **分诊台**判断是小毛病（自己拿创可贴）、中毛病（开药）、大毛病（住院）还是重症（叫家属）
- **每种毛病**有对应的"专科 loop"——一个治 armor、一个治 engine、一个治 assemble、一个治 heavy 任务
- **每个专科 loop**都按 DMRVD 五段跑（看诊 → 应急 → 治 → 复查 → 写病历）
- **跑一圈没治好**就再跑一圈，但每跑一圈等级升一档（重启→换策略→叫人）
- **实在治不好**才把病历递给点点

### 3.2 架构图（文字版）

```
┌─────────────────────────────────────────────────────────────┐
│              Mark42 Self-Healing 总 Loop                      │
│            (mark42_selfheal_supervisor)                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼  感知到问题
┌─────────────────────────────────────────────────────────────┐
│              Severity Classifier                            │
│   info（小事）→ warn（小病）→ error（中病）→ critical（重症） │
└─────────────────────────────────────────────────────────────┘
            │              │              │              │
            ▼              ▼              ▼              ▼
      ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
      │armor loop│   │engine loop│   │assemble loop│  │heavy loop│
      │(内存压缩)│   │(loop守护)│   │(进程守护) │   │(任务回收)│
      └──────────┘   └──────────┘   └──────────┘   └──────────┘
            │              │              │              │
            └──────────────┴──────────────┴──────────────┘
                              ▼
                    ┌──────────────────┐
                    │  DMRVD 五段执行  │
                    │ Diag→Mitigate→   │
                    │ Repair→Verify→   │
                    │ Document         │
                    └──────────────────┘
                              │
                              ▼  验证失败
                       升级一档（最多 N 档）
                              │
                              ▼  还是失败
                      升级给点点（带完整病历）
```

### 3.3 DMRVD 五段详解

| 段 | 干啥 | 产出物 |
|---|---|---|
| **D** Diag 诊断 | 读状态、查 broker、看最近 50 条日志 | 一份"诊断报告"（dict） |
| **M** Mitigate 应急 | 先把影响按住（限流 / 隔离 / 临时关闭） | 应急动作清单 |
| **R** Repair 修复 | 真去修（重启进程 / 触发压缩 / 清理任务） | 修复动作 + 返回值 |
| **V** Verify 验证 | 重新跑 health check，确认指标回到绿 | verify_report（绿/黄/红） |
| **D** Document 留痕 | 写 broker `selfheal.<loop>.<action>` 事件 | 历史可回查 |

**关键约束**：
- 五段必须**全部完成**才能算一次"治愈"
- 任何一段失败，**整体失败**，升级一档重试
- Verify 阶段**必须等指标真变绿**，不能"看起来好了就完事"

### 3.4 升级阶梯（每跑一圈没治好，等级 +1）

| 等级 | 处理方式 | 触发条件 |
|---|---|---|
| L0 | 仅 Document | info 级，仅记 |
| L1 | 自动 Retry（带 backoff + jitter） | warn 级 |
| L2 | 重启对应子进程 | error 级，L1 失败后 |
| L3 | 重启父进程 + 通知 | error 级，L2 失败后 |
| L4 | 熔断该子系统的"自愈"，告警 + 等人 | critical 或 L3 失败后 |
| L5 | 全局告警 + 静默等待点点 | 多个 L4 同时发生 |

**为什么不超过 N 圈**：避免"看似在干活、实际在死循环"。N 默认 3，可在 `mark42.json` 配。

### 3.5 Loop 之间互不打架

- 每个子 loop 有自己的 `loop_id` + 互斥锁
- armor loop 在跑时，assemble loop 不会去碰 armor
- 但**总 supervisor 能同时观察多个子 loop**
- 如果多个子 loop 都 L3+，**合并告警**给点点（不刷屏）

## 4. v2 完整目标拆解表（v1 + 新增）

| OS 类比 | 当前 | v2 目标 |
|---|---|---|
| 内核稳定性 | 守护多进程，缺兜底 | **30s 内自愈拉起；连续 3 次失败才升级给人** |
| 资源管理 | 手动触发 | **loop 自动监控 + 自动分配 + 自动回收 + 验证** |
| 文件系统 | 自动轮替，无告警 | **涨速异常自动告警 + 自动清理** |
| 调度器 | 单进程 | **多模板隔离 + 单崩不波 + 故障模板自动隔离+告警** |
| 用户态 | CLI + dry-run | **故障自愈 + DMRVD 留痕 + L5 升级通知** |
| 错误处理 | try/except 散落 | **统一分类器 + 严重度分级 + 自动 loop 路由** |
| **🆕 自愈能力** | **无** | **总 supervisor + 4 子 loop + DMRVD + 升级阶梯 + 熔断** |

## 5. v2 阶段落地（替代 v1 的 OS-1 ~ OS-4）

| 阶段 | 范围 | 交付物 | 验收 |
|---|---|---|---|
| **SH-1** | 分类器 + 总 supervisor 骨架 + broker 留痕 | severity 判定、状态机、DMRVD 五段函数、broker 事件 | 单测 100% 覆盖 severity 分类 |
| **SH-2** | armor 子 loop（含 DMRVD） | armor 阈值监控 → 压缩 → verify 绿 | 连续运行 7 天，armor 自愈 ≥ 95% |
| **SH-3** | engine 子 loop（含 DMRVD） | 多 loop 隔离 + 单崩自愈 | 模拟单 loop 崩溃，能自愈 |
| **SH-4** | assemble 子 loop（含 DMRVD） | 进程组保活 + 升级阶梯 | 模拟进程崩溃，能自愈 |
| **SH-5** | heavy 子 loop（含 DMRVD） | 超时任务自动降级/回收 | 模拟任务卡死，能自愈 |
| **SH-6** | 熔断 + L5 全局告警 + 升级链路 | 连续失败自动熔断 + 通知点点 | 模拟连续失败，能熔断 |

每个阶段必须：
1. dry-run 验证
2. 单测 + 集成测全绿
3. **连续运行 ≥ 24 小时不出问题**再进入下一阶段（这条是硬规则）

## 6. v2 不做的事（边界，扩 v1）

- ❌ 不让 loop 自动改用户数据（只能改 Mark42 自己的状态/进程）
- ❌ 不让 loop 自动升级自己（必须点点手动）
- ❌ 不让 loop 跨机器联动（单机自愈，不搞分布式协调）
- ❌ 不跳过 DMRVD 任何一段（哪怕看似多余）
- ❌ 不在 verify 失败时假装成功（必须升级，不能糊弄）

## 7. 风险与依赖

| 风险 | 缓解 |
|---|---|
| loop 自身 bug 导致"自愈死循环" | N 圈上限 + 熔断 + L4 强制升级 |
| DMRVD 写出错（写一半崩） | 每段先写"开始"事件，再写"结束"事件；失败也能看到断点 |
| 多个 loop 同时自愈互相打架 | 互斥锁 + supervisor 协调 |
| 自愈动作影响性能 | 后台异步执行，CPU/IO 限流 |
| verify 假阳性（看似绿实际没好） | 多指标交叉验证（至少 2 个指标同时绿才算绿） |

## 8. v2 阶段结论

**v2 把 Mark42 从"看得见问题"升级到"能自己一直修"。**

核心是引入了**总 supervisor + 子 loop + DMRVD + 升级阶梯**四件套：
- 总 supervisor：分诊台
- 子 loop：专科医生
- DMRVD：治疗流程
- 升级阶梯：治不好就叫人

**稳大于强**依然是顶层原则：
- 每个子 loop 都要先回答"我自己挂了谁拉我"（答案是 supervisor）
- supervisor 要回答"我挂了谁拉我"（答案是 systemd 或独立 watchdog）
- 整个体系要回答"全部都挂了怎么办"（答案是 L5 通知点点）

**v3 候选方向**（不写在 v2 里，避免膨胀）：
- 自愈策略学习（从历史成功/失败的 loop 里学）
- 自愈报告周报（每周自动汇总）
- 自愈参数自调（根据成功率动态调整退避时间）
