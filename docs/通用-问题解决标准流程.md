# 问题解决标准流程

> **适用场景**：任何影响系统稳定性、数据完整性、或跨多个文件/模块的修改。
> **建立日期**：2026-06-10
> **触发关键词**：修、改、查、排查、定位、根因、修复、优化、重构、清理

---

## 六步法

### ① 搜索相关内容，确定问题

- **先搜，不猜**：查日志、查配置、查代码、查历史记录、查 memory
- **数据说话**：有日志看日志，有状态看状态，有 metrics 看 metrics
- **复现确认**：能复现的问题才是真问题；不能复现的标记为「待观察」
- **禁止**：没查就下结论、凭记忆臆断

### ② 分析问题，定位根因链

- **不只看表象**：表象是「Gateway 频繁重启」，根因链可能是「timer 死循环 → systemctl restart → 新进程未就绪 → 再次触发」
- **画因果链**：A 导致 B，B 导致 C，找到最初的 A 而不是停在中间的 B
- **区分「相关」和「因果」**：同时发生不一定是因果

### ③ 找对应解决办法

- **查现有工具/脚本**：有没有现成的诊断脚本、修复脚本、应急方案
- **评估影响范围**：修这里会不会影响别处？改这个参数会不会引发其他问题？
- **给多个方向**：至少 2 个可选方案，标注优劣

### ④ 给出初步方案

- **方案必须包含**：
  - 改哪些文件
  - 每个文件改什么（具体到行）
  - 预期效果
  - 回滚方式
- **不能只有「我觉得」**：每个改动都要有依据

### ⑤ 自我审核（必须执行，不可跳过）

**五项必查：**

| 检查项 | 问自己 |
|--------|--------|
| 🔗 逻辑悖论 | 方案中 A 和 B 会不会互斥？有没有自相矛盾？ |
| 🔀 牵连影响 | 改了这里，还有哪些地方会受影响？timer？启动链？其他脚本？ |
| 🕳️ 遗漏 | 有没有没覆盖到的边缘场景？空值？权限？跨天？重启后？ |
| 🐛 Bug 风险 | 新代码有没有竞态？有没有资源泄漏？有没有写/读冲突？ |
| 💾 数据安全 | 会不会丢数据？先备份了吗？删了能恢复吗？ |

**审核结果处理：**
- 无问题 → 直接交付
- 有问题 → 修复后重新自审，通过后再交付

### ⑥ 验证 & 记录

- **改完必验**：运行、看日志、确认生效
- **写入变更流水**：`docs/通用-OpenClaw-补丁变更流水.md`（正式补丁）
- **非正式修改入备忘录**：`docs/通用-OpenClaw-非正式修改备忘录.md`
- **更新 MEMORY.md**：如果涉及新规则/新偏好/新配置

## 反面教材（今日案例）

| 错误做法 | 正确做法 |
|----------|----------|
| 看到 gateway 重启就调大超时阈值 | 追查谁在杀进程（resume-watch timer 死循环） |
| 凭记忆说「配置已经在了」 | 先查配置文件、查备份快照、查 daily 记录 |
| 改完就走 | 验证 → 写流水 → 更新 MEMORY → 更新 daily 摘要 |
