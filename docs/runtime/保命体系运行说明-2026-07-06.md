## 一、这轮实际做了什么

本轮不是停留在方案层，而是把“主会话保命状态聚合器”真正做出来并跑通。

当前聚合器会只读采集以下信息：

1. 主 session 实况
   - 当前逻辑主会话 `sessionId`
   - 主 jsonl 行数 / 大小
   - 当前 trajectory 大小
   - model / token 上下文等基础状态

2. watcher 告警状态
   - `session-size-watcher/alerts.json`
   - `session-size-watcher/state.json`
   - 区分活跃告警 / 历史未清 / 已消化历史告警

3. 保命链路状态
   - 救命 1 cron 最近运行时间 / 状态
   - frontstage guardian 最新状态
   - health collector 最新状态
   - backup 最近快照时间
   - 关键 openclaw / mark42 systemd 单元摘要

4. 自动通知状态
   - 读取聚合器最近事件
   - 在 `WARN / CRITICAL / DEADMAN / 恢复到 OK` 时决定是否前台注入
   - 30 分钟同级别 cooldown，避免刷屏

5. 可插拔修复层
   - 核心保命层只负责触发，不内嵌高副作用修复逻辑
   - repair runner 仅按白名单调度外部 repair pack
   - 当前已支持多插件串行判断与独立 cooldown
   - 删除 repair pack 后，核心巡检/告警仍能工作

## 二、本轮修掉的两个误判点

### 1. health 顶层 summary 和结构化结果打架

最初发现 `health-collector` 顶层：
- `ok: true`
- `summary` 却写着“发现 1 个卡住会话”

如果聚合器只看顶层摘要，就会误把系统判成故障。

现在修正为：
- 不只看 health 顶层 `summary`
- 会优先检查 `checks[]`
- 若 `stdoutRaw` 里嵌着结构化 JSON，则进一步解析其中的：
  - `ok`
  - `blockedMain`
  - `totalStuck`
  - `summary`
- 结构化字段优先于人类摘要文案

结果：避免被一句旧 summary 误导。

### 2. watcher 历史尾巴长期卡住 WARN

原先 `alerts.json` 里有很多旧的 trajectory 告警；即使当前主会话 trajectory 已回落到阈值以下，聚合器仍可能继续报 `WARN`。

现在修正为：
- 若某条 watcher 告警之后，`session-size-watcher` 已再次成功运行
- 且当前主会话 trajectory 已低于 5 MB 阈值
- 则该告警视为“已消化历史告警”
- 保留留痕，但不再抬高总体状态

结果：真正的“当前风险”和“历史尾巴”被分开了。

## 三、当前判定规则

总体状态按以下优先级上浮：

- `DEADMAN`
- `CRITICAL`
- `WARN`
- `OK`

### 会触发 WARN / CRITICAL / DEADMAN 的主要条件

1. watcher
- 最近 1 小时内仍对应超阈值的 trajectory 告警 -> `WATCHER_ACTIVE_ALERTS` / `WARN`
- 只有历史未清、但不再对应当前风险 -> 不再强行上浮

2. 主 session trajectory
- `>= 7 MB` -> `CRITICAL`
- `>= 5 MB` -> `WARN`

3. 主 session 行数
- `>= 12000` -> `CRITICAL`
- `>= 8000` -> `WARN`

4. backup
- 最近快照不可读或超过 25 分钟 -> `DEADMAN`
- 超过 15 分钟 -> `WARN`

5. cron
- 救命 1 状态不可读 -> `DEADMAN`
- 最近运行距离当前超过 15 分钟 -> `DEADMAN`
- 最近状态不是 `ok` -> `CRITICAL`

6. frontstage / health
- 状态过旧或明确异常 -> `WARN`
- health 子检查若明确 `blockedMain` -> `CRITICAL`

7. systemd
- 只把真正 `failed` 的 openclaw / mark42 单元记为严重问题
- 正常的一次性 service `inactive/dead` 不再误判

## 四、事件留痕与自动通知规则

`events.jsonl` 现在区分三类事件：

- `heartbeat`
  - 本轮状态和 finding code 与上一轮一致

- `finding_change`
  - 总体级别没变，但具体风险项变了

- `level_change`
  - 总体状态变化（如 `WARN -> OK`）

这样后面接 cron 时，可以快速看出是正常心跳，还是风险升级/降级。

同时新增了自动通知层：

- `CRITICAL` / `DEADMAN`
  - 默认尝试前台注入告警
- `WARN`
  - 仅在级别首次抬升到 `WARN` 时提醒，避免每 5 分钟重复刷
- `OK`
  - 仅在发生 `level_change`、也就是从非 OK 恢复到 OK 时发送恢复提示
- 同级别 30 分钟 cooldown
  - 防止同一风险在短时间内重复打扰

## 七、自动巡检接线方式

考虑到 OpenClaw 官方文档建议：精确定时任务优先用 cron，且 command job 适合做确定性的 shell / 脚本聚合；为了避免新开并发 job 带来的竞争，本轮没有再新增第二个保命 cron，而是直接复用现有的“救命 1”每 5 分钟入口。

当前链路顺序是：

1. `scripts/emergency1.sh`
2. `scripts/emergency1-orchestrator.py`
   - 统一串行执行 `aggregator -> notify -> repair-runner`
   - 再执行主 session 行数检查与 watcher 未读转发
   - shell 只保留薄壳入口，不再承载判定逻辑
3. `scripts/emergency0-aggregator.py`
4. `scripts/emergency0-notify.py`
5. `scripts/emergency0-repair-runner.py`
6. 最后输出统一的 cron 诊断摘要

这样做的好处：

- 不增加新的 cron 并发源
- 统一由一个入口完成巡检、通知、修复触发、旧保命逻辑
- 任一步骤失败都会让 cron run 明确失败，不会假装全绿

## 八、可插拔修复层设计

当前修复层不是把自动修复硬写进核心巡检器，而是拆成：

- 核心层：`emergency0-aggregator.py` / `emergency0-notify.py` / `emergency0-repair-runner.py`
- 插件层：`skills/emergency-repair-pack`

核心层负责：

- 发现风险
- 判断是否命中修复白名单
- 控制 cooldown
- 记录修复日志

插件层负责：

- 执行非常窄的修复动作
- 必须满足：幂等、可回滚、不碰主 session、不改系统配置

当前已落地六个 repair 动作：

- `repair_stale_watcher_alerts.py`
  - 仅处理 `staleItems`
  - 若存在活跃告警则直接跳过
- `repair_archive_resolved_watcher_alerts.py`
  - 仅处理聚合器已判定为 `resolvedItems` 的历史告警
  - 会把原始告警追加归档到 `watcher-resolved-archive.jsonl`
  - 然后从 `session-size-watcher/alerts.json` 安全剔除这些已解决旧告警
- `repair_backup_kick_once.py`
  - 仅在 backup 超时、且 cron/frontstage/health 都正常时补跑一次备份
  - 调用现有 `openclaw-session-backup.py --quiet`
  - 不改 systemd 配置、不重启服务
- `repair_health_collect_once.py`
  - 仅在 health 状态过旧、且 cron/frontstage 正常时补跑一次健康采集
  - 调用现有 `openclaw-health-collector.py --print-human`
  - 不改 systemd 配置、不重启服务
- `repair_frontstage_guardian_collect_once.py`
  - 仅在 frontstage 状态过旧、且 cron/health 正常时补跑一次前台保护检查
  - 调用现有 `openclaw-frontstage-guardian.py --print-human --no-notify`
  - 不改 systemd 配置、不重启服务
  - 明确禁用 notify，避免修复动作本身制造前台噪音
- `repair_audit_snapshot.py`
  - 只读追加当前聚合状态到 `repair-audit.jsonl`
  - 不执行修复，不触碰系统状态

其中前五个属于轻修复，第六个属于只读审计。

共同约束：

- 不处理活跃 trajectory 告警
- 不改主 session
- 不改系统配置

因此，删除 `skills/emergency-repair-pack` 后：

- repair runner 会自动 `plugin_missing` 并跳过
- 核心监测、告警、留痕仍继续工作
- 系统不会因为删掉 repair pack 而失去基本保命能力
