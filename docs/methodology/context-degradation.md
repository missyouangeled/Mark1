# 上下文退化诊断（改编自 Agent Skills for Context Engineering）

> 来源：muratcankoylan/agent-skills-for-context-engineering
> 当 agent 表现莫名变差时，用这 5 种模式排查根因。

---

## 五种退化模式

上下文退化不是二元的"好/坏"，而是一个连续谱，表现为 5 种可预测的模式：

| # | 模式 | 一句话 | 典型信号 |
|---|------|--------|---------|
| 1 | **Lost-in-Middle** | 在上下文中间的信息被忽略 | 明明给了正确数据却视而不见 |
| 2 | **Poisoning** | 一段错误信息污染整个上下文 | 越纠正越错，幻觉持续出现 |
| 3 | **Distraction** | 无关内容挤占了注意力 | 加了无关文档后表现明显下降 |
| 4 | **Confusion** | 多个任务混合导致约束串用 | 回应文不对题，调用了错误上下文的工具 |
| 5 | **Clash** | 多条各自正确但互相矛盾的信息并存 | 版本冲突、多源数据互相矛盾 |

---

## 1. Lost-in-Middle

### 原理

注意力 U 型曲线：开头和结尾的内容获得可靠注意力，中间内容有 10-40% 的召回率下降。这个效应在 4K token 以上开始显著。

### 信号

- 正确答案在上下文里，但 agent 无视了它
- 回应与已提供的数据矛盾
- "忘记"了长提示词中段给出的指令

### 修复

- 关键信息放在**开头或结尾**，绝不放中间
- 给长文档加前置摘要（开头）+ 结论强调（结尾）
- 用明确的章节标题做"注意力锚点"

---

## 2. Context Poisoning

### 原理

一次幻觉、一个工具错误、一段错误的检索结果进入上下文后，通过反复自引用而放大。一个被污染的"目标"字段会让每一个下游决策都基于错误假设。

### 污染源（三个主要向量）

1. 工具输出中的错误
2. 检索文档中的不实信息
3. 模型自身生成的摘要中的偏差

### 信号

- 之前能成功的任务现在输出质量下降
- 工具调用跑偏（选了错误的工具或参数）
- 明确纠正后幻觉仍持续出现

### 修复

**覆盖纠正不管用**——要在被污染内容上叠加纠正，原始错误仍保留注意力权重。

正确做法：
1. 截断到污染点之前
2. 用只包含已验证信息的新上下文重启
3. 或显式标记污染段落，要求从头重新评估

---

## 3. Context Distraction

### 原理

研究显示：**哪怕只加一篇无关文档，也能测量到性能下降**——效果呈阶跃函数，不是线性曲线。多篇无关文档叠加会恶化。

模型不能"跳过"无关内容，必须关注它被喂入的一切。

### 信号

- 加了检索文档后质量反而下降
- 回应跑题，聚焦在次要细节上
- 工具调用增多却没有推进核心任务

### 修复

- 喂入前先做相关性过滤
- 可能用到但不是立即需要的信息 → 放工具调用后面按需获取
- 不要让上下文变成"什么都在里面"的杂物间

---

## 4. Context Confusion

### 原理

上下文里混了多种任务类型或目标切换时，模型会把错误任务的约束套到当前任务上。这是**约束污染**，不只是注意力稀释。

### 信号

- 回应对着问题的错误方面
- 工具调用适合另一个任务
- 输出混合了多个来源的需求

### 修复

- **不同任务放不同上下文窗口**——最彻底的解决方案
- 同一会话中必须切换任务时，用明确的"上下文重置"标记
- 状态管理要按任务隔离目标/约束/工具

---

## 5. Context Clash

### 原理

多条信息各自正确但互相矛盾（版本冲突、多源检索数据不一致）。模型没有能力自行判定哪个源优先。

### 信号

- 输出在两个矛盾答案间摇摆
- 引用了一个源的同时也引用了矛盾的另一个源
- 同样的查询在不同轮得到不同结果

### 修复

- **建立源优先级规则**：版本号 > 时间戳 > 权威性
- 冲突无法避免时，显式标注：`[冲突] A 说 X，B 说 Y，按规则采用 A`
- 喂入前先过滤过期版本

## 实用参考阈值

| 信号 | 阈值 |
|------|------|
| 上下文利用率 >60% | 开始监控退化信号 |
| 上下文利用率 >70% | 触发 compaction |
| 上下文利用率 >80% | 触发 aggressive compaction + masking |
| 上下文利用率 >85% | 高危——此时做压缩，模型本身质量已下降 |
| 错误/幻觉在 3 轮内持续 | 怀疑 poisoning |
| 已知信息被无视 | 检查是不是在上下文中间位置 |

---

## 对我们最实用的三条

1. **Poisoning 是"叠加纠正不管用"的**——发现我在一个错误上反复绕时，最好的修复可能是截断重启，而不是继续纠正
2. **工具输出别全留在上下文里**——老轮次的冗长输出就是 distraction 源
3. **重要指令放开头或结尾**——放中间位置就是在赌我还能注意到它
