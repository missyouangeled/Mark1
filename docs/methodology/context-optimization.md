# 上下文优化（改编自 Agent Skills for Context Engineering）

> 来源：muratcankoylan/agent-skills-for-context-engineering
> 本文合并了 context-optimization 和 context-compression 两个 skill 的核心内容，适配 OpenClaw 环境。

---

## 核心原则

**优化目标不是 tokens-per-request，而是 tokens-per-task**——完成一个任务总共消耗的 token。当你为了省 token 而压掉了文件路径、错误消息或决策理由，导致 agent 后续不得不重新读取、重新探索、重新推理，省下的 token 远不够浪费的多。

---

## 四层优化策略（按优先级）

| 优先级 | 策略 | 质量风险 | 适用场景 |
|--------|------|---------|---------|
| ① | **KV-cache 优化** | 零风险 | 所有场景（我们 API 模型受限） |
| ② | **Observation Masking** | <2% | 工具输出占 50%+ 时 |
| ③ | **Compaction** | <5% | 上下文利用率超 70% 时 |
| ④ | **Context Partitioning** | 有协调开销 | 单窗口装不下时 |

---

## ① KV-cache 优化（对我们限制较大）

我们用的是 API 模型，无法直接控制推理引擎的 KV cache。但能做的：

- **系统提示词放最前面**且不变——不要在里面插时间戳、版本号或会话 ID
- **不要每轮重建提示词结构**——保持前缀稳定
- 一个空格或换行的变化就可能使整个前缀缓存失效

> ⚠️ 对我们：这条更多是意识，实际收益有限。重点在下面三条。

---

## ② Observation Masking（工具输出压缩）

工具输出在典型 agent 轨迹里吃掉 80%+ 的 token。这条最实用。

### 什么时候 mask

| 情况 | 动作 |
|------|------|
| 最新一轮的输出 | **绝不 mask** |
| 当前任务的活跃推理链中的输出 | **绝不 mask** |
| 错误输出 + 正在调试 | **绝不 mask** |
| 3 轮以上的老输出，关键点已被提取 | mask：`[Obs:ref_id 已压缩。要点: {摘要}]` |
| 重复输出 / 样板头部 | **立即 mask** |

### 目标

- 被 mask 的观察中压缩 60-80%，质量影响 <2%
- 保留可恢复性——存原文到外部，上下文里留引用 ID

---

## ③ Compaction（上下文压缩）

### 触发时机

```
上下文利用率 >70% → 触发 compaction
上下文利用率 >80% → 触发 aggressive compaction + masking（最后有效窗口，质量有风险）
上下文利用率 >85% → 高危——模型自身质量已下降，优先分区/截断而非继续压缩
```

**不要等到 85%+ 再压**——此时模型自身已处于上下文压力下，摘要质量会急剧下降。70% 时动手是最好的窗口：模型还清醒，能产出高质量摘要。

### 按消息类型保留什么

| 消息类型 | 保留 | 丢弃 |
|---------|------|------|
| **工具输出** | 关键发现、指标、错误码、结论 | 冗长原始输出、样板头部 |
| **对话轮次** | 决策、承诺、用户偏好、上下文切换点 | 填充词、寒暄、已经被结论覆盖的探索过程 |
| **检索文档** | 与当前任务相关的声明/事实/数据 | 一次性推理用的支撑证据和展开论述 |
| **系统提示词** | **永不压缩** | — |

### 结构化摘要格式

不要自由格式摘要。用强制字段防止静默丢信息：

```markdown
## 会话目标
[用户要达成什么]

## 已修改文件
- scripts/xxx.py: 改了认证逻辑
- config/yyy.json: 更新连接池配置

## 已做决策
- 采用 Redis 连接池而非每请求创建连接
- 失败重试用指数退避

## 当前状态
- 14 通过，2 失败
- 还剩：mock 配置

## 下一步
1. 修剩余测试
2. 跑全量回归
```

**为什么结构化**：每节都是强制填空的清单，漏了任何一项都肉眼可见。自由格式压缩最容易无声无息地丢信息。

### 目标

- 50-70% token 减少，<5% 质量下降
- 超过 70% 减少 → 检查是否有关键信息被吞

## ④ Context Partitioning（上下文分区）

我们已经在用——把工作拆到分身上。关键判断：

### 什么时候拆

- 预估任务上下文 > 窗口限制 60%
- 有 3 个以上可独立并行的子任务
- 不同子任务需要不同的工具/提示词

### 什么时候不拆

- 少于 3 个独立子任务——协调开销可能超过省下的 token
- 子任务之间有强依赖，无法并行

### 协调开销

每个分身需要自己的系统提示词、工具定义和协调消息。多 agent 系统的 token 成本大约是单 agent 的 **15 倍**——预算时要心里有数。

---

## 预算管理

在长时间会话开始前，心里分一下 token 预算：

| 类别 | 建议占比 |
|------|---------|
| 系统提示词 | 固定 |
| 工具定义 | 固定 |
| 检索文档 | 10-20% |
| 对话历史 | 30-50% |
| 工具输出 | 20-40% |
| 预留缓冲 | 5-10% |

**触发式优化而非周期性优化**——当某个类别超预算或总利用率 >70% 时触发，而不是每 N 轮无脑压一次。

## 对我们实际执行的规则

### 压缩规则（明确判断标准）

| 场景 | 动作 | 原因 |
|------|------|------|
| **明确是日常聊天** | 可压缩，只要主要意思对 | 寒暄/闲聊的精确措辞不需要保留 |
| **涉及工作/工程/决策/修改** | **尽量不要压缩** | 丢一个路径/错误码/决策理由 → 后续全乱 |
| **工具输出中的文件路径/错误/命令** | **绝不压缩** | 这是恢复链条，丢了就断了 |
| **3 轮以上的老工具输出** | 替换为 `[Obs:ref_id 已压缩。要点:{摘要}]` | 但先确认关键点已被提取 |
| **重复输出/样板头部** | 立即 mask | 零信息损失 |

### 场景切换时的压缩衔接

当会话从"日常聊天"切换到"工作型任务"时：

1. **切前检查**：当用户发出"继续刚才的设计/任务/工作"等回归信号后、实际继续工作之前，先花一次推理列出被压缩段落的主题摘要（1-2句），确认不含工作需要的信息。若被压缩段落中包含路径/命令/决策，必须从外部存储恢复后再继续。
2. **引用回溯**：若用户在新任务中提到"之前说的那个"、"刚才提到的路径"等回溯引用，先检查被压缩部分是否包含该信息——如果压缩掉了，主动问用户确认
3. **切换信号**：用户首次提出含具体技术动作的工作诉求时（修改文件/执行命令/调整配置/编写脚本/系统操作/排查故障——具体标志：提到了文件路径、命令名、脚本名、服务名、配置项），压缩策略立即从"可压缩"切换到"尽量不压缩"，不需要等下一轮。仅查询信息类（"查一下 xxx 是什么"）仍按日常聊天处理。
4. **日常聊天的压缩非破坏性**：压缩日常聊天时保留一段简短摘要（1-2 句该段聊天的主题），便于后续回溯时快速判断是否需要恢复原文

### 拆分身规则

**AGENTS.md 场景判定表优先**——拆不拆分身的第一判断依据是「是否阻塞前台」「是否需要协作/插播」。context-optimization.md 下面的技术标准是第二层参考，用于在已决定可分身后优化怎么分。

按 `AGENTS.md` 现有场景判定表执行：
- 纯日常聊天 → 不开分身
- 简单工作不阻塞前台 → 不开分身
- 复杂工程且阻塞前台 → 按需开
- 拿不准 → 按工作型处理

### 执行优先级

1. **先判断场景类型**：日常聊天 vs 工作型
2. **工作型优先不压缩**：宁可多耗 token，不赌信息不丢
3. **日常聊天大胆压缩**：保留大意即可
4. **拆分身按现有表走**：不引入新判断逻辑

### 关于日常聊天压缩与 Distraction

日常聊天压缩后以简短摘要（1-2 句主题）替代原文，实际上**提升了信噪比**——去掉了寒暄/重复/填充词，保留了聊天的大意和情绪走向。这与 context-degradation.md 的 Distraction 模式不矛盾：Distraction 的敌人是无关内容挤占注意力，而压缩后的摘要恰恰是去噪后的信号。

关鍵是：压缩时**不丢关键信息**（若用户聊到了工作相关的路径、命令、决策，就不能压），只压纯粹的闲聊段。
