# 上下文优化（改编自 Agent Skills for Context Engineering）

> 来源：muratcankoylan/agent-skills-for-context-engineering
> 本文合并了 context-optimization 和 context-compression 两个 skill 的核心内容，适配 OpenClaw 环境。

---

## 核心原则

**优化目标不是 tokens-per-request，而是 tokens-per-task**——完成一个任务总共消耗的 token。当你为了省 token 而压掉了文件路径、错误消息或决策理由，导致 agent 后续不得不重新读取、重新探索、重新推理，省下的 token 远不够浪费的多。

---

## 四层优化策略（按优先级）

| 优先级 | 策略 | 质量风险 | 适用场景 |
|--------|------|---------|---------|
| ① | **KV-cache 优化** | 零风险 | 所有场景（我们 API 模型受限） |
| ② | **Observation Masking** | <2% | 工具输出占 50%+ 时 |
| ③ | **Compaction** | <5% | 上下文利用率超 70% 时 |
| ④ | **Context Partitioning** | 有协调开销 | 单窗口装不下时 |

---

## ① KV-cache 优化（对我们限制较大）

我们用的是 API 模型，无法直接控制推理引擎的 KV cache。但能做的：

- **系统提示词放最前面**且不变——不要在里面插时间戳、版本号或会话 ID
- **不要每轮重建提示词结构**——保持前缀稳定
- 一个空格或换行的变化就可能使整个前缀缓存失效

> ⚠️ 对我们：这条更多是意识，实际收益有限。重点在下面三条。

---

## ② Observation Masking（工具输出压缩）

工具输出在典型 agent 轨迹里吃掉 80%+ 的 token。这条最实用。

### 什么时候 mask

| 情况 | 动作 |
|------|------|
| 最新一轮的输出 | **绝不 mask** |
| 当前任务的活跃推理链中的输出 | **绝不 mask** |
| 错误输出 + 正在调试 | **绝不 mask** |
| 3 轮以上的老输出，关键点已被提取 | mask：`[Obs:ref_id 已压缩。要点: {摘要}]` |
| 重复输出 / 样板头部 | **立即 mask** |

### 目标

- 被 mask 的观察中压缩 60-80%，质量影响 <2%
- 保留可恢复性——存原文到外部，上下文里留引用 ID

---

## ③ Compaction（上下文压缩）

### 触发时机

```
上下文利用率 > 70% → 触发 compaction
```

**不要等到 85%+ 再压**——此时模型自身已处于上下文压力下，摘要质量会急剧下降。

### 按消息类型保留什么

| 消息类型 | 保留 | 丢弃 |
|---------|------|------|
| **工具输出** | 关键发现、指标、错误码、结论 | 冗长原始输出、样板头部 |
| **对话轮次** | 决策、承诺、用户偏好、上下文切换点 | 填充词、寒暄、已经被结论覆盖的探索过程 |
| **检索文档** | 与当前任务相关的声明/事实/数据 | 一次性推理用的支撑证据和展开论述 |
| **系统提示词** | **永不压缩** | — |

### 结构化摘要格式

不要自由格式摘要。用强制字段防止静默丢信息：

```markdown
## 会话目标
[用户要达成什么]

## 已修改文件
- scripts/xxx.py: 改了认证逻辑
- config/yyy.json: 更新连接池配置

## 已做决策
- 采用 Redis 连接池而非每请求创建连接
- 失败重试用指数退避

## 当前状态
- 14 通过，2 失败
- 还剩：mock 配置

## 下一步
1. 修剩余测试
2. 跑全量回归
```

**为什么结构化**：每节都是强制填空的清单，漏了任何一项都肉眼可见。自由格式压缩最容易无声无息地丢信息。

### 目标

- 50-70% token 减少，<5% 质量下降
- 超过 70% 减少 → 检查是否有关键信息被吞

## ④ Context Partitioning（上下文分区）

我们已经在用——把工作拆到分身上。关键判断：

### 什么时候拆

- 预估任务上下文 > 窗口限制 60%
- 有 3 个以上可独立并行的子任务
- 不同子任务需要不同的工具/提示词

### 什么时候不拆

- 少于 3 个独立子任务——协调开销可能超过省下的 token
- 子任务之间有强依赖，无法并行

### 协调开销

每个分身需要自己的系统提示词、工具定义和协调消息。多 agent 系统的 token 成本大约是单 agent 的 **15 倍**——预算时要心里有数。

---

## 预算管理

在长时间会话开始前，心里分一下 token 预算：

| 类别 | 建议占比 |
|------|---------|
| 系统提示词 | 固定 |
| 工具定义 | 固定 |
| 检索文档 | 10-20% |
| 对话历史 | 30-50% |
| 工具输出 | 20-40% |
| 预留缓冲 | 5-10% |

**触发式优化而非周期性优化**——当某个类别超预算或总利用率 >70% 时触发，而不是每 N 轮无脑压一次。

---

## 对我们最实用的三条

1. **结构化压缩**：以后 compaction 发生时，用强制字段格式而非自由摘要，防止丢文件路径/决策/待办
2. **工具输出压缩**：3 轮以上的老输出、重复样板，主动替换成 `[Obs:... 要点:...]` 引用
3. **拆分前算账**：派多个分身之前，先估一下协调开销能否回本——少于 3 个独立子任务时优先单 agent 做
