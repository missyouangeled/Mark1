# 工程方法论（改编自 obra/superpowers）

> 当进行以下工作前，必须遵守本流程：修改脚本/补丁、新建项目、调整基础设施、大改配置。
> 纯日常聊天和简单问答不适用。

## 五阶段工作流

```
脑暴设计 → 任务拆解 → 分身执行 → 验证闭环 → 收尾清理
```

---

## 阶段 1：脑暴设计（STOP & THINK）

**硬规则：设计未批准前，禁止动手。**

哪怕看起来再简单——一个补丁脚本、一条 systemd timer、一个配置修改——先停下来想清楚。

### 流程

1. **探查现状**：查现有代码/配置/脚本/文档，理解当前状态
2. **一次一个问题**：不要连珠炮提问，每次只问一个，把意图/约束/成功标准搞清楚
3. **给出 2-3 个方案**：各有优劣，附推荐理由
4. **呈现设计**：分段展示，每段确认。覆盖：做什么、怎么做、涉及哪些文件、怎么验证
5. **写入设计文档**：`docs/plans/YYYY-MM-DD-<主题>.md`
6. **自审**：扫描 TODO/空位/矛盾/范围过大 → 当场修
7. **用户审阅**：展示设计文档路径，等确认

### 反模式

"这个太简单了，不需要设计"→ 越简单越容易因未检查假设而浪费时间。简单项目可以只写几行设计要点，但**不能不写**。

## 阶段 2：任务拆解（PLAN）

### 原则

- **每步 2-5 分钟能完成**——不是"写认证模块"，而是"创建 `auth.py` 骨架"
- **零上下文假设**——未来执行的分身可能不记得项目背景
- **精确文件路径**——不要"改 auth 相关文件"，写 `scripts/auth.py:45-60`
- **具体验证命令**——不是"检查是否工作"，写 `python3 scripts/test.py && echo PASS`
- **禁止占位符**——不允许 TODO / "后续补充" / "类似上一步"

### 计划格式

```markdown
# [项目名] 实施计划

**目标**: 一句话
**涉及文件**: 逐一列出
**验证方式**: 具体命令

## 任务 1: [2-5 分钟]

- [ ] 步骤 1：创建 `exact/path.py`，内容如下：
  ```python
  ...
  ```
- [ ] 步骤 2：验证：`python3 exact/path.py` 应输出 `...`
- [ ] 步骤 3：提交

## 任务 2: [2-5 分钟]
...
```

### 自审清单（写完后）

1. 每项规格要求都有对应任务吗？
2. 有 TODO / "后续" / "类似上一步" 吗？
3. 后续任务引用的类型/函数/变量在前面的任务里定义了吗？

## 阶段 3：分身执行（EXECUTE）

### 模型选择策略

| 任务复杂度 | 用哪个 |
|-----------|--------|
| 单文件、有完整规格 | 快模型（默认 `nvidia/z-ai/glm-5.1`） |
| 多文件协调、需要判断 | 标准模型（`deepseek/deepseek-chat`） |
| 架构设计/审查/排错 | 最强模型（`deepseek/deepseek-v4-pro`） |

### 执行流程

1. **读计划** → 提取所有任务文本和上下文
2. **批量分批** → 每批 3-5 个任务，顺序执行（规则详见下方「分批执行规则」）
3. **逐个派发** → 每个任务一个干净分身（`sessions_spawn(mode:"run", context:"isolated")`）
4. **两级审查**（由主会话执行，不另派审查分身）：
   - 第一级：规格审查——代码是否符合任务描述？多了还是少了？
   - 第二级：质量审查——命名、结构、边界情况
4. **不通过 = 退回重修** → 同一个分身修 → 重新审查 → 直到通过
5. **连续执行** → 不要在任务之间停下来问"要继续吗"，除非遇到 BLOCKED

### 分身状态处理

| 状态 | 行动 |
|------|------|
| **DONE** | 进入审查 |
| **DONE_WITH_CONCERNS** | 读完顾虑再审查 |
| **NEEDS_CONTEXT** | 补充上下文，重新派发 |
| **BLOCKED** | 分析原因：加上下文/换更强模型/拆更小任务/改计划 |
| **用户要求中断/暂停** | ①立刻kill相关任务分身 ②保留已完成中间产物 ③向主会话报告：哪些完成、哪些中断、产物在哪 ④监工保持 `taskActive=true` 等待后续指示 |

### 分批执行规则

multi-agent-patterns.md 指出超过3-5个agent后收益递减。因此：

- 总任务数 ≤5 → 全部并行派发
- 总任务数 6-10 → 分 2 批顺序执行（前一批全通过再派下一批）
- 总任务数 >10 → 每批 3-5 个，顺序执行；不得一次派发超过 5 个分身
- 从第二批起，写计划时就将有依赖关系的任务编排到同一批或相邻批次

## 阶段 4：验证闭环（VERIFY）

### 铁律

```
没有跑过验证命令的输出，就不能声称通过。
```

### 门控函数

```
声称任何状态之前：
1. 确认：哪个命令能证明这个声称？
2. 执行：完整运行这个命令
3. 读取：看输出、查退出码、数失败数
4. 核实：输出是否真的支持这个声称？
   - 不支持 → 如实报告实际状态并附证据
   - 支持 → 报告状态同时附证据
5. 只有这时：才能声称结果
```

### 常见自欺

| 声称 | 必须提供 | 不算数 |
|------|---------|--------|
| "通过了" | 命令输出：0 失败 | 上一次跑的 / "应该没问题" |
| "修好了" | 复现原始问题的测试：通过 | 改了代码 / 假设修好了 |
| "验收通过" | 逐项勾对检查清单 | "看起来差不多" |
| 分身说"成功了" | **自己验证一遍** | 分身自报结果 |

**红牌词**（说了就停）："应该行了" "看起来没问题" "大概率" "上次跑过" "应该不会" "感觉对了"

---

## 阶段 5：收尾清理（FINISH）

### 全部完成之后

1. **验证汇总**：跑一遍全部测试/检查
2. **提交推送**：`git add → commit → push`
3. **变更记录**：`python3 scripts/openclaw-change-log.py capture ...`
4. **清理分身**：关闭已完成的子 agent，释放资源
5. **更新文档**：补丁注册表 / 重建清单 / TOOLS.md 按需更新

### 不在主分支直接改

- 重大改动优先用 git 分支 + 合并
- 小修小补在主分支做但确保 `git pull --ff-only` 先同步

## 系统级任务特殊适配

上述方法论针对代码/脚本开发，但我们还做大量系统运维。系统级任务补充规则：

| 环节 | 代码任务 | 系统任务 |
|------|---------|---------|
| **验证前** | 跑测试 | `systemctl --user status xxx` + `--dry-run` 预览 |
| **执行后** | 测试通过 | `systemctl --user start xxx` + `journalctl --user -u xxx -n 10` 确认 |
| **"测试"** | TDD 红绿循环 | 先 `--dry-run` 看预期行为 → 实际执行 → 确认效果 |
| **回滚** | `git revert` | 备份原配置（`cp xxx.service xxx.service.bak`）→ 出了问题 `cp xxx.service.bak xxx.service` → `systemctl --user daemon-reload` |
| **验收** | 逐项勾对 | `systemctl --user is-active xxx` / URL 可达性 / 日志无 FATAL |

## 🎯 监工系统 × 方法论：完整配合机制

### 架构概览

```
┌──────────────────────────────────────────────────────┐
│                    监工服务 (底座)                      │
│  scripts/openclaw-supervisor-status.py                │
│  策略: auto|force_on|force_off + taskActive=true|false │
│  职责: 全局状态追踪 · 3 分钟回报 · 异常检测             │
└──────────────┬───────────────────────────────────────┘
               │ 盯住 ↓
┌──────────────┴──────────────┬────────────────────────┐
│  方法论：五阶段执行纪律        │  main-supervisor-lite   │
│  脑暴 → 拆解 → 执行 → 验证 → 收尾 │  按需拉起 · 补位协作      │
│  每阶段有硬规则、有输入输出标准   │  兜底回报 · 异常接管      │
└─────────────────────────────┴────────────────────────┘
```

两套体系的分工：
- **方法论** = 做事纪律（怎么把活干好）
- **监工服务** = 安全网（活干得对不对、人找不找得到）
- **监工分身** = 桥梁（真需要插播/协作时补位，平常不占资源）

---

### 五阶段 × 监工行为联动表

| 阶段 | 监工服务状态 | 监工分身 | 此行此刻监工在做什么 |
|------|-------------|---------|-------------------|
| ① 脑暴设计 | `auto + taskActive=true` | 可不拉 | 记录"进入脑暴"，若设计讨论超 3 分钟未进展→补一句提示 |
| ② 任务拆解 | `auto + taskActive=true` | 可不拉 | 等待计划产出；若拆解超时→询问是否卡住 |
| ③ 分身执行 | `auto + taskActive=true` | **必须拉**（有普通分身时） | 盯每个分身状态：DONE→正常 / BLOCKED→接管报告 / 空返回→告警 |
| ④ 验证闭环 | `auto + taskActive=true` | 保留在位 | 见证验证结果；若验证命令跑崩→补报异常 |
| ⑤ 收尾清理 | 完成后切回 `auto + taskActive=false` | 关闭（无后台任务 10 分钟后） | 确认分身关闭、git 已推送、变更流水已写 → 自身退回待命 |

---

### 阶段 ③ 细节：分身执行期监工如何配合

这是方法论和监工配合最紧密的阶段：

```
派发任务分身 (sessions_spawn)
      │
      ▼
监工: 记录 activeRun, taskActive=true ✅
      │
      ├─ 分身正常执行中...
      │    └─ 监工: 每 3 分钟检查一次，有进度→静默 / 无进度→补: "任务仍在执行，当前阶段: xxx"
      │
      ├─ 分身返回 DONE
      │    └─ 监工: 记录完成，等待主会话启动两级审查
      │
      ├─ 分身返回 BLOCKED
      │    └─ 监工: 立刻向主会话报告 "任务 xxx 阻塞，原因: ..."
      │         主会话介入：补充上下文 / 换模型 / 拆更小任务 / 改计划
      │
      ├─ 分身空返回 / 异常退出
      │    └─ 监工: 立刻向主会话报告 "⚠️ 任务 xxx 异常结束（空返回/超时/被杀）"
      │         主会话接管：检查日志 → 决定重试 or 手动修复
      │
      └─ 两级审查未通过
           └─ 监工: 记录审查结果，等待重修
                重修后重新审查 → 通过 → 标记任务完成
```

**关键规则**：
- 只要存在普通任务分身且有前台协作/插播需要，`main-supervisor-lite` 必须在位
- 监工发现异常时，**先报告再处理**，不要默默修
- 正常流程中监工保持静默，不刷屏不打扰

### 监工命令速查

```bash
# 进入复杂工程模式（方法论阶段 ①-⑤ 全程保持）
python3 scripts/openclaw-supervisor-status.py \
  --set-policy-mode auto --activate-task \
  --reason 'methodology-stage-1-brainstorm' --print-human

# 查看当前监工状态
python3 scripts/openclaw-supervisor-status.py --print-human

# 任务完成，恢复日常模式
python3 scripts/openclaw-supervisor-status.py \
  --set-policy-mode auto --deactivate-task \
  --reason 'methodology-complete' --print-human

# 用户强行开启（最高优先）
python3 scripts/openclaw-supervisor-status.py \
  --set-policy-mode force_on --reason 'user-request' --print-human

# 用户强行关闭
python3 scripts/openclaw-supervisor-status.py \
  --set-policy-mode force_off --reason 'user-request' --print-human
```

### 阶段边界动作模板

**进入方法论（收到修改诉求后）**：
1. 短反馈："好的，让我先想想。"
2. 切监工：`--set-policy-mode auto --activate-task --reason '...'`
3. 开始脑暴设计

**退出方法论（全部完成后）**：
1. 验证汇总跑完
2. git push 成功
3. 变更流水已写
4. 清理分身：关闭普通任务分身
5. 切监工：`--set-policy-mode auto --deactivate-task --reason 'methodology-complete'`
   ——手动 `--deactivate-task` 是主动收口，确保验证已跑完才退出
   ——若仍有后台任务在跑 → 不要切，等任务完成再切
6. 向用户报告："全部完成。监工已退回待命。"

（10 分钟无新后台任务自动降回 `taskActive=false` 是兜底安全网——万一忘了手动切，不会永远挂住。手动切的优先级高于自动降。）

## 快速参考卡片

```
收到修改诉求 →
  短反馈 + 切监工(auto+taskActive) →
  先查现有能力 →
  再查冲突边界 →
  脑暴设计（写文档）→
  用户批准 →
  拆任务（写计划）→
  分批执行(按规则) + 拉监工分身(需协作/插播时) →
  验证闭环（看证据）→
  提交推送 →
  变更流水 →
  关任务分身 → 收尾清理 →
  切监工(auto+taskActive=false)
```
