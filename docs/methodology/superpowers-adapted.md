# 工程方法论（改编自 obra/superpowers）

> 当进行以下工作前，必须遵守本流程：修改脚本/补丁、新建项目、调整基础设施、大改配置。
> 纯日常聊天和简单问答不适用。

## 五阶段工作流

```
脑暴设计 → 任务拆解 → 分身执行 → 验证闭环 → 收尾清理
```

---

## 阶段 1：脑暴设计（STOP & THINK）

**硬规则：设计未批准前，禁止动手。**

哪怕看起来再简单——一个补丁脚本、一条 systemd timer、一个配置修改——先停下来想清楚。

### 流程

1. **探查现状**：查现有代码/配置/脚本/文档，理解当前状态
2. **一次一个问题**：不要连珠炮提问，每次只问一个，把意图/约束/成功标准搞清楚
3. **给出 2-3 个方案**：各有优劣，附推荐理由
4. **呈现设计**：分段展示，每段确认。覆盖：做什么、怎么做、涉及哪些文件、怎么验证
5. **写入设计文档**：`docs/plans/YYYY-MM-DD-<主题>.md`
6. **自审**：扫描 TODO/空位/矛盾/范围过大 → 当场修
7. **用户审阅**：展示设计文档路径，等确认

### 反模式

"这个太简单了，不需要设计"→ 越简单越容易因未检查假设而浪费时间。简单项目可以只写几行设计要点，但**不能不写**。

---

## 阶段 2：任务拆解（PLAN）

### 原则

- **每步 2-5 分钟能完成**——不是"写认证模块"，而是"创建 `auth.py` 骨架"
- **零上下文假设**——未来执行的分身可能不记得项目背景
- **精确文件路径**——不要"改 auth 相关文件"，写 `scripts/auth.py:45-60`
- **具体验证命令**——不是"检查是否工作"，写 `python3 scripts/test.py && echo PASS`
- **禁止占位符**——不允许 TODO / "后续补充" / "类似上一步"

### 计划格式

```markdown
# [项目名] 实施计划

**目标**: 一句话
**涉及文件**: 逐一列出
**验证方式**: 具体命令

## 任务 1: [2-5 分钟]

- [ ] 步骤 1：创建 `exact/path.py`，内容如下：
  ```python
  ...
  ```
- [ ] 步骤 2：验证：`python3 exact/path.py` 应输出 `...`
- [ ] 步骤 3：提交

## 任务 2: [2-5 分钟]
...
```

### 自审清单（写完后）

1. 每项规格要求都有对应任务吗？
2. 有 TODO / "后续" / "类似上一步" 吗？
3. 后续任务引用的类型/函数/变量在前面的任务里定义了吗？

## 阶段 3：分身执行（EXECUTE）

### 模型选择策略

| 任务复杂度 | 用哪个 |
|-----------|--------|
| 单文件、有完整规格 | 快模型（默认 `nvidia/z-ai/glm-5.1`） |
| 多文件协调、需要判断 | 标准模型（`deepseek/deepseek-chat`） |
| 架构设计/审查/排错 | 最强模型（`deepseek/deepseek-v4-pro`） |

### 执行流程

1. **读计划** → 提取所有任务文本和上下文
2. **逐个派发** → 每个任务一个干净分身（`sessions_spawn(mode:"run", context:"isolated")`）
3. **两级审查**：
   - 第一级：规格审查——代码是否符合任务描述？多了还是少了？
   - 第二级：质量审查——命名、结构、边界情况
4. **不通过 = 退回重修** → 同一个分身修 → 重新审查 → 直到通过
5. **连续执行** → 不要在任务之间停下来问"要继续吗"，除非遇到 BLOCKED

### 分身状态处理

| 状态 | 行动 |
|------|------|
| **DONE** | 进入审查 |
| **DONE_WITH_CONCERNS** | 读完顾虑再审查 |
| **NEEDS_CONTEXT** | 补充上下文，重新派发 |
| **BLOCKED** | 分析原因：加上下文/换更强模型/拆更小任务/改计划 |

## 阶段 4：验证闭环（VERIFY）

### 铁律

```
没有跑过验证命令的输出，就不能声称通过。
```

### 门控函数

```
声称任何状态之前：
1. 确认：哪个命令能证明这个声称？
2. 执行：完整运行这个命令
3. 读取：看输出、查退出码、数失败数
4. 核实：输出是否真的支持这个声称？
   - 不支持 → 如实报告实际状态并附证据
   - 支持 → 报告状态同时附证据
5. 只有这时：才能声称结果
```

### 常见自欺

| 声称 | 必须提供 | 不算数 |
|------|---------|--------|
| "通过了" | 命令输出：0 失败 | 上一次跑的 / "应该没问题" |
| "修好了" | 复现原始问题的测试：通过 | 改了代码 / 假设修好了 |
| "验收通过" | 逐项勾对检查清单 | "看起来差不多" |
| 分身说"成功了" | **自己验证一遍** | 分身自报结果 |

**红牌词**（说了就停）："应该行了" "看起来没问题" "大概率" "上次跑过" "应该不会" "感觉对了"

## 阶段 5：收尾清理（FINISH）

### 全部完成之后

1. **验证汇总**：跑一遍全部测试/检查
2. **提交推送**：`git add → commit → push`
3. **变更记录**：`python3 scripts/openclaw-change-log.py capture ...`
4. **清理分身**：关闭已完成的子 agent，释放资源
5. **更新文档**：补丁注册表 / 重建清单 / TOOLS.md 按需更新

### 不在主分支直接改

- 重大改动优先用 git 分支 + 合并
- 小修小补在主分支做但确保 `git pull --ff-only` 先同步

## 系统级任务特殊适配

上述方法论针对代码/脚本开发，但我们还做大量系统运维。系统级任务补充规则：

| 环节 | 代码任务 | 系统任务 |
|------|---------|---------|
| **验证前** | 跑测试 | `systemctl --user status xxx` + `--dry-run` 预览 |
| **执行后** | 测试通过 | `systemctl --user start xxx` + `journalctl --user -u xxx -n 10` 确认 |
| **"测试"** | TDD 红绿循环 | 先 `--dry-run` 看预期行为 → 实际执行 → 确认效果 |
| **回滚** | `git revert` | 备份原配置（`cp xxx.service xxx.service.bak`）→ 出了问题 `cp xxx.service.bak xxx.service` → `systemctl --user daemon-reload` |
| **验收** | 逐项勾对 | `systemctl --user is-active xxx` / URL 可达性 / 日志无 FATAL |

## 与监工系统的关系

本方法论与现有监工/分身体系**互补不冲突**：

- **监工服务**（`main-supervisor-lite`）：全局兜底，复杂工程必须在线
- **本方法论**：给出"在监工注视下，每个任务如何执行"的纪律
- **任务分身**：按照本方法论的两级审查来质量把关
- **3 分钟回报**：监工链路兜底，方法论正常执行时不会触发
