## 三、跨会话上下文恢复链

```
会话压缩/重启/切模型
        │
        ▼
  BOOT_INDEX.md 第 2 步
        │
        ├──→ memory/daily/2026-06-10.md (今日摘要)
        │     └─ 含「今日摘要」+「重要事项」区块
        │     └─ 含 Agnes API、resume-watch 等结论
        │
        └──→ /mnt/data/openclaw/session-backup/latest/
              ├─ context-summary.md
              │   ├─ 最近规则/偏好 (从 MEMORY.md 提取)
              │   ├─ daily.md 正文摘要 (800字)
              │   └─ transcript 最后 200 行 (约1小时对话)
              ├─ MEMORY.md (备份副本)
              ├─ SOUL.md / USER.md (备份副本)
              └─ daily-*.md (备份副本)
        │
        ▼
  AI 醒来 → 读 context-summary → 找回上下文 → 不再失忆

  保障机制:
  · session-backup timer: 每10分钟快照 (7天保留·已修复)
  · lifecycle-maintainer: 每日自动 clean --days 7
  · 独立数据盘: 不依赖 workspace / git
```

## 八、总结

这套系统经过约 47 天的持续迭代（从 4/29 第一条 daily 记录起），已从简单的 AI 聊天助手进化为一套**多层级、自监控、自修复的 AI 操作系统**。

核心优势：
- **架构自觉**：启动链 + 规则路由 + 上下文恢复，任何模型/任何重启后都能正确接入
- **自动化密度极高**：6 个 systemd 定时器 + 35 个核心脚本覆盖了几乎所有的运维场景
- **分级治理**：从 818 行的 4 域规则到 28K 行的脚本，每层都有明确的职责边界
- **容错设计**：双盘隔离、数据盘备份、独立上下文恢复链

主要短板：
- 部分自动化组件稳定性仍需打磨（resume-watch 今日刚修，frontstage-guardian 仍 failed）
- 多机间的协同仍偏手工
- 根盘空间需要持续关注

**一句话概括**：一个正在从"能用"走向"稳定可靠"的 AI 管家系统，今天（6/10）完成了关键的上下文恢复链路修复，是一个重要的成熟度里程碑。
