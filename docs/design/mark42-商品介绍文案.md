# Mark42 — 模块化智能铠甲系统

> **给 AI Agent 穿上战甲。**
> 上下文不再悄悄炸掉，大任务不再卡死主线程，重复的事自己转起来。

---

## 这是什么

Mark42 是一套为 AI Agent（当前为 OpenClaw）设计的**模块化运行时铠甲系统**。

它不是聊天插件的堆叠，而是一套有架构、有纪律、有自我意识的基础设施——让 AI Agent 在长时间运行中**保持稳定、保持清醒、保持可控**。

灵感来自钢铁侠的 Mark42 战甲：**每一块拆下来能独立战斗，拼在一起是一套完整的甲。**

---

## 它解决什么问题

AI Agent 跑久了，有三件事一定会发生：

| 问题 | 没有 Mark42 时 | 有 Mark42 后 |
|---|---|---|
| **上下文爆炸** | 聊着聊着上下文撑满，模型开始遗忘、幻觉、答非所问，用户只能手动 `/compact` | 铠甲持续监控，到阈值自动压缩，压缩前先提取记忆索引注入下轮对话 |
| **重复任务靠人盯** | 下载转码、状态巡检、模型切换……每个都要人盯着等结果 | 循环引擎把任务编成 Loop（观察→决策→执行→验证），自动跑、自动重试、自动汇报 |
| **大工程卡死前台** | 跑一批 100 个文件的重命名，主会话卡住，上下文被污染 | 重型战甲把大任务拆成子任务丢到后台隔离执行，每个分身只回报摘要 |

## 三层架构

### 🛡️ 上下文铠甲（Context Armor）

**别让上下文在你眼皮底下悄悄炸掉。**

- 实时检测上下文使用率，分级预警（70%→准备 / 85%→压缩 / 95%→紧急分区）
- 智能诊断退化类型：Lost-in-Middle / Poisoning / Distraction / Confusion / Clash
- 压缩前先提取「记忆索引」——保留用户偏好、活跃决策、任务进度，丢掉闲聊和死胡同
- 每次出手写入审计日志，丢了什么、为什么丢，有据可查
- PII 脱敏：手机号、身份证、邮箱、API Key、IP 地址、敏感路径——压缩前自动脱掉

### 🔄 循环引擎（Loop Engine）

**别再让我盯着屏幕等结果。**

把任何重复性任务编成标准 Loop：

```
观察 → 决策 → 执行 → 验证 → 回到观察
```

- 4 个内置模板：上下文守护 / 健康监控 / 模型切换 / 记忆索引
- 支持自定义 Loop：指定观察命令、决策逻辑、执行动作、验证方式
- 嵌套并行：大 Loop 套小 Loop，多个独立 Loop 同时跑
- daemon 守护 + watchdog 保活，挂了自动拉起

### ⚙️ 重型战甲（Heavy Task Suit）

**大工程不卡主线程、不炸上下文、不掉结果。**

- 开工前预检：文件量、内存、磁盘、失败服务全扫描
- 上下文感知：启动前先查铠甲余量，空间不够先压缩
- 子任务隔离：每个子任务在独立上下文中执行，只回报摘要（<500 tokens）
- scratch 空间隔离：每个任务有独立的输出/错误/状态目录
- 全程监控：通过循环引擎的 task-watch 模板持续盯进度

## 战甲意识层（v3 新增）

Mark42 不只是工具，它有「意识」——一套自我感知和主动交流的能力：

### 八核心位注册表

| 核心 | 能力 | 当前实现 |
|---|---|---|
| 1 主意识 | 对话主模型 | MiniMax-M3 |
| 2 战甲意识 | 自检+评估+决策 | Agnes 2.0 flash |
| 3 向量引擎 | 语义检索 | MiniLM-L12-v2 |
| 4 文本压缩 | 同义词+填充词压缩 | TextCompressor + LLM |
| 5 代码理解 | 代码分析+Bug检测 | GLM-5.2 via API |
| 6 日志分类 | 日志级别+动作建议 | 关键词规则引擎 |
| 7 PII 脱敏 | 隐私信息识别 | 正则+Luhn校验 |
| 8 异常检测 | 系统指标监控 | 阈值+Z-Score |

8 核全部健康，每个核心可独立探活、隔离、替换。

### C1-C5 意识决策链

当系统出现异常，意识层会走一条完整的决策链：

```
C1 自检 → C2 评估确定性 → C3 自动修复 / C4 主动对话 → C5 档案查询
```

- **C1**：每 30 秒自检 8 核心 + 4 Loop + 系统资源
- **C2**：根据规则表评估异常的确定性——100% 确定 / 高 / 低 / 未知
- **C3**：100% 确定的异常自动修复（调铠甲压缩 / 重启进程 / 重建索引 / 重新注册 Loop）
- **C4**：不确定的异常**先问 Advisor（外部 LLM）**，拿到 approve/modify/reject 判定后再决定
- **C5**：查错误档案，如果是已知问题且有批准的修复方案，直接复用

### 主动交流协议（Advisor）

这是 Mark42 最有意思的设计：**战甲不确定时，会主动找另一个 AI 讨论**。

- 4 类场景：新异常 / 不确定问题 / 修复方案评估 / 历史问题复用
- Advisor 返回结构化判定：verdict（approve/modify/reject）+ confidence + reasoning
- 置信度够高 → 自动执行；不够 → 问用户
- 当前 Advisor：GLM-5.2（火山方舟 Agent Plan）

### 熔断器（Circuit Breaker）

某个核心连续失败 3 次后自动断路，不再反复重试浪费时间：

- 断路后直接走 fallback（<1 秒）
- 30 秒后半开试探，好了就恢复
- 每个核心独立熔断，互不影响

### 错误档案

每次异常自动归档，形成知识库：

- 同一问题第二次出现 → 直接查档案复用方案
- 档案有 4 状态：NEW（新问题）/ APPROVED（已批准自动修复）/ REJECTED（已否决）/ RESOLVED（已解决）
- 档案可追溯：什么时候第一次出现、出现了几次、上次怎么修的

## 工程治理

### R1-R14 设计原则

Mark42 由 14 条设计原则支撑，核心包括：

- **R1 可插拔**：所有模型可替换，改配置不改代码
- **R7 有据可查**：所有自动行为写日志，换了模型也能复现
- **R9 强制读协议**：模型上岗前先考试（10 题答对 2 题算通过）
- **R11 混沌工程**：4 个故障注入场景，验证故障检测能力
- **R14 集群思维**：核心可一键替换，从备份或 Git 回滚

### 模块级协议

每个模块有三态健康检查：

| 状态 | 含义 |
|---|---|
| 🟢 Green | 能用，延迟正常 |
| 🟡 Yellow | 维修中，已降级但还在跑 |
| 🔴 Red | 不能用，走 fallback |

4 Golden Signals 监控：延迟 / 错误率 / 饱和度 / 流量。触线自动降级。

## 弹性伸缩

Mark42 的核心哲学是**弹性伸缩**——从树莓派到云端服务器，同一套代码不需要改一行：

```
┌─────────────────────────────────────────────┐
│  L3 重装模式 — 云端 / 好设备                  │
│  对话 + 铠甲 + 引擎 + 战甲 + 八核 + Advisor    │
├─────────────────────────────────────────────┤
│  L2 标准模式 — 中等设备                       │
│  对话 + 铠甲 + 引擎 + 战甲                     │
├─────────────────────────────────────────────┤
│  L1 极简模式 — 树莓派 / 旧手机                 │
│  只要对话，别的都能关                           │
└─────────────────────────────────────────────┘
```

接口不变，挂载配置驱动。设备差就少挂几个模块，设备好就全挂上。

## 一键启动

```bash
# 一键启动完整战甲
python3 scripts/mark42.py assemble

# 等同于同时启动：
#   mark42.py armor --guard        （上下文守护）
#   mark42.py engine --daemon      （循环引擎）
#   （重型战甲按需激活）
```

每个模块也可以独立使用：

```bash
# 只查上下文健康
python3 scripts/mark42.py armor --check

# 只看活跃 Loop
python3 scripts/mark42.py engine --list

# 只看核心位状态
python3 scripts/mark42.py cores list

# 只看熔断器
python3 scripts/mark42.py breaker list
```

## 技术栈

- **语言**：Python 3.12（纯 Python，零编译依赖）
- **LLM 接入**：可插拔设计，支持 Stub / API / Ollama 三种 runtime
- **向量引擎**：sentence-transformers MiniLM-L12-v2（本地 Embedding）
- **守护进程**：systemd user units
- **配置**：YAML + 环境变量，XDG 标准路径
- **测试**：pytest + unittest，45% 覆盖率（持续提升中）
