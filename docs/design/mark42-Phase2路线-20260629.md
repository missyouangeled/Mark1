## 二、统一模式分析（关键！）

**8 个压缩模块的公共模式**：

```python
# 1. 工厂函数（懒加载）
def get_X() -> XClass:
    global _X
    if _X is None:
        _X = XClass()
    return _X

# 2. 包装函数（标准接口）
def Xcompress(content: str) -> tuple[str, dict]:
    """压缩 + 返回元数据"""
    comp = get_X()
    return comp.compress(content)
    # 或: comp.compress(content, ...) + 返回 (str, dict)

# 3. 烟测函数（开发期）
def _run_tests() -> bool:
    """跑烟测（手动调）"""
    ...
```

**测试设计含义**：
- **统一测 2 个公共函数**：`get_X()` 工厂 + `Xcompress()` 包装
- **不要测 `_run_tests()`**（那是手测工具）
- **不要测 `XClass.compress()` 内部实现**（太细，破坏性高）
- 测**契约**：输入字符串 → 输出 `(压缩后, 元数据 dict)`

## 五、关键挑战 + 解决

### 挑战 1：算法依赖链

`smart_crusher` 依赖 `llm_text_compressor` + `text_compressor` + `compression_algorithms`。
**问题**：测 smart_crusher 时要不要全 mock 掉底层？

**解决**：只 mock 顶层函数（如 `llm_text_compress`），不 mock 类内部。这样：
- 不破坏智能降级逻辑
- 跑得快
- 测试聚焦于 smart_crusher 的调度决策

### 挑战 2：LLM API mock

`llm_text_compressor` 真调 LLM API（OpenAI/Agnes/...）。
**问题**：测试不调真 API。

**解决**：用 `mocker.patch("openai.OpenAI")` 或 LiteLLM 入口。
**必须**：`mocker.patch.dict(os.environ, {"OPENAI_API_KEY": "fake"})` 防止导入时崩溃。

### 挑战 3：fcntl 文件锁

`logs.py` + `engine.py` 用 fcntl（Phase 1 已知）。
**问题**：fcntl 在 mock 上下文里失败。

**解决**：在 conftest 顶层加 `pytest.fixture(autouse=True)` 跳过 fcntl：
```python
@pytest.fixture(autouse=True)
def _skip_fcntl_on_mock_files(monkeypatch):
    """测试期间禁用 fcntl.flock。"""
    import fcntl
    monkeypatch.setattr(fcntl, "flock", lambda *a, **k: None)
```

**⚠️ 注意**：这会改变实际行为，**不能**在 `compaction_apply` 这种"真改配置"测试里用。

### 挑战 4：compaction_apply 真改配置

`compaction_apply` 会改 `~/.openclaw/openclaw.json`。
**问题**：测试可能污染真配置。

**解决**：
1. 给 `compaction_apply` 测试加 `"runslow"` 标记，默认跳过
2. 跑这些测试时用 `--runslow` + 真临时 HOME
3. 或完全 mock `openclaw.json` 读写

### 挑战 5：LLM 调用超时

异步 LLM 测试可能卡住（连不上真 API）。
**解决**：用 `pytest-timeout` 插件，所有异步测试加 `@pytest.mark.timeout(5)`。
