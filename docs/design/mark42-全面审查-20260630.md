## 重大问题 (🔴 1)

### 🔴-1 `heavy_execute` 是"假执行",不是真自动执行

**文件**: `scripts/mark42_modules/heavy.py` line 230-280

**症状**: `heavy_execute()` 创建执行脚本 `task_dir/{batch_id}-exec.sh`,但**从不自动调用 `bash {script_path}`**。脚本里只写 `# TODO: replace with actual file operation` 占位。

**对照设计**:
- 设计 4.2: "Heavy 战甲 --start 应该自动分批 + 后台执行"
- 实际: `heavy_start` 把 batch 入队 → `heavy_execute` 写脚本 → **等用户手动跑**

**影响**: `heavy_execute_all` 也只是循环写脚本,**没有任何 batch 被真执行**。这是"重型战甲"名不副实的核心问题。

**修法**(选一):
1. `heavy_execute` 末尾加 `subprocess.Popen(["bash", str(script_path)])` 启动后台执行
2. 或: 大工程只生成执行计划 + dry-run 报告,不做执行(明确"重型战甲 = 预检 + 计划,执行 = 用户")

**建议**: 选项 1(真执行),但加 `--dry-run` 标志默认 True 让用户确认后真跑。

## 中等问题 (🟠 3)

### 🟠-1 assemble() 启动的 2 个 daemon 与设计不符

**文件**: `scripts/mark42_modules/cli.py:67-105`

**设计文档** (mark42-架构设计.md 5.2) 说 `mark42.py assemble` "等同于同时启动 armor --guard & + engine --daemon &",**只 2 个守护**。但 **架构设计 0 节**说"四个守护服务(armor-guard / engine-daemon / memory-index / task-watch)"。

**实际 systemd 状态**:
- ✅ `mark42-armor-guard.service` active
- ✅ `mark42-engine-daemon.service` active
- ❌ `mark42-memory-index.service` **不存在**
- ❌ `mark42-task-watch.service` **不存在**

**真相**: `memory-index` 和 `task-watch` 不是独立 service,是**引擎内部的 Loop 模板**:
- `memory-index` Loop 由 `bootstrap.sh` 启动时注册
- `task-watch` Loop 由 `engine_daemon` 动态创建(检测 `heavy.task.started` 事件时)

**问题**: "4 个守护服务"的说法不准确,文档误导。

**修法**:
1. 更新架构设计文档,把"4 个守护"改为"2 个 daemon + 1 个 bootstrap(注册 Loop) + 1 个 watchdog(保活)"
2. 或: 补齐 `mark42-memory-index.service` / `mark42-task-watch.service` systemd unit(用 `Type=oneshot` + `RemainAfterExit=yes` 跑 bootstrap)

### 🟠-2 broker mark42-events.jsonl 接近 10MB 临界

**真生产状态**: `mark42-events.jsonl` 10485733 bytes = 9.99997 MB,**正好在 `MAX_BROKER_EVENTS_MB=10` 阈值边缘**。

`rotate_broker_events` 的条件是 `if size_mb <= MAX_BROKER_EVENTS_MB: return ...` — **`<=`** 不会裁!只有**超过** 10MB 才裁。

**修法**:
1. 把 `<=` 改为 `<` — 10MB 就裁,留安全余量
2. 或: 阈值改为 9MB,留 1MB buffer

### 🟠-3 actions.jsonl 不记录 P0 修复的 preBytes/postBytes 字段

**文件**: `scripts/mark42_modules/armor.py:526-533`

**问题**: P0 修复在 `memory-index.json` 里加了 `preCompactBytes/postCompactBytes/bytesSaved/compressionEffective` 字段,**但 actions.jsonl 仍然只记 `preCompressUsage`**,**没有 bytes 字段**。

**影响**:
- actions.jsonl 是"出手记录"主表,无法审计"压缩是不是真截短"
- 必须看 `memory-index.json` 才能知道有没有真截短,但 memory-index 每次被覆盖

**修法**: actions.jsonl 写入时也加 `preBytes/postBytes/bytesSaved/effective` 字段。

### 🟡-6 heavy.py `.keep` 命名反了

**文件**: `scripts/mark42_modules/heavy.py:184`

`heavy_start` 写 `(task_dir / ".keep").write_text("keep\n")`,但 `rotate_scratch_old` / `heavy_cleanup` 用 `.keep` 存在**作为"受保护"标记**。`heavy_cleanup` 用 `shutil.rmtree` 整个删,`.keep` 也被删。**下次清理时,`rotate_scratch_old` 因为没 `.keep` 标志而误判为"未受保护"**。

**修法**: `.keep` 命名合理(惯例:防止 git/sync 删空目录),但 `rotate_scratch_old` 逻辑检查 `.keep` 意义不大 — `keep` 应是"keep 整个目录"而不是"keep 文件本身"。建议:
- `heavy_cleanup` 改 `shutil.rmtree(task_dir, ignore_errors=True)` 然后**重建** `.keep` 占位空目录
- 或: `.keep` 改名 `.protected`
