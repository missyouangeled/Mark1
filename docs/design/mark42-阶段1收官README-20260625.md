## 一、Mark42 是什么

**Mark42** 是点点 (袁文涛) 设计的**模块化智能铠甲系统**, 分三层:

```
┌─────────────────────────────────────────┐
│ Armor 铠甲层 (最外层)                     │
│  - 上下文压缩 / LLM 智能分析              │
│  - Session 保护 / 自动收缩                │
│  - 🔄 内容类型自动路由 (Day 6)            │
│  - ⚡ 异步压缩队列 (Day 7)                 │
│  - 🧠 LLM 语义压缩 (Day 8)                │
├─────────────────────────────────────────┤
│ Engine 循环引擎 (中间层)                  │
│  - 5 个 Loop 注册:                        │
│    context-guard / health-watch /         │
│    memory-index / model-fallback /        │
│    task-watch                            │
│  - 同步 while+sleep 守护循环              │
├─────────────────────────────────────────┤
│ Heavy 重型战甲 (最内层)                   │
│  - 按需激活, 深度推理                     │
│  - 日常零成本, 仅复杂任务触发              │
└─────────────────────────────────────────┘
```

**核心目标**: 保护上下文不溢出, 同时最大化信息密度 + 最小化计算成本。

## 二、阶段 1 完整能力 (Day 1 → Day 8)

### 2.1 压缩子系统 — 5 算法 + 智能路由 + 异步队列 + LLM 接入

#### 5 个压缩算法 (按内容类型选最优)

| 算法 | 模块 | 适用 | 典型压缩率 | 状态 |
|------|------|------|-----------|------|
| **SmartCrusher** | compression_algorithms.py | JSON / 结构化 | 60-90% | ✅ Day 1 |
| **CodeCompressor** | code_compressor.py | 源码 (Python/JS) | 44% (保签名) | ✅ Day 6 |
| **LogDeduplicator** | log_deduplicator.py | 重复行日志 | 46% | ✅ Day 6 (从旧版拆分) |
| **DiffCompressor** | diff_compressor.py | git diff / 补丁 | 82% | ✅ Day 6 |
| **TextCompressor** | text_compressor.py | 长文本 (rule) | 12% | ✅ Day 6 |
| **LLMTextCompressor** | llm_text_compressor.py | 长文本 (语义) | **84-90%** | ✅ Day 8 |

**实测压缩效果** (5 段技术描述 × 5 重复, 4130B):
- rule_based: fallback (1.8% < 5% 阈值)
- **LLM: 648B (84.3% 压缩率, 4.4 秒)** — 强 47 倍

#### 智能路由 (Day 6) — 5 种内容类型自动嗅探

```
输入内容
   ↓
[嗅探] diff > code > log > text > JSON
   ↓
route_algo: "diff" | "code" | "log" | "text" | "smartcrush"
   ↓
按 route_algo 调对应算法
   ↓
护栏: ratio < 5% / 压后 > 95% 视为无效 → 回退原文
```

JSON 路由保留原 Day 3 契约 (compress+pii 行为不变), 不破坏旧测试。

#### 异步压缩队列 (Day 7) — daemon 永不阻塞

```
旧: armor_compress() 同步串行 5×N 秒 (阻塞 daemon tick)
新: armor_compress_async() 入队即返, worker pool 并行处理
```

- `CompressRequest` 数据封装 + `threading.Event` 同步等待
- `CompressQueue` 线程实现 (不依赖 asyncio, 兼容 OpenClaw sync daemon)
- 优先级队列: 紧急请求可挤掉低优先级
- 4 重降级: 算法异常 → worker 存活 → 队列有界 → 永不崩
- `get_compress_queue()` 单例 (懒启动), `daemon=True` 线程
- 6 个统计字段可观测: enqueued / processed / failed / dropped_queue_full / dropped_low_priority / active_workers

#### LLM 语义压缩 (Day 8) — 真接 LiteLLM

- 注册 `llmCompress` 到 Mark42 统一模型表 (与 `_llm_analyze` 同源)
- 3 种压缩模式: **summarize** (摘要) / **simplify** (简化) / **extract** (抽取)
- 真接 MiniMax-M3, 沿用 OpenClaw minimax provider 路由 — **零额外配置**
- 4 道护栏: `min_text_size=500B` / `max_input_bytes=12000B` / `min_useful_ratio=5%` / `max_useful_ratio=98%`
- 3 层降级: LLM 错 → rule_based text_compress → 原文透传
- 清理输出: 剥离 <think> 块 / markdown 包裹 / 空白
- 单例 + 函数式入口 `llm_text_compress()`

## 三、阶段 1 关键设计决策

### 3.1 内容类型嗅探 vs 大小分层

- 大小分层 (Day 3) 适合 JSON, 但代码 / diff / 日志 / 长文本**结构化特征比大小更重要**
- 内容类型嗅探 (Day 6) 在大小分层**之前**, 优先选最匹配的算法
- 兼容: 嗅探**只在非 JSON 场景触发**, 旧 JSON 契约完全保留

### 3.2 异步: 线程 vs asyncio

- OpenClaw engine daemon 是**同步 while+sleep 循环**, 非 asyncio
- Day 7 选 **threading + queue.Queue**:
  - 轻、与 sync daemon 兼容
  - 未来 OpenClaw 升级 asyncio 时, CompressQueue API 可无缝迁移
  - 保留 asyncio 风格 (enqueue / start / shutdown) 便于未来切换

### 3.3 LLM 默认 vs opt-in

- LLM 4 秒一次 + 1000 tokens 输入 + ~200 tokens 输出 = **不是免费的**
- Day 8 选 **opt-in** (text_compressor 显式 `method="llm"`):
  - scheduler 默认仍走 rule_based
  - 用户想用 LLM 时显式 opt-in
  - 未来通过环境变量 (见开发目标) 可让 scheduler 默认 LLM

### 3.4 4 层降级链

```
LLM 异常
  → LLM 失败 (3 层重试后放弃)
  → rule_based text_compress
  → 原文透传
  → 永不崩 (worker / daemon / 任何上层)
```

每层都验证 ratio 有效, 无效就降到下一层。

### 3.5 向后兼容

- `armor_compress()` / `armor_pre_compact_hook()` / `algo_scheduler.process()` 全部**签名不变**
- 唯一新增字段: `ScheduleDecision.route_algo` (老调用方不读就忽略)
- `MARK42_ALGO_USE_SCHEDULER=false` / `MARK42_ALGO_FAIL_SAFE` 等环境变量全保留

3. 加超时控制: 队列满 → 回退 rule_based; worker 异常 → fallback
4. 测试: 异步路径 + 同步路径两条都要测

#### 目标 2: MARK42_TEXT_USE_LLM 环境变量

**背景** (2026-06-25 用户要求):
> "MARK42_TEXT_USE_LLM 环境变量 (让 scheduler 也能默认走 LLM)"

**当前状态**:
- Day 6 scheduler 路由 text 走 `text_compress()` (默认 rule_based)
- 想用 LLM 必须显式 `TextCompressor(method="llm")`, scheduler 不支持
- 用户希望加个**全局开关** (env var), 让 scheduler 默认走 LLM

**目标**:
- 新增环境变量 `MARK42_TEXT_USE_LLM=true|false|auto`
  - `true` (默认关): scheduler 路由 text 时强制走 LLM
  - `false`: 维持现状 (rule_based)
  - `auto` (默认): scheduler 路由 text 时根据输入大小自动选 (如 > 5KB 走 LLM, < 5KB rule_based)
- 同时支持 `MARK42_LLM_MODE=summarize|simplify|extract` 选择 LLM 模式

**实施计划**:
1. `algo_scheduler.py` 顶部读 env var
2. 路由逻辑: text → 检查 `MARK42_TEXT_USE_LLM` → 决定调 `text_compress(method=...)` 还是 `llm_text_compress()`
3. 测试: env var 各种取值下的行为

**预期交付**:
- 改 `algo_scheduler.py` (~+20 行)
- 新测试 4+ 个 (3 种 env var 取值)
- 更新 `mark42-运维日志.md` (新 env var 文档)
