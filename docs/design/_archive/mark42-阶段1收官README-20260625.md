## 一、Mark42 是什么

**Mark42** 是点点 (袁文涛) 设计的**模块化智能铠甲系统**, 分三层:

```
┌─────────────────────────────────────────┐
│ Armor 铠甲层 (最外层)                     │
│  - 上下文压缩 / LLM 智能分析              │
│  - Session 保护 / 自动收缩                │
│  - 🔄 内容类型自动路由 (Day 6)            │
│  - ⚡ 异步压缩队列 (Day 7)                 │
│  - 🧠 LLM 语义压缩 (Day 8)                │
├─────────────────────────────────────────┤
│ Engine 循环引擎 (中间层)                  │
│  - 5 个 Loop 注册:                        │
│    context-guard / health-watch /         │
│    memory-index / model-fallback /        │
│    task-watch                            │
│  - 同步 while+sleep 守护循环              │
├─────────────────────────────────────────┤
│ Heavy 重型战甲 (最内层)                   │
│  - 按需激活, 深度推理                     │
│  - 日常零成本, 仅复杂任务触发              │
└─────────────────────────────────────────┘
```

**核心目标**: 保护上下文不溢出, 同时最大化信息密度 + 最小化计算成本。

```
输入内容
   ↓

   ↓
route_algo: "diff" | "code" | "log" | "text" | "smartcrush"

#### 异步压缩队列 (Day 7) — daemon 永不阻塞

```
旧: armor_compress() 同步串行 5×N 秒 (阻塞 daemon tick)
新: armor_compress_async() 入队即返, worker pool 并行处理

- 6 个统计字段可观测: enqueued / processed / failed / dropped_queue_full / dropped_low_priority / active_workers

## 三、阶段 1 关键设计决策

### 3.1 内容类型嗅探 vs 大小分层

- 大小分层 (Day 3) 适合 JSON, 但代码 / diff / 日志 / 长文本**结构化特征比大小更重要**
- 内容类型嗅探 (Day 6) 在大小分层**之前**, 优先选最匹配的算法
- 兼容: 嗅探**只在非 JSON 场景触发**, 旧 JSON 契约完全保留

### 3.2 异步: 线程 vs asyncio

- OpenClaw engine daemon 是**同步 while+sleep 循环**, 非 asyncio
- Day 7 选 **threading + queue.Queue**:
  - 轻、与 sync daemon 兼容
  - 未来 OpenClaw 升级 asyncio 时, CompressQueue API 可无缝迁移
  - 保留 asyncio 风格 (enqueue / start / shutdown) 便于未来切换

### 3.3 LLM 默认 vs opt-in

- LLM 4 秒一次 + 1000 tokens 输入 + ~200 tokens 输出 = **不是免费的**
- Day 8 选 **opt-in** (text_compressor 显式 `method="llm"`):
  - scheduler 默认仍走 rule_based
  - 用户想用 LLM 时显式 opt-in
  - 未来通过环境变量 (见开发目标) 可让 scheduler 默认 LLM

### 3.4 4 层降级链

```
LLM 异常
  → LLM 失败 (3 层重试后放弃)
  → rule_based text_compress
  → 原文透传
  → 永不崩 (worker / daemon / 任何上层)
```

每层都验证 ratio 有效, 无效就降到下一层。

### 3.5 向后兼容

- `armor_compress()` / `armor_pre_compact_hook()` / `algo_scheduler.process()` 全部**签名不变**
- 唯一新增字段: `ScheduleDecision.route_algo` (老调用方不读就忽略)
- `MARK42_ALGO_USE_SCHEDULER=false` / `MARK42_ALGO_FAIL_SAFE` 等环境变量全保留
