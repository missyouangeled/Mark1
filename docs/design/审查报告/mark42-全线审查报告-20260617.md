## 二、模块级逐项审查

### 2.1 Armor（上下文铠甲） — 9/10

**已做好的**：
- `armor_check()` 文件大小估算法 + du 辅助，轻量可高频调用
- `_llm_analyze()` DeepSeek API 调用链路完整（从 openclaw.json 取 key、Mark42 config 取模型名）
- LLM → 启发式回退策略正确
- 守护模式 `armor_guard()` 简洁清晰
- `_find_active_session()` 按 .lock mtime 选择，有 LOCK_MAX_AGE 过滤死 session

**发现的问题**：

🟡 **A1. 文件大小估算法的精度问题**
```
estimatedTokens = size_bytes // BYTES_PER_KTOKEN * 1000
```
`BYTES_PER_KTOKEN = 14*1024` 这个常数对 DeepSeek（中文模型）偏大——中文每 token 约 1.5-2 字符，而 JSONL 文件里大量是英文控制字符和 JSON 结构。实际测试：一个 10MB JSONL 文件可能对应 ~40K tokens 而非 ~700 tokens。这个估算会在大文件上系统性低估，导致压缩不及时。

**建议**：改为从 JSONL 文件中采样前 100 行计算实际的 `chars/line` 和 `chars/token` 比率，动态校准而不是硬编码一个固定常数。

🟡 **A2. `_read_session_tail` 在守护模式高频调用的内存开销**
`armor_guard()` 每 300 秒调用 `armor_check()`，后者不调 `_read_session_tail`，所以没问题。但 `armor_compress()` 每次都 `_read_session_tail(60)` 然后 `_llm_analyze(messages[-40:])`。如果 JSONL 文件超过几 MB，`f.seek(0,2)` 倒着读会重复扫描大量重复字节。

**建议**：给 `_read_session_tail` 加一个简单的行号缓存（如 `(file_path, mtime) → line_count`），只读新增部分。

✅ **A3. API key 安全** — 已正确从 openclaw.json providers 中读取，不硬编码。

### 2.2 Engine（循环引擎） — 8/10

**已做好的**：
- daemon 模式完整：broker 扫描 + 游标追踪 + Loop 到期执行 + 统一持久化
- Loop 模板 5 个有实际逻辑（context-guard / health-watch / model-fallback / task-watch / memory-index）
- 文件锁 + 竞态修复已完成
- Heavy 任务自动创建 watch Loop
- 每 10 次 tick 自动 log rotation + status 快照

**发现的问题**：

🔴 **B1. `engine_daemon` 主循环无心跳/健康自检**
当前 daemon 如果静默崩溃（比如一个未捕获的异常），父进程 assemble 因 `wait()` 阻塞在 armor_proc 上而无法及时感知。对外没有任何心跳信号。

**建议**：daemon 每 N 次 tick 写一个 heartbeat 时间戳到 `ENGINE_STATE/daemon-heartbeat.json`。assemble 用 `os.kill(pid, 0)` + 心跳文件双重验证子进程存活。

🟡 **B2. `engine_start()` 模板 Loop 的编号后缀逻辑有 bug**
```python
if name in loops and not template:
    print(f"⚠️ Loop '{name}' 已存在，覆盖注册")
elif name in loops:
    # 同名 template Loop 已存在 → 加编号后缀
    idx = 2
    while f"{template}-{idx}" in loops:
        idx += 1
    name = f"{template}-{idx}"
```
问题：这里 `name = template`（L117），如果 `name in loops` 且 `template` 非空，它去找 `template-2`。但如果 `template-2` 不存在而 `template`（同 name）已存在，它会正确创建 `template-2`。但如果 `template-2` 也存在（例如被手动注册过），会递增到 `template-3`——但 `template` 本身没变。这个逻辑只处理了序号冲突，没处理原始 template 名冲突的语义。实际没出 bug 是因为当前只有 3 个 Loop 且不会被重复注册。

**建议**：在 `engine_start()` 开头就检查模板 Loop 是否已存在且 active，如果是则直接返回 "已在运行"，而不是创建副本。

🟡 **B3. `health-watch` 的 disk/mem 解析脆弱**
```python
disk_root = os.popen("df -h / | tail -1 | awk '{print $4}'").read().strip()
```
依赖 `df -h` 输出格式。如果某天 `df` 输出列顺序变了（例如加了新字段），这段代码静默失败且返回空字符串而不会报错。

**建议**：改用 `shutil.disk_usage()` 和 `psutil.virtual_memory()`（如果愿意加依赖），或至少检查命令返回值非空再解析。

### 2.3 Heavy（重型战甲） — 7/10

**已做好的**：
- `heavy_preflight()` 预检全面（文件数/大小/上下文/磁盘/内存）
- `heavy_detect()` 多条件判定 + 上下文感知
- `heavy_start()` 自动分批 + context-aware 压缩
- 任务状态文件 + scratch 隔离 + `.keep` 保护

**发现的问题**：

🟡 **C1. `heavy_detect()` 和 `heavy_start()` 的文件扫描规则重复**
两个函数里各有一套 `rglob` + `__pycache__` / `.pyc` / `.git/` / `node_modules/` 过滤逻辑，完全重复。`heavy_start()` 还额外多了一层对 `HeavyState.glob("*.json")` 的循环（`engine_run_loop` 里 task-watch），但这三个地方的过滤条件不完全一致。

**建议**：抽取公共函数 `_list_project_files(path: Path) -> list[Path]`，三个地方统一用。

🟡 **C2. `heavy_execute()` 生成的 bash 脚本是 TODO 占位符**
```bash
echo '  processing: {f}'
# TODO: replace with actual file operation
```
当前生成的脚本只打印文件名，不做实质性文件操作。这是设计上预留的扩展点，但建议至少提供一个 `--command` 参数让用户注入实际命令。

🟡 **C3. `heavy_start()` 的 batch_size 算法**
```python
batch_size = max(3, min(30, int(total_files * remaining_pct / 200)))
```
这个公式的结果：100 个文件、20% 余量 → batch_size=10。直觉上合理，但 200 分母是凭空出现的 magic number，无人调过参数。

**建议**：改成可配置的，至少加个注释说明为什么是 200。

## 三、联网搜索结果与对比

### 3.1 事件总线模式
Mark42 的 Broker 采用的是 **文件追加式 JSONL** 模式——和 RabbitMQ/Redis pub-sub 相比，这是一种『零依赖、单机优先』的轻量事件总线。优势是不需要额外服务，劣势是**不支持真正的 subscribe/push**——只有 polling（daemon 每 30s 扫描一次新事件）。对标行业实践：MicroPython IoT、单机 cron-like 系统常用此模式，适合 Mark42 的"拆开可独立"哲学。

### 3.2 daemon 进程管理
联网搜索确认了 `start_new_session=True` 是阻止 SIGINT 穿透的正确做法。行业最佳实践还包括：
- **双 fork daemonize**（经典 UNIX 守护）：不适用 Mark42，因为 assemble 本身就是管理者
- **supervisor 进程监控**：Mark42 的 assemble 相当于一个迷你 supervisor，设计合理
- 触发重新打开日志文件句柄常用 `SIGHUP` 信号，当前 Mark42 未实现但也不需要（日志直接追加，无外部分割器）

### 3.3 JSONL 多进程并发写入
联网确认：Linux 上的 `O_APPEND` 写入小于 PIPE_BUF（通常 4KB）时是原子操作。Mark42 的 broker 事件每条 <1KB，多进程同时追加在当前规模下安全。但**建议不要依赖这个隐式保证**——统一所有 broker 写入走 `_append_broker()` 即可，当前也是这样做的。
