## 1.5 联网核查结论（2026-07-08 09:02 实查）

> 4 个不确定点已在写报告前联网核实，**全部得到确认**。

| 不确定点 | 核实结果 | 来源 |
|---|---|---|
| `openclaw sessions compact` 退出码语义 | 官方文档明确："command exits non-zero when the Gateway reports a failed compaction or is unreachable, so crons and scripts never mistake a silent no-op" | docs.openclaw.ai/cli/sessions |
| compaction 超时是否让 session hang | github.com/openclaw/openclaw/issues/43661 明确证实为已知 bug | OpenClaw issue tracker |

| systemd user service 杀子进程的行为 | systemd 会 SIGTERM → 等 stop-sigtimeout → SIGKILL；subprocess 用 `start_new_session=True` 可解耦 | serverfault.com/q/1162066 |

**关键推论**：R0 的"compact 超时导致 session hang"风险，已被官方 issue 证实为同症根因。**Mark42 业务代码处理超时是兜底完善的**（armor.py:641 `except subprocess.TimeoutExpired`），**但 session 端是 OpenClaw 的 bug**——即便我们完美处理超时，session 本身仍可能 hang。这是 OpenClaw 上游问题，不是 Mark42 能根治的。

## 三、改 Mark42 代码的真实崩溃风险点

> 这部分只列**真实风险**，不夸张、不凑数。每一条都有代码位置 + 触发条件 + 影响范围。

### 风险点 R0（升级）：`armor_guard()` 守护会自动触发真压缩（【重大发现】需重读）

**位置**：`scripts/mark42_modules/armor.py:743-759`
**原代码**：
```python
def armor_guard(interval_s: int = 300) -> None:
    """守护模式：每 N 秒检查一次，超阈值自动出手。"""
    print(f"🛡️ 上下文铠甲守护模式启动（每 {interval_s}s 检查）")
    try:
        while True:
            check = armor_check()
            usage = check.get("usagePercent", 0)
            ts = datetime.now().strftime("%H:%M:%S")
            print(f"[{ts}] 上下文 {usage}% — {check.get('summary', '')}")
            if usage >= THRESHOLD_ALERT:                        # 阈值 85%
                print(f"[{ts}] 🟠 自动触发压缩")
                result = armor_compress()                       # ← **不传 dry_run！**
                print(f"    → {result.get('action')}")
            time.sleep(interval_s)
    except KeyboardInterrupt:
        print("\n🛡️ 守护模式已退出")
```

**关键发现**：
- `armor_compress()` 函数签名：`def armor_compress(dry_run: bool = False)` —— **默认 False**
- `armor_guard()` 调用时不传 `dry_run`，**意味着默认走真压缩路径**
- `THRESHOLD_ALERT = 85%`（环境变量 `MARK42_CTX_ALERT_PCT`）
- **当前实时 usage = 3.6%，距离 85% 差 81 个百分点**，暂未触发

**这条之前是 R1 "理论风险"，现在升级为 R0 "在线行为"**。当 usage 上升至 ≥85% 时会发生：
1. armor-guard 每 5 分钟检查一次 → 命中阈值 → 调 `armor_compress()`
2. armor_compress 不传 dry_run → 走真压缩分支
3. `armor_pre_compact_hook()` 生成记忆索引（不影响主会话）
4. **如果 usage ≥ THRESHOLD_WARN (70%) → 调 `openclaw sessions compact --max-lines 200` 截短主会话到 200 行**

**崩溃风险评估**：
- 🔴 **OpenClaw sessions.compact 超时 → session hang** —— 已在 github.com/openclaw/openclaw/issues/43661 证实为已知问题。7-07 渲染事故的同症根因之一。
- 🔴 **如果 compact subprocess 抛 `subprocess.TimeoutExpired`** → armor.py:641 已 `except subprocess.TimeoutExpired`，正确处理 → armor-guard 守护不会挂，但主会话可能已被部分截短
- 🟡 **如果返回 0 但 session 未变小**（exit 0 但 bytes_saved=0）→ armor.py:599 处理：写 `compactError="no-bytes-saved"` + 报 broker，但可能重复触发
- 🟢 **FileNotFoundError / 其他 Exception** → armor.py:649-655 处理：写 `compactError`，守护不挂

**为何当前不危险**：
- 实时 usage = 3.6% 远低于 THRESHOLD_ALERT=85%
- armor-guard 在 systemd user 单元中以独立进程跑（cli.py:258-262 spawn）
- 即便触发，异常处理完备，**不会导致守护无限重启**

**意外发现的正面事实**：
- armor_compress 的所有 except 分支（TimeoutExpired / FileNotFoundError / Exception）都有兜底，**不会让 armor-guard 守护挂掉**。这是 commit `90738c2` 的成果（12 个审查问题修完）。
- 当前 armor-guard.log 4.7MB（7-02 09:06 落盘）、内容全是"上下文 X% — 正常"——**从未触发过自动压缩**。

**改动这条代码必须遵守的额外约束**：
- 任何对 `armor_guard()` 的修改必须保留 `except KeyboardInterrupt` 兜底
- 任何对 `armor_compress()` 默认参数的修改都会改变守护行为（**反向链路**：armor_guard → armor_compress）
- 如果想给 armor_guard 加 dry_run=True 保护，**必须先确认当前 armor_compress 默认值的语义**——不能盲改

### 风险点 R1：`armor_compress()` 真触发路径（详细）

**位置**：`scripts/mark42_modules/armor.py:541-650`（约 110 行）
**触发条件**（R0 的子条件）：
1. **R0.1** `armor_guard()` 检测到 `armor.usagePercent >= THRESHOLD_ALERT (85%)` —— **在线自动触发**
2. **R0.2** 显式 CLI 调用 `mark42.py armor --compress`（不带 --dry-run）
3. **R0.3** engine_daemon 收到 `context_monitor.alert` / `context_monitor.critical` 事件且 usage ≥ 85% → `subprocess.Popen` 启动 `mark42.py armor --compress`（engine.py:412）
4. **R0.4** `armor_pre_compact_hook()` 内部逻辑（仅生成记忆索引，不调 compact）
**做了什么**：
1. 调 `subprocess.run(["openclaw", "sessions", "compact", "agent:main:main", "--max-lines", "200", "--timeout", "180000", "--json"], timeout=200)`
2. 成功后用 `_save_json` 写入 `index["compactTriggered"]`、`compactMethod`、`preCompactBytes`、`postCompactBytes`、`bytesSaved`、`compressionEffective`
3. **Bug 历史**（7-01 commit `57b5345` 修复过）：`_save_json` 必须在字段设置完成后再调，否则 dict 引用导致字段丢失
**真崩溃风险**：
- 🔴 **如果 compact subprocess 自身异常 + 超时** → 抛 `subprocess.TimeoutExpired` → armor.py:641 正确处理 → 守护不挂
- 🔴 **如果 compact 成功但 _save_json 路径出问题**（磁盘满 / 权限错）→ 索引文件损坏 → 下次 status --json 报错 → 但 daemon 不会挂
- 🟡 **如果当前主会话本身正在被 compact** → 双重 compact 互锁（OpenClaw 端有 takeover 保护，会抛 takeover 错误，armor.py 处理后会写 `compactError`）

**为何当前不危险**：
- 当前 `armor.usagePercent = 3.6%`，**距离 THRESHOLD_ALERT 差 81 个百分点**
- `armor_compress()` 默认 `dry_run=False`，但只有显式调用或 armor_guard 自动触发才会跑

## 五、修改的"四不铁律"（核心约束）

> **任何 Mark42 代码修改必须遵守这 4 条**。违反任何一条 → 系统崩溃风险显著上升。

### 铁律 1：不动 `armor.py:541-650` 真触发分支

**理由**：这是 `armor_compress()` 的真压缩路径，唯一会调 `openclaw sessions compact` 的地方。当前不危险是因为 armor.usagePercent=3.6% + 默认 dry_run。一旦改了 + 阈值触达 + 真跑 → 主会话会被截断到 200 行。
**例外**：只在以下情况才允许碰：
- armor.usagePercent 已达 ≥80%
- 已有 VM 快照
- 改动后必须先 `--max-lines 1000` 干跑（不截短），再切回 200

### 铁律 2：不动 `engine.py` 读取逻辑

**理由**：engine.py 主循环 spawn `python3 mark42.py assemble`，任何读取 / 编码异常都会导致子进程退出失败 → 守护空转。
**例外**：可以在末尾追加新 loop 定义，但**不能修改现有 loop 的 spawn 命令**。

### 铁律 3：不直接 spawn 长跑 daemon 进程

**理由**：当前 engine-daemon / armor-guard 已经是 systemd user 单元托管。再 spawn 一份会导致端口冲突 / 重复写日志 / state 错乱。
**例外**：用 `systemctl --user status` / `restart` 操作现有 unit，不要直接 `subprocess.Popen(["python3", "scripts/mark42.py", "assemble"])`。

### 铁律 4：不跳过测试就 commit

**理由**：当前 700 passed / 97.5% 覆盖是最后的回归保护网。改完不跑测试 = 自杀。
**例外**：无。
