本轮继续沿既定接力主线补强单测，新增/补强了多个关键模块的测试，当前基线推进到：

- **526 passed, 2 skipped**

其中一个很关键的落点是：

- `scripts/mark42_modules/perf_bench.py`

  - 采用的是 **pure-helper smoke** 策略
  - 没有为了覆盖率去硬跑真实 LLM、真实写盘、真实重路径

这意味着 Mark42 的测试推进不是“刷数字”，而是在尽量保持工程克制的前提下，把真正值得收编的隐含契约纳入 pytest。

### 2）修复了一条真实运行中的坏链路：L2.5 语义索引缺失

这次不是只看代码，而是做了真实运行态巡检。巡检中发现：

- watchdog 持续报警
- `memory-embed-index` 目录存在但内容缺失
- `embeddings.npy` 不存在
- L2.5 语义检索链路因此失效

随后完成了完整修复：

- 使用 embed venv 重建索引
- 新索引恢复为 **7206 段 / 384 维**
- 重启 `openclaw-embed-sidecar.service`
- 确认 sidecar 已加载新索引
- 确认语义搜索恢复
- 确认 watchdog 不再继续追加 `embed-missing` 告警

这件事的价值很高，因为它证明了：

> **Mark42 不只是“看起来在跑”，而是真的能被巡检、被定位、被修复、被复核。**

### 3）补齐了两个影响运维体验的小口径问题

#### `memory-embed-search.py` 运行口径补齐

原来用户如果直接运行：

```bash
python3 scripts/memory-embed-search.py 'Mark42 启动'
```

会因为系统 python 缺 `numpy` 而失败。

现在脚本已经支持：

- 自动切换到 embed venv 重执行
- `--check` 可单独运行，不再被 query 必填参数卡住

这让语义搜索脚本从“知道门道的人能用”，变成了“更接近直接可用”。

#### watchdog 日志双写问题修正

定位到 `watchdog.log` 同一条告警会出现两次，根因不是 timer 重复，而是：

- 脚本自己 `tee -a`
- systemd 又把 stdout/stderr append 到同一日志文件

现在已经修正为：

- systemd 场景不再重复写
- 手动终端运行仍保留现场可读性

这是个小问题，但很伤信任感；修掉以后，日志会更干净、更像一个靠谱的运行系统。
