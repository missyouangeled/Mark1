## 一、守护模式资源占用

### 1.1 systemd 服务状态

| 服务 | 运行时长 | 内存(RSS) | 内存峰值 | CPU 累计 | 状态 |
|---|---|---:|---:|---:|---|
| armor-guard | 5h 17m | 9.7 MB | 570.3 MB | 2.0s | ✅ active |
| engine-daemon | 5h 17m | 24.3 MB | 257.8 MB | 24.2s | ✅ active |
| watchdog | 2min/timer | <1 MB | <1 MB | <0.1s | ✅ active (timer) |

**分析**：
- armor-guard 常驻 9.7MB，CPU 5 小时只用 2 秒 -> **极低开销**

- 内存峰值来自 LLM 调用时的临时分配（armor 570MB / engine 258MB），但常态 RSS 远低于峰值
- **结论：守护模式对系统无感知负担**

## 二、压缩子系统基准 (2026-07-20 实测)

### 2.1 本地算法层（裸算法延迟 / 内存 / 压缩率）

| 算法 | 1KB P50 | 10KB P50 | 100KB P50 | 1MB P50 |
|---|---:|---:|---:|---:|
| smartcrush (JSON) | 0.01ms / 3KB | 0.07ms / 22KB | 0.69ms / 252KB | 8.31ms / 2.7MB |
| text | 0.46ms / 3KB | 0.44ms / 17KB | 0.57ms / 164KB | 1.55ms / 1.6MB |
| code | 0.33ms / 118KB | 0.94ms / 1MB | 35.55ms / 12.9MB | 435.33ms / 133MB |
| log | 0.06ms / 8KB | 0.63ms / 40KB | 6.15ms / 316KB | 62.21ms / 3.2MB |
| diff | 0.03ms / 6KB | 0.23ms / 51KB | 2.29ms / 494KB | 24.32ms / 5.1MB |

**关键发现**：
- **text 压缩器最快**：1MB 文本仅 1.55ms，压缩率 100%
- **code 压缩器最慢**：1MB 代码 435ms + 133MB 内存（AST 解析开销大）
- **smartcrush 对 JSON 极快**：1MB 仅 8.31ms
- **diff 压缩器表现稳定**：线性增长，1MB 24ms
- **log 压缩器中等**：1MB 62ms，但压缩率 99.6%

### 2.2 调度层（algo_scheduler.process 端到端）

| 输入大小 | P50 | P95 | 峰值内存 |
|---|---:|---:|---:|
| 1KB | 0.02ms | 0.05ms | 28KB |
| 10KB | 0.07ms | 0.13ms | 37KB |
| 100KB | 20.44ms | 27.97ms | 128KB |
| 1MB | 171.27ms | 174.47ms | 1MB |

**分析**：
- 调度层开销极小（1KB 仅 0.02ms），主要是路由判断
- 100KB+ 时开销增大，因为包含了算法执行时间
- 1MB 时 171ms -> 用户无感知（异步队列里跑）

### 2.3 异步队列层（CompressQueue）

| 输入大小 | P50 | P95 | 峰值内存 |
|---|---:|---:|---:|
| 1KB | 0.13ms | 0.32ms | 4KB |

**分析**：
- 入队 + 等待完成仅 0.13ms -> **队列开销可忽略**
- async_entry（含 LLM 调用）单次 7-16s，受网络/模型波动影响，不计入本地基线

### 2.4 与 6/26 历史基线对比

| 算法 | 6/26 P50 (1MB) | 7/20 P50 (1MB) | 变化 |
|---|---:|---:|---|
| smartcrush | 7.64ms | 8.31ms | +8.7% (正常波动) |
| text | 2.74ms | 1.55ms | -43.4% (更快了) |
| code | 509.54ms | 435.33ms | -14.6% (更快了) |
| log | 63.14ms | 62.21ms | -1.5% (持平) |
| diff | 24.17ms | 24.32ms | +0.6% (持平) |

**结论**：性能无退化，text/code 算法略有提升（可能是系统负载差异）。

## 三、watchdog 修复记录

### 问题
`mark42-watchdog.service` 的 `ExecStart` 调用 `mark42 watchdog --check`，但 pip 化后 CLI 未注册 `watchdog` 子命令，导致 service 每 2 分钟崩溃一次（exit code 2 = argparse 错误）。

### 修复 (2026-07-20)
- 新增 `mark42/watchdog.py` 模块（Python 重写原 shell 脚本）
- 在 CLI 注册 `watchdog --check` 子命令
- 功能：心跳检查 + 进程检查 + 自动重启 + 日志留痕
- 正常时静默退出，异常时写 `watchdog.log`

## 四、结论与建议

1. **守护模式开销极低**：armor 9.7MB + engine 24.3MB = 34MB 常驻，CPU 近 0
2. **压缩子系统性能良好**：1MB 文本压缩 < 2ms，代码压缩 < 500ms（可接受）
3. **code 压缩器是性能瓶颈**：1MB 代码 435ms + 133MB 内存，大代码库建议分批
4. **异步队列开销可忽略**：入队 0.13ms，不阻塞主流程
5. **watchdog 已修复**：此前 2 周反复崩溃的问题已解决
