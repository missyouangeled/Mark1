## 1. 项目背景

OpenClaw 主 agent 的"长期记忆"机制本身有缺陷：

1. **compaction 覆盖 bug** — OpenClaw 的 `compaction.memoryFlush` 默认 prompt 会**覆盖**之前已写入的项目信息（[issue #34919](https://github.com/openclaw/openclaw/issues/34919)）
2. **session 失控** — 主 session jsonl 累积超过一定行数（实测崩溃阈值约 1.5-2 万行）后会出现读不全、刷新卡顿、自动 corrupt 备份等问题
3. **救命类动作反而危险** — 任何"自动截断 session"的设计，**目标就是当前主 agent 正在用的 session**——自残

**用户（点点）的核心诉求**：
- "怕有什么意外或者自动压缩 你又不记得了"
- 项目信息**绝对不能丢**
- 救命动作**必须让我知道**（不能默默做了）

## 2. 方案组成（三件套）

### 2.1 A 方案：compaction.memoryFlush prompt 改造

**目标**：绕开 OpenClaw 的覆盖 bug，**强制把项目信息写到外部文件而不是内嵌到记忆系统**

**实施位置**：
```
~/.openclaw/openclaw.json
agents.defaults.compaction.memoryFlush.prompt     (新)
agents.defaults.compaction.memoryFlush.systemPrompt (新)
```

**核心思路**：
- compaction 触发时，prompt 告诉 LLM：**不要写回主对话/记忆系统**（那里会被覆盖）
- 改写 `memory/projects/<project>/` 目录下的项目文件
- 写完**用 `memory/projects/_pending/index.json` 登记**（主会话恢复时能优先读）

**配置示例**（已在 2026-07-06 落地）：
```json
{
  "compaction": {
    "memoryFlush": {
      "enabled": true,
      "prompt": "你是 memory/projects/<project>/ 的守门员。\n\n[核心原则] 这次 flush 不要再写回主对话上下文，那里会被覆盖。\n\n[动作] 1. 读 memory/projects/_pending/index.json 看哪些项目待整理；2. 把这次对话里的项目信息写到对应项目的 .md 文件；3. 在 index.json 里登记；4. 写 actions.jsonl 留痕。\n\n[绝对不要] 不要调用 compact、不要回写主对话、不要总结成会话总结。",
      "systemPrompt": "你是贾维斯的项目锚定助手。\n\n救命 1：信息优先落盘，落不进盘就告警。\n救命 2：项目状态随时可比对。\n救命 3：宁可冗余也不要丢。\n\n绝不压缩、绝不总结、绝不回写主对话。"
    }
  }
}
```

### 2.2 救命 1：session 长度监控 cron（v3 含 watcher 告警转发）

**目标**：
1. 在主 session jsonl 行数接近崩溃阈值时**只告警**，**绝对不动 session 文件**
2. 转发 OpenClaw `session-size-watcher` 的未读告警到主会话（之前盘点发现 watcher 不主动通知，告警被埋，2026-07-06）

**实施位置**：
- cron id: `b560cd2c-b4e3-415c-96f7-3074e8f11f62`
- 调度：`*/5 * * * *` @ Asia/Shanghai（每 5 分钟）
- sessionTarget: `isolated`（独立进程，不影响主会话）
- payload: `command` 模式
- delivery: `none`（不靠 announce 路由）

**核心逻辑**（写死在脚本里，**不可通过 cron edit 改阈值**）：
```sh
# 找到最新的 main session jsonl（排除 lock/trajectory/deleted/corrupt/bak/pre-test-backup）
SESS=$(ls -t ~/.openclaw/agents/main/sessions/*.jsonl | grep -v "\.lock" | grep -v "\.trajectory" | grep -v "\.deleted" | grep -v "\.corrupt-" | grep -v "\.bak\." | grep -v "\.pre-test-backup" | head -1)
LINES=$(wc -l < "$SESS")
THRESHOLD=12000  # 写死

if [ "$LINES" -gt $THRESHOLD ]; then
  # cooldown 30 分钟（用 /tmp/emergency1-alert.flag 的 mtime 判断）
  ALERT_FLAG=/tmp/emergency1-alert.flag
  if [ ! -f "$ALERT_FLAG" ] || [ $(find "$ALERT_FLAG" -mmin +30) ]; then
    MSG="[救命 1 告警] main session 行数 ${LINES} > ${THRESHOLD}..."
    python3 scripts/openclaw-proactive-inject.py --source "emergency-1-alert" "$MSG"
    touch "$ALERT_FLAG"
  fi
else
  rm -f "$ALERT_FLAG"  # 正常了就清掉告警标志
fi
```

**关键设计原则**：
1. **不修改 session 文件**（不 cp、不 tail、不 mv、不 rm lock）
2. **阈值写死**（不能用 `cron edit --command` 临时改）
3. **Cooldown 30 分钟**（避免每 5 分钟重复告警轰炸）
4. **告警通过 `openclaw-proactive-inject.py` 推到 webchat 主会话**

**为什么不"自动截断"**：
参见 CASE-20260706-005 —— 救命 1 的截断动作会**破坏正在使用的主 session**，触发 OpenClaw 自动 corrupt 备份，**导致主会话中断**。

### 2.3 救命 2/3：项目状态可追溯 + 冗余落盘（设计中）

- **救命 3**：项目信息**双写** —— `memory/projects/` + `docs/design/<项目>.md`，两边同步

### 5.1 救命 1 v2 隐藏 bug 修复记录（2026-07-06 12:30）

救命 1 v2 上线时**漏了 2 个隐藏 bug**（只测了静默分支，**告警分支没真测**）。点点的"明显没修好啊"反馈让重新审查时发现：

**Bug #1：相对路径**

- **症状**：告警分支必失败（python 找不到 inject.py）
- **原因**：cron runner 的 cwd 是 `/home/missyouangeled`（**不是 workspace**），用 `python3 scripts/...` 必失败
- **修复**：用绝对路径 `python3 /home/missyouangeled/.openclaw/workspace/scripts/openclaw-proactive-inject.py`
- **怎么发现的**：写临时探针 cron `echo $PWD` 发现 cwd 错了

**Bug #2：`ls -t | head -1` 选错 session**

- **症状**：救命 1 会判错对象（拿 fake session 测时没触发告警，因为它选了主 session）
- **原因**：`ls -t` 在同时间戳文件排序不稳定
- **修复**：改读 `sessions.json` 找 `agent:main:main` 的 `sessionId`（这是当前主会话的权威来源）
- **怎么发现的**：造 fake 13000 行 session 测试，告警没触发，看 cron runs 才发现选了主 session（156 行）

### 5.2 救命 1 v3 新增 watcher 告警转发（2026-07-06 12:50）

问题起源：**OpenClaw 自带的 `session-size-watcher` 从今天 04:00 就在 `alerts.json` 写 trajectory 超阈值的 WARN（7 条）**，但这个 watcher 不主动通知主会话，**告警被埋了**。如果不做盘点永远不会发现。

修复：

- 救命 1 cron 扩展成两个并行检查：
  1. **session 行数监控**（v2 已有）
  2. **watcher 告警转发**：读 `~/.local/state/openclaw/session-size-watcher/alerts.json`，如果有 `unread=True`，去重聚合后推一条到主会话
- 用 `/tmp/emergency1-watcher-alert.flag` 做 cooldown（30 分钟）
- **不修改 watcher 自己的 alerts.json 标记已读** —— 由救命 1 推完后**主动 mark read**（避免 watcher 又生成新 alert 时双发）

**验证（2026-07-06 12:50 cron run）**：

```
status: ok
[救命 1 静默] session=68e35bdd-... LINES=228
✅ 注入成功: [救命 1 watcher 转发] session-size-watcher 未读告警: WARN=7
[救命 1 watcher 告警已转] WARN=7
exit: 0
```

教训：

- **救命类逻辑**必须实测两条分支（静默+触发），**只看 exit 0 = 没测**
- 不能用 cron runner 的相对路径假设（cwd 可能是 gateway 启动目录）
- 选 session 文件不能靠 `ls -t`（同时间戳排序不稳定）
- **用户反馈比自测更可靠** —— 如果点点没坚持"明显没修好"，这个 bug 会留到真出事时
- **盘点自动行为非常重要** —— 8 个 user systemd timer 之前根本没盘点过，找到 watcher 告警被埋这件事就是靠盘点

### 5.3 救命 1 配套：trajectory 压缩（2026-07-06 13:00）

session-size-watcher 的轨迹告警（trajectory > 5MB）根源是 OpenClaw runtime 把每次 agent.run 的完整上下文快照（含 tools schema、prompt、messagesSnapshot）写到 trajectory.jsonl。**watcher 只能清陈旧 trajectory（mtime 早 300s 以上）**，**当前活跃 trajectory 永远不会被它清**。

配套脚本：`scripts/compress-trajectory.py`

**用法**：

```bash
# 默认 dry-run，看会变成多大
python3 scripts/compress-trajectory.py <session-id> --keep 3

# 真写：备份 + 压缩 + 验证
python3 scripts/compress-trajectory.py <session-id> --keep 3 --execute
```

**策略**（保守）：

- 保留所有 lifecycle 事件（session.started, trace.metadata, session.ended, model.completed, trace.artifacts）
- context.compiled / prompt.submitted 只保留最近 3 个
- **不删除** model.completed 的 messagesSnapshot（保留完整对话快照供调试）

**安全保证**：

- 默认 dry-run
- --execute 才真写
- 写新文件用 `.tmp + rename`（原子替换）
- 备份保留在 `<原文件名>.pre-compress-<HHMM>`
- gateway 进程不持有 trajectory fd（事件驱动关闭）

**实测**（2026-07-06 13:00）：

- 主会话 trajectory: 7.56 MB → 4.28 MB (-48%)
- session-size-watcher 之后跑: level CRITICAL → INFO，告警清零
