## 四、🚨 真实问题（不写就是失职）

### 问题 1：**上下文压缩链路断裂**（最严重）

**现象**：armor 检测到 ≥ 阈值 → 生成 memory-index → 但**真 session 从未被截短**

**证据**：
- armor `actions.jsonl`：314 次 `compress` + **0 次 `compact`**
- session 文件 **2.9 MB / 892 messages / 0 个 `{"type":"compact"}` 触发标记**
- armor 检测 `usagePercent` 从 91% 单调升到 143%（实测，20 分钟涨 50%）

**根因（armor.py 第 561 行）**：
```python
# 想通过外部 CLI 触发 /compact:
result = subprocess.run(["openclaw", "agent", "--message", "/compact", ...])
# 但 compactTriggered 永远是 False —— /compact 无法从外部 --message 触发
```

**OpenClaw 设计**：`/compact` 是**主会话内**的斜杠命令识别，要在 OpenClaw agent session 内才能识别。从外部 `openclaw agent --message "/compact"` 注入时，被当作普通消息而非命令。

**结果**：
- armor 一直在"自拍"（生成 memory-index）但从不让 OpenClaw 真截短
- memory-index 本身才 2.1KB，抵不上 2.9MB session 的累计
- 上下文持续失控

**修复优先级**：🔴 **P0** — 这是项目当前最大的真 bug

**建议修复路径**：
1. **临时**：让 armor 直接通过 `session_fence_safe.py` 写 `{"type":"compact"}` 到 session（注意 ownership，可能再撞 fence）
2. **正确**：armor 调用 OpenClaw 的真 compact API（如果有）
3. **降级**：让 armor 周期性 archive session jsonl（如 `*.jsonl.bak.<ts>`）+ 重启 OpenClaw，让它读精简版

### 问题 2：**contextWindow 配置不一致**

**实测 `armor_check()` 返回**：
```
"contextWindow": 1,000,000   ← 实际配置
"BYTES_PER_KTOKEN": 2048
```

**config.py 默认**：
```
DEFAULT_CONTEXT_WINDOW = 131072  ← 设计假设
BYTES_PER_KTOKEN = 2 * 1024
```

**问题**：用户配置覆盖了默认窗口到 1M（可能是 Claude 3.5 Sonnet 实际），但 armor 仍按 128K "应该压" 的逻辑在判断。这意味着：
- 1.4M tokens（实测）÷ 1M 窗口 = 141% ← 算出来"超过"
- 但 1.4M ÷ 128K 旧窗口 = **1109%**

**问题**：阈值是基于旧窗口调的，现在窗口变了，阈值也跟着错配。

**修复优先级**：🟠 P1

### 问题 3：**评估指标与实际效果脱节**

**memory-index.json** 一直生成，但**没有任何监控指标**：
- 没记录"压缩前 vs 压缩后"的真实 byte 数
- 没记录"session 是否真被截短"
- 没报警"压缩 N 次但 session 没变小"

也就是说：**今天 armor 跑了一整天，看起来在忙，实际在做无用功**。

`lastResult = {"before": 140.8, "after": 140.8}` —— 这个数据 **N 份**这样的 lastResult，没任何 prompt 出来。

**修复优先级**：🟠 P1 — 应该加报警"连续 3 次 compress 后 session 没小 → 触发升级事件"

### 问题 4：**armor 真生产零测试覆盖**

今天装的 pytest：**armor.py 50%+ 覆盖**，但**全测的是 `armor_check()` 和 `armor_compress()` 的输入输出**。

**没测的**：
- `_trigger_compact()` 真触发流程
- `compactTriggered` 字段是否真的写入
- 与 OpenClaw CLI 的子进程交互
- session 文件 fence 防护

也就是说：**今天的测试让 armor 模块"看上去"被盖了，实际核心流程没盖**。

**修复优先级**：🟡 P2 — 需要写"集成测试"覆盖跨进程流程

### 问题 5：**Phase 2 范围未启动**

`mark42-Phase2路线-20260629.md` 写得很细，但**今天没动 76 个测试里的任何一个**。
