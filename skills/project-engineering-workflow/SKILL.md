# 通用工程管理流程

> 轻量但严格。不影响日常开发速度，但保证工程大了不乱。
> 适合：个人项目、小团队、AI 辅助项目。
> 不适合：需要 Scrum/Kanban/全员站会的正规大团队。

---

## 一、版本号规范（Semantic Versioning）

```
v<主版本>.<次版本>.<修订号>
  MAJOR . MINOR . PATCH

MAJOR = 架构级改动、模块重写、不向后兼容
MINOR = 新功能、新模块、显著改进
PATCH = bug 修复、健壮性改进、文档更新
```

**版本号写在哪**：项目的配置文件 + 运行时可读取的状态文件，两处都要有。

## 二、文档体系（每个项目标配）

```
<project>/docs/
├── README.md           ← 📂 入口索引（含路径速查表）
├── 架构设计.md          ← 怎么拆的、为什么这么拆
├── 更新日志.md          ← 每个版本改了什么
├── 开发经验.md          ← 踩过的坑 + 自检清单（⚠️ 下次写代码前先看这个）
├── 工程管理方案.md       ← 📐 本文件
└── 审查报告/            ← 每次全线审查存档
```

文档规则：
- 架构和设计分开（设计可以变，架构记录为什么当时这样选）
- 更新日志按时间倒序，每个版本一条
- 开发经验记录根因（不是表面现象），按类别归并
- 审查报告单独存放，不在主目录堆砌

## 三、开发流程（四步）

```
🎯 计划  →  🔧 开发  →  🧪 验证  →  📋 收尾
```

### 3.1 计划（5-10 分钟）
- 明确要改什么、为什么改
- 读项目的「开发经验」自检清单
- 评估影响范围（改哪个模块、会碰到谁）
- 大改动先写一句话说明文档

### 3.2 开发
- **修改前先 grep 全量搜索**——确认所有调用方
- 每个模块改完即时跑语法检查
- 不要攒一堆改动再一次验证——改一个测一个

### 3.3 验证
- 跑项目的测试脚本（全量）
- 如果有守护/服务进程，确认启动 → 运行 → 优雅关闭
- 确认 stderr 无新增错误

### 3.4 收尾（必做，不能跳过）
1. 更新「更新日志」（版本号 + 改动清单 + 验证结果）
2. 涉及文档变更 → 同步更新「README/入口索引」
3. 踩到新坑 → 追加「开发经验」+ 更新自检清单
4. git commit（Conventional Commits 格式）
5. git push

## 四、Git Commit 规范（Conventional Commits）

```
<type>: <简短描述>

类型：
feat     = 新功能
fix      = bug 修复
docs     = 文档
refactor = 重构（不改功能）
perf     = 性能优化
test     = 测试相关
chore    = 杂务（日志、路径、配置管理、依赖更新）

示例：
feat: armor-guard 新增上下文 70% 预警自动通知前台
fix: task-watch 在无活跃任务时 UnboundLocalError
docs: README 新增路径速查表
chore: 日志迁移到数据盘 + 自动截尾
```

## 五、分支策略（默认单分支）

```
master  ─── 永远可部署、永远干净

日常开发直接在 master。
高风险改动 → 临时分支 → 验证完合并回 master 删分支。
临时分支命名：<type>/<简短描述>，如 fix/session-read-nesting
```

---

## 六、质量门禁（每次收尾前过一遍）

| # | 检查项 |
|:---:|------|
| 1 | 语法检查零报错 |
| 2 | 测试脚本全通过 |
| 3 | 入口命令正常输出 |
| 4 | stderr 无新增错误 |
| 5 | 更新日志已追加当前版本条目 |
| 6 | commit message 符合 Conventional Commits |
| 7 | 涉及路径/常量变更 → 入口索引已同步 |
| 8 | （如有守护）重启后 1 分钟心跳正常 |

---

## 七、新项目启动时先做这几件事

1. 创建文档骨架：README / 架构设计 / 更新日志 / 开发经验 / 本文件
2. 设定初始版本号 v0.1.0
3. git init + 首次 commit
4. 写一个最小测试脚本（哪怕只测 import）
5. 更新 workspace 的项目索引

---

## 八、适配指南

套用到具体项目时：
- 文档路径把 `<project>/` 替换为实际路径
- 测试脚本命令改为实际命令
- 如果项目有守护/服务，加第 8 条检查
- 如果项目多语言，语法检查命令改为对应语言
- 「开发经验」从第一天就开始写，哪怕只有一条
