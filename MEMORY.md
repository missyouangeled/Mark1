## 用户安全原则：所有自动 / 不可见行为必须有据可查（2026-06-30 记录）

> **点点原话**："怕有什么意外或者自动压缩 你又不记得了"
> **场景**：点点担心 AI 跑的时间长、模型换了几次后,自动触发的行为(自动压缩、自动重起、自动清理)会"被忘记"或"记错",系统跑偏了用户也不知道。
> **原则**：
>
> 1. **所有自动行为必记文档**。任何自动触发的操作(compress / rotate / cleanup / restart),每次都要在对应模块文档或 daily 里留痕。AI 模型换了也能从文档复现,不用依赖 AI 自己记得。
> 2. **不可见的副作用必告知**。自动做事之前要给用户说一声,做完要给汇报。不能让"AI 默默做了什么"成为常态。
> 3. **写下来 > 记着**。"我以为我记得" = 不可靠。"文件里有" = 可靠。优先写文件而不是依赖会话记忆。
> 4. **动作链路要可复盘**。每天结束时,今天做了什么(自动 + 手动)都要在 daily 里能查到,谁触发的、为什么、有没有异常。
>
> **执行细则**（在 mark42 落地的体现）：
> - `heavy_execute` 默认 dry-run,要 `--execute-now` 显式才能真跑(防止自动任务静默执行)
> - broker / history / actions 轮替前必须通知(已实现:rotation 状态写入 log-rotation.json)
> - armor_compress 默认 skip 阈值外,真触发要写 actions.jsonl + memory-index.json 双份(P0 修复后已做)
> - 任何修改 systemd service / 守护进程配置前,必须读 `docs/对系统操作必须要参考的崩坏案例.md`
>
> **为什么这条最重要**：点点一个人在公司,AI 跑偏了没人发现。等发现时已经太晚。"有据可查"是他最后的安全网。
