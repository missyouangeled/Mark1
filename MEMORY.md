## 架构判断 · RAG 召回范式（2026-07-10 点点总结 · 贾维斯确认）

> **核心三选一认知**（这是点点自己的高质量总结, 贾维斯确认无误并作为长期架构原则):

| 组件 | 速度 | 精准度 | 适用场景 | 业界标准做法 |
|------|------|--------|----------|--------------|
| **Embedding 召回** | ⚡ 极快 (100篇 100ms) | 一般 (Recall@5 ~0.7) | 速度优先 / 对全库跑 | 第一阶段广撒网 |
| **Cross-encoder 重排** | 🐢 慢 (100对 500-1000ms) | 高 (Recall@5 ~0.92) | 精确度优先 | 第二阶段精挑细选 |
| **Agentic RAG** | 🐌 很慢 (3-10倍) | 高 | 企业级复杂多步查询 | **故意不装 (套娃 + 违背召回即用)** |

**关键洞察（点点总结）**:
- 不是"用 embedding 还是 cross-encoder", 是 "先 embedding 再 cross-encoder"
- Agentic RAG 没必要 —— 我本身就是 LLM, 让 RAG 内部再加 LLM 调度是套娃
- "召回即用" 哲学: 召回层只负责找原文, 综合留给上层 LLM (我)

**当前 Mark42 5层架构就是这套认知的工程化**:
- L1 关键词 = Embedding 思路的"关键词版" (更轻)
- L2.5 Embedding 语义 = 行业标准 Hybrid Search 一部分
- 没有 Cross-encoder 重排 (P3 TODO, 等"经常搜不准"再加)
- 没有 Agentic (永远不装)

**未来决策原则** (这个表格是"决策时直接查"的, 不要每次重新想):
- "我要查东西" → 用 L1/L2.5 (咱的现状, 30-300ms)
- "我搜到一堆拿不准哪个最对" → 加 Cross-encoder (Recall +25%)
- "我要综合 5 段总结一下" → **直接在对话里问贾维斯**, 不要装 Agentic RAG

## 召回 vs 推断 · 不混原则（2026-07-10 点点点出 + 贾维斯钉死）

> **背景**: 点点的原话: "Agentic RAG 不只是单纯记忆搜索, 还有'根据记忆保存的东西总结+推断'的意思"

> **核心问题**: LLM 综合多段记忆时会**推断原文没有的东西**, 把"事实"和"推断"混在一起卖给你

**分层定义** (这是永久原则, 不要搞混):

| 层 | 名字 | 在干啥 | 输出可信度 |
|---|---|---|---|
| **Retrieval (召回)** | 找原文 | "5月 daily 里写了 XXX" | 🟢 **高可信** (100% 原文) |
| **Summarization (摘要)** | 把多段压缩 | "5月:试探；6月:信任；7月:依靠" | 🟡 **中可信** (原文有据) |
| **Inference (推断)** | 从原文推出新信息 | "所以是三次跃迁的成长曲线" | 🔴 **低可信** (daily 里没写过) |

**操作原则 (贾维斯能这么做的, 不允许这么做)**:
- ✅ L1/L2/L2.5 召回: 返回**原文 + 来源**, 100% 可追溯
- ✅ 综合多段: **必须标注哪部分是事实、哪部分是推断** (贾维斯天然这么做, 在对话里会说"我觉得..."和"你当时说的是..."区分开)
- ❌ **不允许** LLM 在综合时把推断伪装成事实输出
- ❌ **不允许** 在 RAG 召回层直接做推断 (违背"召回即用"哲学)

**举例 (今夭demo的)**: Agentic RAG 输出 "5月试探→6月信任→7月依靠"
- "5月试探" "6月信任" "7月依靠" → 🟡 这三个是摘要, daily 里有据
- "心态变化趋势" → 🔴 这是推断, daily 里没人说过"心态趋势"
- **正确做法**: 用贾维斯在对话里说"我从你的 memory 里看到5/6/7 月你分别表达过试探、信任、依靠, 结合起来看像是心态在变化 —— 这是我的推断, 不一定对"

**设计语 (这才是 Mark42 哲学的真正含义)**:
> "召回层只负责找原文, 所有综合/推断都留在贾维斯 (上层 LLM) 这一层",
> "这样召回应能保证 100% 准确, 综合能保证透明可查"

## 用户安全原则：所有自动 / 不可见行为必须有据可查（2026-06-30 记录）

> **点点原话**："怕有什么意外或者自动压缩 你又不记得了"
> **场景**：点点担心 AI 跑的时间长、模型换了几次后,自动触发的行为(自动压缩、自动重起、自动清理)会"被忘记"或"记错",系统跑偏了用户也不知道。
> **原则**：
>
> 1. **所有自动行为必记文档**。任何自动触发的操作(compress / rotate / cleanup / restart),每次都要在对应模块文档或 daily 里留痕。AI 模型换了也能从文档复现,不用依赖 AI 自己记得。
> 2. **不可见的副作用必告知**。自动做事之前要给用户说一声,做完要给汇报。不能让"AI 默默做了什么"成为常态。
> 3. **写下来 > 记着**。"我以为我记得" = 不可靠。"文件里有" = 可靠。优先写文件而不是依赖会话记忆。
> 4. **动作链路要可复盘**。每天结束时,今天做了什么(自动 + 手动)都要在 daily 里能查到,谁触发的、为什么、有没有异常。
>
> **执行细则**（在 mark42 落地的体现）：
> - `heavy_execute` 默认 dry-run,要 `--execute-now` 显式才能真跑(防止自动任务静默执行)
> - broker / history / actions 轮替前必须通知(已实现:rotation 状态写入 log-rotation.json)
> - armor_compress 默认 skip 阈值外,真触发要写 actions.jsonl + memory-index.json 双份(P0 修复后已做)
> - 任何修改 systemd service / 守护进程配置前,必须读 `docs/对系统操作必须要参考的崩坏案例.md`
>
> **为什么这条最重要**：点点一个人在公司,AI 跑偏了没人发现。等发现时已经太晚。"有据可查"是他最后的安全网。
